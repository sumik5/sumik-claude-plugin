# データストーリーテリング・ファイル形式・ソフトウェア選択

## 1. 画像ファイル形式

### ビットマップ vs ベクトルの判断基準

| 特性 | ビットマップ（ラスタ） | ベクトル |
|------|----------------------|---------|
| 拡大時の品質 | ピクセル化・劣化 | 完全に鮮明を維持 |
| ファイルサイズ | 解像度依存 | 要素数依存 |
| 適した用途 | 写真・連続階調画像 | 図・グラフ・テキスト |
| 編集の柔軟性 | 低い | 高い |

**基本ルール**: グラフ・図解には**ベクトル形式を優先**する。写真を含む場合のみビットマップを検討する。

### 主要フォーマット比較

| 形式 | 種類 | 圧縮 | 推奨用途 |
|------|------|------|---------|
| **PDF** | ベクトル | 非該当 | 出版・印刷・最終成果物 |
| **SVG** | ベクトル | 非該当 | Web・インタラクティブ可視化 |
| **EPS** | ベクトル | 非該当 | 印刷・出版（レガシー） |
| **PNG** | ビットマップ | 可逆 | Web・スクリーン表示・テキスト含む画像 |
| **TIFF** | ビットマップ | 可逆 | 高品質印刷・アーカイブ |
| **JPEG** | ビットマップ | **非可逆** | 写真（グラフには不適） |
| **GIF** | ビットマップ | 可逆（256色） | 簡単なアニメーション |

### 可逆圧縮 vs 非可逆圧縮

- **可逆（PNG・TIFF）**: 情報損失なし。テキスト・細線・グラフに必須
- **非可逆（JPEG）**: ファイルサイズ削減の代わりに品質劣化が生じる
  - グラデーションのブロック化・テキスト周辺のノイズ・不鮮明な細部
  - 10分の1に圧縮しても知覚可能な劣化が生じ始める

### 解像度（DPI）の基本ルール

| 用途 | 推奨 DPI |
|------|---------|
| スクリーン表示 | 72–96 DPI |
| 印刷（一般） | 300 DPI |
| 高品質印刷 | 600+ DPI |

ベクトル形式（PDF・SVG）は解像度に依存しないため、**出版物には常にベクトル形式を優先**する。

### 形式変換の注意点

- JPEG → PNG に変換しても、既に生じた劣化は**回復しない**
- ベクトル → ビットマップへの変換は不可逆（逆変換不可）
- PDF は多くの場合ベクトルを内包し、高品質な中間形式として機能する

## 2. 可視化ソフトウェア選択

### 再現性と反復性の重要性

| 概念 | 定義 | 重要性 |
|------|------|--------|
| **再現性** | 同じデータ・同じコードから同じ図を生成できる | 科学的信頼性の基盤 |
| **反復性** | 乱数シードを含め完全に同一の図を再生成できる | バグ追跡・図の差し替え |

スクリプトベースのアプローチ（R/Python）は再現性・反復性を自然に担保する。GUI ツールは手順が記録されないためリスクが高い。

### データ探索 vs データ表現

| フェーズ | 目的 | 優先事項 |
|---------|------|---------|
| **EDA（探索）** | パターン発見・仮説生成 | 速度・柔軟性 |
| **Publication（表現）** | 発見を伝える | 品質・再現性・美観 |

### コンテンツとデザインの分離原則

データ変換・集計のコードと、視覚的スタイル（テーマ・色・フォント）を**分離して管理**する。これにより:
- スタイルの一括変更が容易になる
- コンテンツの再利用性が高まる
- コードレビューが効率化される

### ソフトウェア選択判断テーブル

| 目的 | 推奨アプローチ | 具体的ツール例 |
|------|--------------|--------------|
| 統計可視化・出版品質 | プログラミング言語 | R（ggplot2）、Python（matplotlib/seaborn） |
| インタラクティブ Web 可視化 | JavaScript ライブラリ | D3.js、Vega-Lite、Observable |
| ビジネスダッシュボード | BI ツール | Tableau、Power BI、Looker |
| データ探索・素早い試作 | ノートブック環境 | Jupyter + pandas、RStudio |
| 仕上げ・デザイン調整 | ベクトルエディタ（最終手段） | Illustrator、Inkscape |

## 3. データストーリーテリング

### ストーリーとは何か

効果的なデータストーリーは**4段階構造**を持つ:

```
Opening（導入）→ Challenge（課題）→ Action（展開）→ Resolution（解決）
```

- **Opening**: 読者の関心を引く文脈設定
- **Challenge**: 解決すべき問題・疑問の提示
- **Action**: データによる分析・証拠の提示
- **Resolution**: 結論・示唆・次のアクション

### 司令官に伝わる図を作る

忙しい意思決定者は図を線形に読まない。**一目で要点が伝わる**設計が必要:

- タイトルに結論を含める（「何が起きているか」ではなく「何を意味するか」）
- 最重要メッセージを視覚的に強調（色・サイズ・注釈）
- 不要な要素を全て除去する（"less is more"）
- 読者の背景知識レベルに合わせた説明量を設定する

### 図を徐々に複雑にする（段階的構築テクニック）

複雑なデータを伝える場合、**一つの図で全てを見せようとしない**:

1. まずシンプルな図で核心的なメッセージを伝える
2. 次の図でコンテキストを追加する
3. 最後の図で全体像を示す

例: 単一航空会社の遅延 → 航空会社比較 → 空港・曜日・航空会社の複合分析

### 図を覚えてもらいやすくする

棒グラフ・折れ線グラフは明快だが記憶に残りにくい。記憶定着を高めるには:

- **アイソタイプ（等量図）**: 単位量を表す画像アイコンを繰り返す視覚表現
- **ユニークなビジュアルメタファー**: データの意味と視覚形式を結びつける
- 注釈・テキスト解説でコンテキストを補完する

研究により、アイソタイプは標準的な棒グラフと比較して記憶定着率が高いことが示されている。

### 一貫性を持たせ、反復はしない

| 概念 | 定義 | 問題 |
|------|------|------|
| **一貫性** | 複数図で同じ視覚言語を使用する | ない場合は混乱を招く |
| **反復** | 複数の図で同じ可視化手法・同じ軸範囲を使う | 各図の差異が伝わりにくくなる |

複合図の各パネルが**異なるメッセージ**を持つ場合、視覚的手法を変えて差異を強調する。

### 過剰に複雑な図を分解する

一つの図に詰め込みすぎると、多くの読者が理解できなくなる。目安: **図を1〜2文で説明できない場合は複雑すぎる**。

**分解の手順:**

| ステップ | 操作 | 目的 |
|---------|------|------|
| 1 | 核心メッセージを1つだけ選ぶ | 伝えたいことを明確にする |
| 2 | そのメッセージに必要な変数だけを残す | 不要な次元を除去する |
| 3 | 補足メッセージを別の図に分ける | 読者の認知負荷を下げる |
| 4 | 各図のタイトルでメッセージを宣言する | 読者が迷わないようにする |

### 一貫性と反復の使い分け

複数パネルの複合図では、「同じ視覚言語」と「異なる可視化手法」を使い分ける:

| 概念 | 意味 | 具体例 |
|------|------|--------|
| **一貫性** | 同じグループには全パネルで同じ色・形を使う | 女性=赤、男性=青を全図で統一 |
| **反復を避ける** | 異なるメッセージのパネルでは異なる可視化手法を使う | (a)折れ線グラフ → (b)棒グラフ |

同じ棒グラフを横並びにすると「全部同じように見える」ため差異が埋もれる。パネルごとにメッセージが異なるなら、可視化手法も変える。

### ストーリーテリングチェックリスト

- [ ] ストーリーの4段階構造（Opening → Resolution）が明確か
- [ ] タイトルに結論・発見が含まれているか
- [ ] 最重要メッセージが視覚的に際立っているか
- [ ] 不要な情報・過剰な複雑さを除去したか
- [ ] 段階的に複雑さを増す構成になっているか（複数図の場合）
- [ ] 一貫した視覚言語（色・形・スケール）を使用しているか
- [ ] 同じ可視化手法を無意味に反復していないか
- [ ] 対象読者の知識レベルに合わせた説明量になっているか
