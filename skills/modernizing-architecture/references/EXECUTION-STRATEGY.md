# 実行戦略（Execution Strategy）

Architecture Modernization Enabling Team (AMET)、Strategy Deck、Roadmap、Learning & Upskillingをカバーする。

---

## 1. Architecture Modernization Enabling Team (AMET)

### AMETの6つの主要責任

| 課題 | AMETの責任 |
|------|----------|
| 分析麻痺で動けない | **Kickstart** — モダナイゼーションを始動 |
| BAU作業に押されて進まない | **Sustain Momentum** — モメンタムを維持 |
| 現代的な設計知識が不足 | **Facilitate Better Design** — より良い設計をファシリテート |
| 新アプローチが定着しない | **Facilitate Durable Change** — 持続的な変化を促進 |
| 外部の人が価値を理解しない | **Communicate Vision** — ビジョンと進捗を発信 |
| ある領域の学びが他に伝わらない | **Promote Learnings** — 成功事例と学びを展開 |

### AMETのアンチパターン

| アンチパターン | 正しい姿 |
|-------------|---------|
| 「モダナイゼーションチーム」になる | **支援者**であり、実行者は各チーム |
| 中央集権的なアーキテクトチーム | **Enabler**として各チームの能力を向上 |
| 設計を手渡して終わり | 継続的な**コーチング**と**ファシリテーション** |

### AMETのライフサイクル

```
Phase 1: Kickstart（高関与）
  → リスニングツアー、ビジョン策定、最初のスライス

Phase 2: Sustain & Scale（中〜高関与）
  → ワークショップファシリテーション、チームコーチング
  → 成功事例の展開、プレイブック作成

Phase 3: Embed & Withdraw（低〜中関与）
  → Architecture Operating Modelの確立
  → 組織能力の定着確認
  → AMETの縮小・解散
```

### Architecture Operating Model

AMETがなくなった後も持続するアーキテクチャの運営モデル:

| コンポーネント | 説明 |
|-------------|------|
| **Architecture Advisory Forum** | 横断的なアーキテクチャ判断の場 |
| **Architecture Decision Records (ADR)** | 重要な決定の記録と共有 |
| **Communities of Practice** | アーキテクチャプラクティスの継続的改善 |
| **Regular Architecture Reviews** | 定期的なアーキテクチャの健全性評価 |

### AMETの人材要件

| スキル | 説明 |
|--------|------|
| **忍耐力と関係構築力** | 変化は時間がかかる。ステークホルダーとの信頼構築が鍵 |
| **ファシリテーション能力** | ワークショップの設計・進行 |
| **技術力とビジネス理解** | 両方の言語を話せること |
| **コーチングスキル** | 答えを与えるのではなく、チームが自ら見つけるのを支援 |
| **適応力** | リード、ファシリテート、一歩引く、の切り替え |

## 2. Strategy Deck の構成

モダナイゼーションの説得力あるナラティブを構築する:

### 4つの基本要素

```
1. What are we trying to achieve?（何を達成したいか）
   → ビジネス目標、North Star、成長戦略

2. What's holding us back?（何が妨げているか）
   → レガシーの課題、組織的障壁、技術的負債

3. How will modernization help?（モダナイゼーションがどう助けるか）
   → 具体的な改善計画、Before/After

4. When and how will we do it?（いつ、どう実行するか）
   → ロードマップ、マイルストーン、投資計画
```

### Strategy Deckのコンテンツ要素

| セクション | 内容 | 効果 |
|-----------|------|------|
| **Business Context** | 市場動向、競合分析、顧客ニーズ | ビジネスの緊急性を示す |
| **Current Challenges** | 技術的負債、パフォーマンス問題、チームの苦痛 | 共感を得る |
| **Vision** | モダナイゼーション後の理想像 | インスピレーション |
| **Strategy** | 具体的なアプローチとフレームワーク | 実現可能性を示す |
| **Roadmap** | フェーズ分けされた計画 | 段階的な進行を示す |
| **Investment Case** | ROI予測、リスク分析 | 経営判断の根拠 |
| **People Quotes** | チームメンバーの声、顧客フィードバック | 感情的なつながり |
| **Success Stories** | 類似企業の成功事例 | 信頼性の向上 |

## 3. Nail It Then Scale It

### Phase 1: Think Big（大きく考える）

- 全ステークホルダーを巻き込む説得力あるビジョンの構築
- 技術用語だけでなく、**ビジネス言語**で語る
- 個人の声やフィードバックで**感情的なつながり**を作る

### Phase 2: Nail It（釘を打つ）

最初のスライスを3-6ヶ月以内に届ける:

#### どこから始めるか

| 選択基準 | 説明 |
|---------|------|
| **ビジネスインパクト** | 高いビジネス価値を早期に示せる領域 |
| **技術的実現性** | 比較的リスクが低い領域 |
| **チームの準備度** | モチベーションとスキルがある領域 |
| **Learning Potential** | 組織全体に応用可能な学びを得られる領域 |
| **Visibility** | 成功が目に見えやすい領域 |

#### 最初のスライスの計画

- [ ] ドメインの選定と境界の定義
- [ ] チームの編成
- [ ] 技術スタックの決定
- [ ] MVP（Minimum Viable Product）の定義
- [ ] 成功基準の明確化
- [ ] リスクの特定と対策
- [ ] 3-6ヶ月のスプリント計画

### Phase 3: Scale It（スケールする）

#### Playbooks

成功パターンをPlaybookとして文書化し、組織全体に展開:

```
Playbook の構成:
├── コンテキスト（どのような状況で使うか）
├── アプローチ（具体的な手順）
├── テクニック（使用するワークショップ、ツール）
├── チェックリスト（完了基準）
├── 落とし穴（避けるべきアンチパターン）
└── ケーススタディ（実際の適用例）
```

#### モダナイゼーション作業の順序付け

| 基準 | 考慮事項 |
|------|---------|
| **依存関係** | 基盤サービスを先に移行 |
| **リスク** | 低リスクの領域で学びを得てから高リスクへ |
| **ビジネス価値** | 最大の価値を提供する領域を優先 |
| **技術的制約** | Strangler Fig等、段階的移行が可能か |

#### 発見・設計・デリバリーのバランス

```
         発見（Discovery）
         ╱        ╲
      設計        デリバリー
     (Design)    (Delivery)
```

- **発見過多**: 分析麻痺。実装が進まない
- **設計過少**: 技術的負債の再生産
- **デリバリー過多**: 方向性を見失う

### バランシングの原則

| 段階 | Discovery | Design | Delivery |
|------|-----------|--------|----------|
| **初期** | 高 | 中 | 低 |
| **中期** | 中 | 中 | 中 |
| **安定期** | 低 | 低 | 高 |

### モダナイゼーションと他の作業のバランス

| アプローチ | 説明 | 適用場面 |
|-----------|------|---------|
| **Dedicated Teams** | モダナイゼーション専任チーム | 大規模、集中的な移行が必要 |
| **Time Allocation** | 各チームが一定時間をモダナイゼーションに | 分散的、継続的な改善 |
| **Boy Scout Rule** | 触れるコードを少しずつ改善 | 小規模で継続的な改善 |
| **Opportunistic** | 機能開発と一緒にモダナイズ | リソースが限られる場合 |

## 4. 継続的な評価と適応

### メトリクス

| カテゴリ | メトリクス例 |
|---------|-----------|
| **デリバリー** | デプロイ頻度、リードタイム、変更失敗率、復旧時間（DORA 4 keys） |
| **チーム** | 認知負荷スコア、チーム満足度、オンボーディング時間 |
| **アーキテクチャ** | 結合度、テストカバレッジ、技術的負債の推移 |
| **ビジネス** | Time-to-market、顧客満足度、収益への影響 |

### Pulse Surveys

短い定期的なサーベイでモダナイゼーションの健全性を測定:

- モダナイゼーションの方向性への信頼度
- 個人レベルでの影響（ポジティブ/ネガティブ）
- サポートの十分さ
- ブロッカーの有無

### Gatherings

定期的な集まりでアライメントを維持:

| タイプ | 頻度 | 目的 |
|--------|------|------|
| **Weekly Sync** | 週次 | 短期的な進捗と障害の共有 |
| **Monthly Review** | 月次 | メトリクスのレビュー、戦略の微調整 |
| **Quarterly Planning** | 四半期 | ロードマップの更新、優先順位の再評価 |
| **Annual Strategy** | 年次 | ビジョンの見直し、大きな方向転換 |

## 5. Learning & Upskilling

### 種まき（Planting Seeds）

新しいアイデアや手法を組織に紹介する初期段階:

- 小さな成功事例を作り、興味を引く
- 押し付けではなく、自然な好奇心を喚起
- 共感する仲間（Allies）を見つける

### プロジェクトニーズへのUpskilling

- モダナイゼーションに必要なスキルを事前に特定
- ジャストインタイムの学習（使う直前に学ぶ）
- ペアプログラミング、モブプログラミングで実践的に学ぶ

### 継続的学習環境の構築

| 手法 | 説明 | 効果 |
|------|------|------|
| **Communities of Practice** | 共通の関心を持つ人の横断的コミュニティ | 知識共有の文化 |
| **Small Learning Opportunities** | Tech Talks、Book Clubs、Coding Dojos | 日常的な学習習慣 |
| **Mentoring** | 経験者が後進を指導 | 深い知識の伝達 |
| **Empowering Influencers** | 影響力のある人を巻き込む | 組織的な変化の加速 |
| **Blogging & Speaking** | 社内外での発信 | 知識の定着、外部からの評価 |
| **Internal Conferences** | 社内カンファレンス | 大規模な知識共有 |
