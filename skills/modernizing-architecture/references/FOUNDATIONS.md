# 基礎・準備（Foundations）

アーキテクチャモダナイゼーションの出発点。ビジネス目的の明確化、組織の準備状態の評価、ステークホルダーとの対話を扱う。

---

## 1. アーキテクチャモダナイゼーションとは

### 定義

時代遅れのアーキテクチャ（ビジネスの負債）を、競争優位を提供する現代的アーキテクチャに変換すること。

### なぜ今重要か

- ソフトウェアがビジネスの中核を担う時代、アーキテクチャの重要性は増大し続ける
- モダナイゼーションは短期的な妥協を伴うが、先延ばしは**負のスパイラル**を生む:
  - 遅延 → 競合に後れ → さらなる技術的負債 → さらなる遅延

### Better Value Sooner Safer Happier (BVSSH)

モダナイゼーションが提供すべき価値の指針:

| 要素 | 意味 |
|------|------|
| **Better** | より良い品質の成果 |
| **Value** | ビジネス価値に直結 |
| **Sooner** | より早く市場に投入 |
| **Safer** | リスクを低減 |
| **Happier** | チームと顧客の満足度向上 |

## 2. 組織の準備状態を評価する

モダナイゼーション開始前に、リーダーシップに対して以下を確認する:

### リーダーシップ準備チェックリスト

- [ ] **機能開発の減速を受け入れる覚悟**: モダナイゼーション期間中、新機能の開発ペースは一時的に落ちる
- [ ] **レガシーの複雑さへの理解**: レガシーシステムと働き方は複雑で変更が困難であることを理解している
- [ ] **予期せぬ事態への対応力**: 遅延やコスト増は不可避。パニックにならず対応できるか
- [ ] **自らの働き方を変える準備**: 資金モデル、作業優先順位、開発プロセスの変更を支持できるか
- [ ] **学習への投資意欲**: 全従業員への研修に十分な時間と資金を投入する覚悟
- [ ] **ビジネス価値の説明力**: 技術者がビジネスリーダーにモダナイゼーションの事業的メリットを説明できるか

### マインドセット変革

| 旧来のマインドセット | 新しいマインドセット |
|-------------------|-------------------|
| アーキテクチャは技術部門の仕事 | アーキテクチャはSocio-technicalな関心事 |
| アイボリータワー・アーキテクチャ | 協調的アーキテクチャプラクティス |
| ビジネスとITは別々のサイロ | ビジネスとITの垣根を超える |
| アーキテクチャと戦略は無関係 | アーキテクチャと戦略を接続 |

### 銀の弾丸への警戒

- **ボルトオン・モダナイゼーション**: 根本的な問題を解決せず表面的なツール導入だけで済ませる危険
- **構造とプロセスの誤謬**: 組織図やプロセスを変えるだけで問題が解決するという錯覚
- **技術的プラクティスへの投資**: 良い技術的プラクティス（CI/CD、テスト、コードレビュー等）なしにモダナイゼーションは成功しない

## 3. ビジネス目的の明確化

### モダナイゼーションのビジネス正当化

| 動機 | 説明 | 例 |
|------|------|-----|
| **競合に後れを取る** | 競合がより速く動いている | デリバリー速度の差が顧客流出を引き起こす |
| **ビジネス成長の阻害** | アーキテクチャが新市場・新製品への展開を妨げる | 新しい国への展開にシステムが対応できない |
| **Exit戦略** | 売却・IPO準備 | レガシーは企業価値を大幅に下げる |
| **買収による成長** | 買収企業のシステム統合 | M&A後のシステム統合コスト |
| **UXの劣化** | 顧客体験が競合に劣る | モノリスがUX改善のボトルネック |
| **内部ツールの非効率** | 社内プロセスが非効率 | 手動作業が多くオペレーションコスト高 |
| **採用・定着の改善** | 技術的魅力の欠如 | レガシー技術で優秀なエンジニアが集まらない |

### 成長戦略との接続

モダナイゼーションをAnsoffマトリクスの成長戦略に接続する:

| 成長戦略 | アーキテクチャの役割 |
|---------|-------------------|
| **Product Development** | 新機能を迅速に開発・デプロイできるアーキテクチャ |
| **Market Penetration** | スケーラビリティとパフォーマンスの最適化 |
| **Market Development** | 新市場（地域、チャネル）への柔軟な対応 |
| **Diversification** | 新しいビジネスモデルを支えるプラットフォーム |

### North Star（北極星）の設定

モダナイゼーションの指針となる最終目標:

- **正しいNorth Starの条件**:
  - ビジネス成果に直結している（技術メトリクスだけでは不十分）
  - 測定可能
  - 全ステークホルダーが理解できる
  - 長期的な方向性を示す

- **North Star Frameworkの構成要素**:
  1. **North Star Metric**: 1つの主要指標
  2. **Input Metrics**: North Starに貢献するサブ指標群
  3. **Work**: メトリクスを改善する具体的な作業

## 4. リスニング＆マッピングツアー

モダナイゼーション計画の前に、組織全体の課題と機会を発見する探索活動。

### 誰と会うか

- ビジネスリーダー・プロダクトリーダー
- エンジニアリングチーム
- 運用・サポートチーム
- エンドユーザー（可能な場合）
- 外部パートナー

### 効果的なツアーの実施

| 原則 | 説明 |
|------|------|
| **安全な空間の創出** | 参加者が率直に問題を共有できる環境 |
| **技法のツールボックス活用** | 1つの技法に固執せず、状況に応じて複数の手法を使い分ける |
| **構造化 vs 非構造化** | 準備されたアジェンダと自由な対話のバランス |

### Kickstarter Workshop

リスニングツアー後に実施する集中ワークショップ:

1. ビジョンの共有と整合
2. ドメインの探索（EventStorming等）
3. 3-6ヶ月以内の最初のスライスの具体的計画策定
4. チーム編成とロードマップの初期設計

## リーダーの各レベルでの支援

モダナイゼーションはトップダウンとボトムアップの両方が必要:

```
エグゼクティブ    → ビジョン、資金、組織的コミットメント
ミドルマネジメント → 日々の優先順位調整、チーム間調整
チームリーダー    → 技術的判断、実装品質、チームのモチベーション
個人貢献者       → ドメイン知識、実装スキル、フィードバック
```

各レベルのリーダーがモダナイゼーションの価値を理解し、推進する力を持つことが成功の鍵。
