# スキルマップ設計・OJTリファレンス

STEP 2-2: OJTとしてのスキルマップ策定の実践手順。
スキルマップの目的・5プロセスによる設計から、運用・活用・生成AI活用まで包括的に解説。

---

## スキルマップとは

業務に必要とされるスキルを洗い出し、従業員がそれを遂行するスキルを保有しているかどうかを可視化するためのツール。

**別称**: スキルズインベントリー、スキルマトリックス

**発祥**: 日本の大手製造業の生産現場。経済産業省・厚生労働省も推奨。

**ツールの形式**: Excel等の表計算ソフトで作成可能。専用システムやタレントマネジメントシステムの一部としても提供されている。

**策定期間**: 5つのプロセス全体を進めるには通常3〜4か月程度が必要。

> **前提**: スキルマップ策定には、STEP 1で解説した「求める人材像」の策定が完了していること。
> 未完了の場合は NEEDS-ANALYSIS.md に戻って策定・抽出を済ませること。

---

## スキルマップの4つの目的

### 主目的（必須）

**育成**: 人材育成を計画的・効率的に進めるためのツール。
スキルマップの本旨。まずここから始める。

### 付随的な目的（プラスアルファ）

| 目的 | 内容 |
|------|------|
| 採用 | スキルマップの項目を採用時の選考基準に流用。特に中途採用で活きる |
| 配置 | 部門内のスキルレベルのばらつきを可視化し、配置転換・昇格の参考資料に活用 |
| ケイパビリティ | 社員全員のスキルを総合して、会社としての強い/弱いスキルを把握。戦略立案の基礎情報 |

> **推奨**: 最初は「育成」のみに絞ったスモールスタートが最善。
> 他の要素は運用に慣れてから段階的に拡大する。

```
    ↓
⑤ 評価方法の確定・マニュアル作成
```

## プロセス①：スキルマップ範囲の設定

4つの観点で範囲を決定する。

### Why：何のためにスキルマップをつくるか

「育成」「採用」「配置」「ケイパビリティ」のうち、どの目的を対象とするかを決定。

> **推奨**: 最初は「育成」のみ。スモールスタートで形骸化を防止。

### Who：どの階層・職種を対象とするか

**推奨**: 全社一斉導入より、**1〜2部門への導入でスタート**。

| 選択基準 | 推奨アプローチ |
|---------|-------------|
| 対象部門 | 特に強化すべき職種がある部門、スキルの重要性が高い部門を優先 |
| 対象階層 | 優先的に強化すべき階層から始める |

### What：スキル・スタンス・ナレッジのどれを扱うか

**推奨構成**

| 優先度 | 要素 | 備考 |
|-------|------|------|
| 必須 | ベーシックスキル | 職種を問わず求められるスキル |
| 推奨 | テクニカルスキル | 職種固有の専門スキル |
| 任意 | スタンス・ナレッジ | 必要に応じて追加 |

> **注意**: 何でも入れすぎると「タレントマネジメントカルテ」になりスキルマップの主旨がぼやける。
> 項目数は10〜30項目に制限することを厳守。

### How：人事評価と紐づけるか

> **AskUserQuestion**: 人事評価との紐づけは重要な判断分岐。
> 以下のメリット・デメリットをユーザーに説明した上で、方針を確認する。

| 方針 | メリット | デメリット |
|------|---------|----------|
| 人事評価に紐づける | 運用が確実に行われる | 評価に「調整弁」として使われ、点数に歪みが生じやすい |
| 人事評価に紐づけない | 純粋に育成・スキル向上に活用できる | 現場で軽視されて形骸化しやすい |

**スキルを評価するパターン（4種類）**

| パターン | 内容 |
|---------|------|
| パターン1 | 能力開発のみ。上司フィードバックのコミュニケーションツールとして使用 |
| パターン2 | 人事評価に反映しないが、高スキル者を表彰して動機づけ |
| パターン3 | スキルマップの点数を基本給に反映（リスク大） |
| パターン4 | スキルマップの点数を昇降格に反映 |

> **推奨**: パターン1または2。基本給への反映（パターン3）は引き下げが困難なためリスクが大きい。

## プロセス②：業務の洗い出し

スキルを洗い出す前提として、部門・職種ごとに業務を細かいプロセスに分解する。

### なぜ業務分解が必要か

「営業職に必要なスキル」では漠然としすぎて評点をつけられない。
業務を細かいプロセスに分解することで、具体的なスキルが洗い出せる。

### ロジックツリーによる業務分解

ロジックツリー: 考える対象の全体像を捉えて、抜け・漏れなく構成要素を分類・整理するツール。

**MECE（Mutually Exclusive and Collectively Exhaustive）の原則**

洗い出された要素が「漏れがなく、ダブりもない」状態であること。

| 状態 | 例 | 評価 |
|------|-----|------|
| MECE ✓ | 移動手段：「陸路、海路、空路」 | 漏れなし・ダブりなし |
| 漏れあり ✗ | 「車、船、飛行機」（鉄道・自転車が漏れ） | MECE でない |
| ダブりあり ✗ | 「陸路、海路、空路、徒歩」（陸路と徒歩がダブる） | MECE でない |

### 業務プロセス分解の例

**営業職の場合（プロセス別）**
```
顧客リスト作成 → 関係構築 → ヒアリング → 提案 → クロージング
```

**ITエンジニアの場合（プロセス別）**
```
現状分析 → 要件定義 → 詳細設計 → テスト・検証 → 導入 → フォロー
```

**マネジメント業務の場合（マトリックス別）**
```
       │ 短期  │ 長期  │
───────┼───────┼───────┤
人の管理│ 日常的 │ 育成  │
       │ 指導  │ 計画  │
───────┼───────┼───────┤
仕事の  │ タスク │ 目標  │
管理   │ 管理  │ 設定  │
```

## プロセス③：アンケート・インタビュー実施

スキルの洗い出しは現場の協力を得ながら進める。

### 現場を巻き込む重要性

単なる情報収集だけでなく、「現場の声を採り入れた」という手続きを踏むことで、
運用段階での現場からのクレーム・非協力を防止する根拠となる。

### アンケートとインタビューの使い分け

| 方法 | 向いている場面 | 特徴 |
|------|-------------|------|
| アンケート | 対象者・職種が多い場合 | 効率的に収集可能だが信憑性はやや劣る。記名式推奨 |
| インタビュー | 人事担当者の現場知識が少ない場合 | 深く細かい情報が得られるが時間・手間がかかる |

**インタビュー対象者（推奨）**
- 部門長
- ハイパフォーマーの現場担当者
- ネガティブな意見の人・主張したがる声の大きな人（後のトラブル防止のため）

**主要な質問項目例**

| カテゴリ | 質問例 |
|---------|------|
| 業務内容 | 「どんな業務プロセスを行っていますか？」「重要度の高い業務は何ですか？」 |
| スキル・知識 | 「業務に必要なスキル・知識は何ですか？」「高パフォーマーと低パフォーマーの違いは？」 |
| 課題 | 「現在不足していると感じるスキルは？」「これから必要になるスキルは？」 |

> **注意**: 「これから必要となる」新規戦略・事業に関わるスキルは現場へのアンケートでは調査不可。
> 必ずインタビューでヒアリングすること。

## プロセス④：業務ごとのスキル・基準作成

### スキル項目の選定と名称設定

業務プロセスに対応するスキルを選定して名称をつける。

**営業職のスキル項目例**

| 業務プロセス | 下位プロセス | スキル項目名 |
|------------|-----------|------------|
| 顧客リスト作成 | 母集団形成 | 母集団形成力 |
| | リスト情報管理 | リスト情報管理力 |
| 顧客リレーション構築 | 人への対応 | 個人関係構築力 |
| | 組織への対応 | 組織横断コミュニケーション力 |

### スキル項目数の設定（厳守）

| 範囲 | 推奨 |
|------|------|
| 最小 | 10項目程度 |
| 最大 | **30項目以下（絶対厳守）** |

**項目数が30を超えた場合の絞り込み方法**

各スキル項目を以下の2軸でマトリックス評価し、高得点の上位30項目を採用：
1. 事業戦略推進上の重要性（高・中・低）
2. 従業員の不足感（高・中・低）

### スキル定義文の作成

**重要**: スキル定義文は評価が可能な具体的な内容でなければならない。

**良い定義文の条件**
- 確認の観点が **2〜3個** 含まれている
- 観点が多すぎず（4〜5個でも複雑すぎる）、少なすぎない（1個だと難度が低すぎる）

**定義文の例（2つの観点を含む）**
```
スキル名: 市場リサーチスキル

定義:
① リサーチの目的を意識して調査設計ができる
② 最終的にどのようなアウトプットで顧客のビジネスに活用されるかを想像できる

評価方法:
- 両方十分 → 5点
- 一方が十分、一方が不十分 → 3点
- 両方不十分 → 1点
```

### スキル定義作成の2つのコツ

**コツ1: 公開されているテンプレートを活用**
- 厚生労働省「職業能力評価基準」（業種・職種ごとに用意）
- IPA「デジタルスキル標準」（IT系職種向け）

**コツ2: 生成AIをたたき台として活用（COLUMN参照）**

### プロトタイプ段階でのテスト運用

スキルマップがある程度完成したら、現場の部門長・ハイパフォーマーにテスト運用を依頼。

**確認ポイント**
- スキル項目の粒度・項目数の妥当性
- スキル定義文のわかりやすさ
- 運用の負荷感

## プロセス⑤：評価方法の確定・マニュアル作成

### 運用ルールの5項目

#### 1. 評価段階

推奨: **4〜5段階**

| 段階数 | 評価 |
|------|------|
| 3段階以下 | 能力開発の差分・進度が伝わりにくい |
| 4〜5段階 | 標準的で適切 |
| 6段階以上 | 評価段階ごとの差の根拠が明確化しにくい |

> 段階名称（新人・白帯・ビギナー等）は社風に合わせて任意に設定。

#### 2. 評価シート項目

標準的な3項目構成：

| 項目 | 内容 |
|------|------|
| ①前年度（前期）評価 | 昨年度の自己評価・上司評価。今年度の向上度合いの基準 |
| ②自己評価 | 被評価者本人が向上したと感じる項目の評価段階を上げる |
| ③上司評価 | 上司から見て向上があった項目の評価段階を上げる |

> **注意**: 自己評価は原則として入れることを推奨。上司・部下の相互コミュニケーションに有益。

#### 3. 評価者

通常は**直属の上長**。チームリーダーに任せる場合もある。

#### 4. 面談頻度

| 頻度 | 推奨度 | 理由 |
|------|-------|------|
| 月次 | ✗ | 変化が少なく手間が大きい |
| 四半期 | ◎ | 推奨（標準） |
| 半年 | ○ | 最長でも半年 |
| 年次 | ✗ | 期間が空きすぎて効果が薄れる |

#### 5. 評価との紐づけ

プロセス①の「How: 人事評価と紐づけるか」で決定した方針に従って設定。

### スキルマップの管理・修正ルール

業務に必要なスキルは時間とともに変化するため、定期的な見直しが必要。

> **推奨更新頻度**: 年1回
> （半年は短すぎて手間大、2年は忘れがち。年1回がちょうどよい経験則）

**見直し時の手順**
1. マネジメント会議・部門会議での議題化
2. 現場へのヒアリングで更新すべきスキルを収集
3. スキル項目の追加・削除・定義変更を実施

### スキルマップマニュアルの作成

**標準的な目次構成**

```
1. スキルマップとは
2. スキルマップの役割
3. スキルマップと人事評価の関係性（重要：丁寧に説明）
4. スキルマップ化する人材タイプ（求める人材像）
5. スキルマップの構造
6. 評価基準
7. 運用プロセス
8. 運用フロー
9. 面談手順
10. 評価時の注意事項
```

## STEP 3: 研修体系とスキルマップの運用

### 制度3割、運用7割

「仏作って魂入れず」にならないよう、運用・活用こそが人事担当者の本領発揮の場。
せっかく構築した制度を形骸化させないための運用・活用方法が重要。

---

## スキルマップ運用の6つのポイント

### ポイント1: 最初は小さく始める（スモールスタート）

- 施策目的を「育成」のみに絞る
- 対象は1〜2部門からスタート
- スキル項目数を少なく設定

> 多くの企業がこれを行わずに失敗している。

### ポイント2: 人事評価面談と一緒に行う

スキルマップ面談を独立した日時で実施せず、人事評価面談に組み込む。
評価面談のうち5〜10分程度をスキルマップ面談に充てるのが推奨。

### ポイント3: スキル開発目標と成長計画をつくる

スキルマップをスキル開発目標・人材の成長計画と関連づけて活用。
評価シートに開発目標・成長計画の記載欄を設ける。

**Will・Can・Mustフレームワークとの連携**

| 要素 | スキルマップとの関連 |
|------|-----------------|
| Can（できること） | スキルマップで現在身についている項目がそのまま当てはまる |
| Must（すべきこと） | 上司から向上を求められたスキルマップの項目が対応する |
| Will（したいこと） | Can・Mustを将来のありたい姿Willと関連させてキャリア目標に組み込む |

### ポイント4: マネージャーのフィードバックスキルを強化する

スキルマップは人材育成のコミュニケーションツール。
マネージャーの適切なフィードバックがあって初めて効果が期待できる。

**マネージャーフィードバックスキル強化の3点**
1. 日常的な行動観察記録の蓄積（評価の根拠となる具体的な事実記録）
2. 正しい面談の手順を踏む（各ステップで手抜きをしない）
3. 評価エラーへの理解と対策（ハロー効果・中心化傾向・寛大化傾向・対比誤差・直近効果）

### ポイント5: 研修体系と紐づける

70：20：10の法則（経験学習サイクル）にあるように、Off-JTの研修とOJTのスキルマップを紐づけて活用することで育成効果が最大化。

**対応表の作成**

| 人事評価の評価項目 | スキルマップのスキル項目 | おすすめ研修 |
|-----------------|---------------------|------------|
| 問題解決力（評価項目A） | ロジカルシンキングスキル（項目2） | 研修X |
| チームマネジメント（評価項目B） | メンバー育成スキル（項目5） | 研修Y |

### ポイント6: スキルマップを定期更新する

求められる業務スキルは時間によって変化する。現在の業務実態に即したスキルマップになるよう年1回アップデート。

---

## よくある悩みと解決策

| 悩み | 原因 | 解決策（ポイント番号） |
|------|------|------------------|
| 運用負荷が高すぎて形骸化した | 複雑すぎる設計・独立した運用 | ポイント1（スモールスタート）＋ポイント2（評価面談と統合） |
| 現場のモチベーションが高まらない | スキルとキャリアが結びついていない | ポイント3（WCMフレームワーク連携） |
| 管理職がスキルを正しく評価できない | フィードバックスキルの不足 | ポイント4（フィードバックスキル強化） |
| スキルマップと人事評価項目がバラバラ | 制度間の紐づけがない | ポイント5（研修体系との紐づけ・対応表作成） |
| スキルマップのスキルが経年劣化 | 定期更新の仕組みがない | ポイント6（年1回定期更新） |

## COLUMN: スキルマップ作成への生成AI活用

### 活用方法

スキル項目の洗い出し・定義作成に生成AIを「たたき台」として活用。

**ステップ例（ITエンジニア職のスキルマップ作成）**

```
プロンプト1: 「ITエンジニアに必要な職能要件を具体的に洗い出せ」
    ↓
生成AI: 主要な機能要件を洗い出して提示

プロンプト2: 「各職能要件を細分化して、必要スキルと評価基準を表形式で整理せよ」
    ↓
生成AI: スキルが細分化された「AI版スキルマップ」を作成

この「たたき台」を自社に合わせて調整
    ↓
完成（ゼロから作るより格段に効率的）
```

**他の活用例**
- 等級定義・評価定義のたたき台作成
- 評価段階の名称案（「エキスパート」「マスター」「ビギナー」等）を生成

### 利用時の注意点

| 注意点 | 内容 |
|------|------|
| 内容の正確性 | 誤情報・古いデータに基づく回答の可能性。必ず内容を確認する |
| 著作権 | 他者の著作権を侵害する内容が含まれる可能性 |
| 機密情報 | 無料・低額プランでは入力データが学習に使われる可能性。社外秘情報・個人情報は入力しない |

## タレントマネジメントへの発展

スキルマップの活用が広がったら、「育成」以外の目的（ケイパビリティ・配置・採用）にも活用。

**タレントマネジメントシステムとの統合**

従業員一人ひとりの以下の情報を一元管理：
- スタンス・スキル・ナレッジ
- 経験・行動履歴
- スキルマップの評価履歴

「人材カルテ」として可視化し、人的資本の活用・開発を効率化。

---

## 育成制度と成長意欲の関係

**重要**: 育成制度を整えても、本人に成長意欲がなければ大きな成長は望めない。

育成制度はあくまで「成長のための環境整備」。
成長意欲を高めるためには、人事制度全体（等級・評価・報酬・ミドルマネジメント）を通じた整合性が不可欠。

**整合性の重要例**
- 研修で「新しいチャレンジへの成長」を促す一方で、評価制度が「ミスの少ない減点主義」では矛盾
- 研修で「ロジカルシンキング」を教える一方で、現場上司が「営業は足で稼げ」では定着しない

すべての人事施策が整合性をもって連動し、全体として最適化されていることが大切。
