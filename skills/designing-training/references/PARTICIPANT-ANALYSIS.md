# 研修参加者分析ガイド

効果的な研修設計のために参加者を多角的に理解する方法。
ステークホルダー分析・4観点（知識・興味・言語・権限）による参加者分析から、
案内文設計・上司の巻き込みまでの実践的な手順を提供する。

---

## 1. なぜ参加者分析が必要か

### 本人調査だけでは不十分な理由

参加者本人がニーズを自己申告するアンケートは一般的だが、それだけでは偏りが生まれる。

**たとえば、** カーリングを見たことはあっても経験がない人が「腕を鍛える必要がありそう」とイメージできても、具体的にどんな技術をどういう段階で身につけるかはわからない。コーチに上達までの道筋を提案してもらう必要がある。

研修においても同様：
- 本人が気づいていないスキルギャップがある
- 組織が求める人材像と本人のニーズが一致しないことがある
- 解決策として「研修」が最適かどうかは俯瞰的な分析が必要

---

## 2. ステークホルダー分析（3本脚の椅子）

### 「送られる人」「送る人」「払う人」の3者

3本脚の椅子は1本や2本では成立しない。研修のニーズ分析も同様に、最低3つのグループへの調査が必要。

| 呼び名 | 誰か | 調査すること |
|-------|------|------------|
| **送られる人** | 研修参加者本人 | 現状の課題・学びたいこと・不足していると感じるスキル |
| **送る人** | 参加者の直属上司 | 参加者に何を学んでほしいか・研修に何を期待するか |
| **払う人** | 研修費用の予算権限者（部門長・事業部長・社長等） | 組織の方向性・優先課題・投資対効果への期待 |

### 調査対象の拡大

研修内容・対象者によっては3者以外にも調査が必要な場合がある。

```
課長対象研修の例:
  本人（課長）
    ↑ 1ランク上
  部長（上司）← 調査対象
    ↓ 1ランク下
  部下 ← 調査対象
```

その他の対象候補：参加者の部下、同じチームのメンバー、他部署のメンバー、顧客などのステークホルダー

### 調査の2原則

| 原則 | 内容 |
|------|------|
| **原則1（調査対象）** | 本人のほかに少なくとも2人以上の対象を設ける |
| **原則2（調査方法）** | 同じ調査方法を繰り返すのではなく、同じ状況について異なる手法を用いて調査・分析する |

### 研修は答えではないかもしれない

複数の対象者への複数の調査方法で情報収集した結果を分析する際、必ず以下の視点を持つ。

> **「研修は答えではないかもしれない」**

**例:** 「クロージングスキル研修が必要」というニーズが上がっても、データ分析すると訪問回数が減少しており、原因が社内ミーティングの増加だとわかる場合→ まず業務効率化の対策が必要で、研修は二次的な施策かもしれない。

## 3. 参加者分析の4つの観点

ニーズ分析で研修を行うことが確定したら、次は参加者を以下の4観点で分析する。

### 観点①：知識レベル

#### 現状と研修後のレベルを見極める

- 参加者の現状の知識・スキルレベルを把握する
- 研修後に達成すべきレベルを定義する
- 両者のギャップが「研修で埋める対象」

**重要な注意点：** 参加者がすでに持っている知識・過去の経験を過小評価しない。

```
過小評価の典型的な発言:
  「確実に理解してもらいたいので、基本的なことから丁寧に説明する」
  ↓
参加者の受け取り方:
  「すでに知っていることの説明を長々と聞かされる」
  → 退屈・苦痛 → 研修へのネガティブな印象
```

**知識の再活用:** 部下育成研修では「部下として上司と接してきた経験」、接客研修では「顧客として受けた接客体験」を起点にできる。ゼロベースで教える必要はない。

#### 知識レベルの確認方法

**事前確認（理想）:**

| 方法 | 内容 |
|------|------|
| 事前アンケート | 研修で取り扱う項目ごとの習熟度を自己分析してもらう |
| 上司へのヒアリング | ニーズ分析段階で収集した情報から推測 |

**研修当日の確認（現実的な対応）:**

| 方法 | 特徴 |
|------|------|
| 研修はじめにテストを行う | 客観的な把握が可能 |
| 挙手や実演で確認 | 簡易だが、参加者の自尊心に配慮が必要 |
| 課題に取り組んでもらい反応を見る | EATアプローチとセットで活用 |

#### 知識レベルの見極めが難しい場合の対処法

参加者の知識レベルが多様で「どこに焦点を合わせて説明しても全員が納得する状態はほぼ不可能」な場合の解決策。

**EAT（経験→気づき→理論）アプローチを採用する:**

```
理論から入る（従来型）の問題:
  →「理論の説明をどのレベルにするかが難しい」
  →「知識格差がある参加者が混在すると機能しない」

EATアプローチのメリット:
  1. 最初に課題に取り組んでもらう（経験）
  2. 知識レベルの見極めを行いながら、参加者同士が補完し合う（気づき）
  3. 必要な理論・知識を補う形で講師が説明（理論）
  → どのレベルの参加者にも対応できる設計
```

---

### 観点②：興味レベル

#### 興味の4つのレベル

参加者がこの研修に対してどれくらいの興味・意欲を持っているかを把握する。

| レベル | 呼び名 | 特徴 |
|-------|-------|------|
| 最低 | **囚人** | 本当は研修に参加したくないが、強制的に参加させられている人。なぜ参加しなければならないかわからない |
| 低 | **休暇中の人** | 普段の業務より研修参加のほうがラクで楽しいと思っている。学びへの意欲は必ずしもない |
| 中 | **社交家** | 研修の内容よりも、ほかの参加者と話す機会を楽しみにしている。グループワーク時に脱線しやすい |
| 高 | **学習者** | 研修の内容に真摯に取り組み、実践しようとする意欲がある |

**設計段階のポイント:** 研修当日にどう対応するかではなく、**学習者の割合を研修前に増やすにはどうするか**を設計段階で考える。

#### 「学習者」を増やすポイント①：参加者へのメリット訴求

参加者にとってのメリットを明確に発信する。

```
効果的な発信内容:
  ● どんな課題が解決できるのか
  ● どんな知識・スキルが習得でき、業務にどう役立つのか
  ● 参加者の成長やキャリアにどんな意味をもつのか
  ● 過去の参加者がどのように活用して成果を出しているのか
```

#### 案内文に入れるべき項目

日時・場所だけの事務的な文章にしない。「この研修、受けたい！」と思ってもらえる案内を設計する。

```
推奨する案内文の構成:
  □ 研修タイトル
  □ 研修目的
  □ 対象者・参加者が感じているであろう業務上の課題
  □ その課題に対してこの研修で学ぶことを実践すれば、
    どう解決するか・どんなメリットが得られるか
  □ 研修内容・トピック
  □ 日時などの連絡事項
  □ 過去の参加者の声（学びを実践して課題が解決できたとわかるもの）
  □ 事前課題の案内
```

**事前課題の活用:** セルフチェックで自己分析する、上司・同僚からフィードバックをもらうなどの事前課題によって「何を学べばいいかが実感できる」と学習者になる確率が高まる。

#### 「学習者」を増やすポイント②：上司を巻き込む

同じ参加者でも、上司がどのように送り出すかによって、囚人にもなれば学習者にもなり得る。

| 上司の行動 | 参加者への影響 |
|-----------|-------------|
| 「今月研修だから行くように」（事務的通達） | 囚人になる可能性が高い |
| 「この研修で○○を学んで、帰ってきたら一緒に実践しよう」（目的・期待の共有） | 学習者になる可能性が高い |

**研修設計者の役割:** 「上司に任せてうまくやってくれることを願うだけ」ではなく、「上司に何と言って送り出してほしいか」を明確に発信する。

---

### 観点③：言語レベル

#### 4つの言語レベル

参加者がこの研修で使う専門用語をどれくらい知っているかを見極める。

| レベル | 状態 | 研修での対応 |
|-------|------|-----------|
| 知らない | 用語の意味すら知らない | 用語の定義から説明が必要 |
| 聞いたことがある | 何となく知っているが使えない | 概念の整理が必要 |
| 理解している | 意味を理解している | 活用事例・演習に進める |
| 使いこなしている | 実務で使いこなしている | 発展的な内容・ファシリテーター役 |

#### 言語レベルの確認方法

**事前確認（理想）:**
- 使用予定の専門用語をリストアップし、習熟度（1〜5段階）をセルフチェックする事前課題
- 事前課題として書籍やeラーニングで基礎知識を習得してもらう
- 用語集を研修教材に含める

**研修当日の確認（NG例とOK例）:**

```
❌ NG: 「●●という言葉を知らない人はどれくらいいますか？」
   → 挙手しにくい（恥をかくかもしれないという心理）

❌ NG: 「●●という言葉を知っている人はどれくらいいますか？」
   → 挙手すると説明を求められるかもしれないという恐れ

✅ OK: 個人でチェック → チームで「知らない用語リスト」を作成 → 全体で共有
   → 誰が言ったかわからない形にして、参加者の自尊心を傷つけずに確認

✅ OK: 用語カードとその意味カードをバラバラにし、チームでマッチングするグループワーク
   → ゲーム形式で確認することで、知らないことへの心理的ハードルを下げる
```

### 観点④：影響力（権限）

#### 研修で学んだことを職場で実践する権限があるか

研修後を見据えた事前設計。参加者に権限がない場合、研修での学びが職場で活かせない。

**よくある障壁の例:**

```
● 「研修で学んだことは机上の空論」と言う上司がいる
● 全社戦略と部門戦略にずれがあり、研修内容が現場で実行しにくい
● 業務の優先順位の問題で実践する時間が取れない
● 実践したくても練習期間に上司などのサポートが得られない
```

#### 権限の問題への早期対処

研修後ではなく、**研修前に障害を取り除く**。

| 対策 | 内容 |
|------|------|
| **上司を巻き込む** | 研修目的・内容を上司と事前に共有し、賛同を得る |
| **研修後のサポート体制を決めておく** | 練習の機会・上司からのコーチング期間を確約する |
| **ステークホルダーとのミーティング** | 経営層・関係者と協力体制を構築する |

## 4. 事前課題・事後課題の設計

知識レベルのギャップを確認したら、「何を研修中に習得するか」と「何を事前・事後課題にするか」を決める。

### 3つの区分

```
① 事前課題で補う内容
  → 基礎知識の習得、自己分析、現状の課題分析、専門用語の確認

② 研修中に習得する内容（集合研修でしかできないこと）
  → 知識の深掘り、アイデアの共有・拡散
    スキル練習（ロールプレイ等）、アクションプラン作成
    参加者同士の対話・事例共有

③ 事後課題で補う内容
  → 知識の追加・定着、アクションプランの実践と振り返り、職場への応用
```

### 研修を「イベント」ではなく「プロセス」として捉える

```
〔事前〕 → 〔研修当日〕 → 〔事後〕

事前課題:         研修当日:          事後課題:
基礎知識習得     集合でしかできない    実践・振り返り
自己分析         対話・演習・共有      定着・習慣化
上司との面談     アクションプラン作成   上司へのフォローアップ
```

## 5. 上司を巻き込む設計

### 研修前に上司を巻き込む

| アクション | 内容 |
|-----------|------|
| **①ニーズ分析への協力** | ヒアリング・アンケートで上司のニーズを吸い上げる（本人と上司のニーズは必ずしも一致しない） |
| **②研修目的・内容を本人に伝えてもらう** | 「なぜこの研修に参加するか」を上司の言葉で参加者に直接伝えてもらう |
| **③参加者への期待値を伝えてもらう** | 「何を学んでほしいのか」「その意義は何か」を上司から解説してもらう |
| **④研修終了後のイメージを共有する** | 研修後に知識・スキルをどのような場面でどう発揮するかを上司と参加者が共有できている状態を作る |

### 研修後に上司を巻き込む

| アクション | 内容 |
|-----------|------|
| **①参加者から上司への報告** | 学んだことではなく「どう実践するか」を具体的に報告。上司へのサポート依頼も含める |
| **②試行錯誤期間のサポート** | 最初からうまくいかないことを想定。上司が障害を取り除き、進歩を称賛する |
| **③継続的なコーチング** | うまくいっていることは褒め、うまくいっていないことは解決に向けたコーチングを根気強く継続 |

### 上司の巻き込みを「しくみ」として組み込む

個々の上司の手腕に任せるのではなく、**研修の事前課題・事後課題として上司との対話をしくみ化する**。

```
フォーマット例（研修前）:
  上司記入欄: この研修でXXさんに期待する成長・変化
  参加者記入欄: 研修で学びたいこと・上司に支援してほしいこと

フォーマット例（研修後）:
  参加者記入欄: 研修で学んだこと・これから実践すること
  上司記入欄: サポートできること・定期確認の予定
```

## 6. 参加者分析チェックリスト

### ニーズ分析完了後

```
□ 「送られる人」「送る人」「払う人」の3者に調査したか？
□ 複数の調査方法を組み合わせて情報収集したか？
□ 「研修は答えではないかもしれない」という視点で分析したか？
```

### 参加者分析（4観点）

```
【知識レベル】
□ 参加者の現状の知識・スキルレベルを把握したか？
□ 研修後に到達すべきレベルを定義したか？
□ ギャップの大きさ・性質を把握したか？
□ 参加者の既存知識・経験を過小評価していないか？
□ 事前確認の方法を設計したか（アンケート/当日テスト/EATアプローチ）?

【興味レベル】
□ 囚人になりそうな参加者がどれくらいいるか推測したか？
□ 案内文で参加者へのメリットを訴求できているか？
□ 上司に何と言って送り出してほしいかを発信したか？
□ 事前課題で課題意識を醸成する設計をしたか？

【言語レベル】
□ 研修で使う専門用語のリストを作成したか？
□ 参加者の言語レベルを事前に確認する方法を設計したか？
□ 用語集を教材に含める等の対策をとったか？

【影響力（権限）】
□ 研修で学んだことを職場で実践する権限があるか確認したか？
□ 権限がない場合の対処策を事前に手を打ったか？
□ 上司との連携体制（事前・事後）を設計したか？
```

### 事前・事後課題の設計

```
□ 事前課題：研修前に学んでおいてもらう内容を設計したか？
□ 研修中：「集合でしかできないこと」に注力できる設計か？
□ 事後課題：研修後の実践・定着を支援する設計をしたか？
□ 上司の巻き込みを「しくみ」として課題に組み込んだか？
```
