# MEASUREMENT-AND-MATURITY.md

グリーンソフトウェアの測定方法論とGreen Software Maturity Matrix（GSMM）を解説します。完璧な測定、プロキシ活用、測定標準（GHGプロトコル、SCI、ISO 14064）、利用可能ツール、モニタリング統合、成熟度評価の6領域をカバーします。

---

## 1. 完璧な測定

### 1.1 完璧なエネルギーデータ

**測定の原則:**

| 測定対象 | 理想的な方法 | 粒度 |
|---------|------------|------|
| **物理デバイス** | ワットアワーメーター（壁とデバイス間に設置） | リアルタイム、プロセス/サービスレベル |
| **クラウド/分散システム** | ISP/クラウドプロバイダー提供のエネルギー計装データ | リアルタイム、十分な粒度 |

**物理測定の注意点:**

- **Static Power Draw（静的電力消費）**: ハードウェアはアイドル状態でも電力を消費する。アプリケーションの絶対値を測る場合、ベースライン消費を測定し、全体消費から差し引く
- **Steady State（定常状態）**: 測定は定常状態から開始。電気コンポーネント起動時のトランジェントイベント（バースト）を測定に含めない
- **バッテリー**: 測定時は満充電かつ電源接続状態で実施（バッテリー充電の電力消費を測定に含めない）

**大規模システム:**

ラボ環境では優れているが、大規模分散システムのリアルタイムエネルギー測定は複雑。ISPやクラウドプロバイダーがエネルギー計装を提供するまで待つ、またはKepler等のOSSツールを活用する。

### 1.2 完璧なカーボンインテンシティデータ

**理想的な要件:**

- リアルタイム提供
- 高粒度（位置・時間帯）
- エネルギー使用データと組み合わせて炭素メトリクスを算出

**主要データソース:**

| プロバイダー | データタイプ | API | 特徴 |
|-----------|------------|-----|------|
| **WattTime API** | リアルタイム、履歴、予測 | ✅ | 高粒度、複数グリッドカバー |
| **Electricity Maps** | リアルタイム、履歴、予測 | ✅ | 世界規模、ビジュアライゼーション |
| **Our World in Data** | 履歴データ | ❌ | 公開データセット、リアルタイムなし |

**シグナルタイプ:**

- **LRMER (Long-Run Marginal Emission Rates)**: 長期的な限界排出率
- **SRMER (Short-Run Marginal Emission Rates)**: 短期的な限界排出率
- **AER (Average Emission Rates)**: 平均排出率

### 1.3 Market-Based Reductionsの位置付け

**定義:**

- **Power Purchase Agreements (PPAs)**: 再生可能エネルギー生産者との長期契約。消費エネルギーの固定量/割合を購入
- **Carbon Offsets（カーボンオフセット）**: 固定量の炭素削減/除去を表すクレジット。植林、炭素回収等のプロジェクト

**グリーンソフトウェア測定における扱い:**

❌ **完璧な測定には含めない**

- Market-based reductionsは経済的措置であり、ソフトウェアの実際のカーボン強度を変化させない
- 将来のオフセットは現在のソフトウェア強度に影響しない
- SBTi Net-Zero標準では、2050年までに90%の排出削減が求められる（オフセットではなく削減が優先）

**意義の否定ではない:**

PPAsは再生可能エネルギープラントへの資金提供に寄与（Additionality）。2021年には137社が32カ国でPPAsを購入。ただし、ソフトウェアメトリクスとしては混乱を招くため分離すべき。

### 1.4 完璧なエンボディドカーボン追跡

**必要なデータ:**

- **製造時の炭素コスト**: ハードウェアメーカーが提供する総排出量
- **償却期間**: デバイスのライフサイクル全体で炭素債務を分散
- **廃棄時の炭素コスト**: リサイクル/廃棄時の排出量

**オンプレミス vs クラウド:**

| 環境 | データアクセス | 粒度制御 |
|-----|-----------|---------|
| **オンプレミス** | 容易（直接資産管理） | 完全 |
| **パブリッククラウド** | ISP依存 | 粒度はISP次第 |
| **フロントエンド** | 困難（ユーザーデバイスデータ不足） | プロキシ推奨 |

**推奨ツール:**

- **Cloud Carbon Footprint (CCF)**: AWS/Azure/GCPの多数のマシンのライフサイクル分析データをオープンソースで提供

### 1.5 完璧なモニタリングの未来

**理想像: リアルタイム炭素モニタリング**

カーボンデータがレイテンシ・可用性と同様の扱いになる未来。

**実現に必要な2つの要素:**

1. **Time-Series Data（時系列データ）**
   - Prometheus, Grafana等の業界標準ツールに統合
   - 炭素フットプリントをGolden Signalとして扱う
   - SLO違反時のアラート設定

2. **Standardization of Metrics（メトリクス標準化）**
   - OpenTelemetryのようなテレメトリー標準が炭素データにも適用されれば、データ品質向上、比較可能性確保、分析ツール開発が促進される

**恩恵:**

- ソフトウェア開発者は測定の詳細から解放され、削減活動に集中可能
- 少数の専門家が透明なメトリクスを完成させ、大衆が利用する構図

## 2. Good Enough: プロキシの活用

### 2.1 完璧なデータがない場合のアプローチ

完璧なデータにアクセスできない、または継続的にデータ収集できない場合でも、測定を諦める必要はない。プロキシ（代理指標）を一貫して使用すれば、削減を達成できる。

### 2.2 主要プロキシ指標

| プロキシ | 炭素排出との関連性 | メリット | 注意点 |
|---------|-----------------|---------|-------|
| **エネルギー** | 高い（炭素排出の直接要因） | サーバー/クライアント双方で計装可能 | カーボンインテンシティ（グリッドの炭素濃度）と1:1ではない |
| **CPU使用率** | 中程度（エネルギーのプロキシ） | 計装が容易 | エネルギーとの相関は近似的 |
| **コスト** | 高い（ホスティングコスト） | CFOとの連携可能、ビジネス指標と一致 | カーボンアウェアネス（時間帯・場所）は含まれない |
| **ハードウェア数** | 高い（エンボディドカーボン） | エネルギー比例性により効率向上を反映 | システム全体への影響を考慮必要 |
| **レイテンシ/パフォーマンス** | 中程度（効率性の指標） | コード効率化で自動的にリソース削減 | 単独では不十分（ハードウェア最適化との組み合わせが必要） |

### 2.3 プロキシ使用の条件

**科学的アプローチを維持:**

- プロキシを一貫して使用
- 複数のプロキシを組み合わせて全体像を描く
- ソフトウェアへの深い理解が必要

**ホスティングコスト = カーボンのプロキシ:**

ホスティングコストを削減するすべてのアクションは、炭素排出を削減する。これを初期プロキシとして使用し、今日から開始できる。

## 3. 測定方法論

### 3.1 GHG Protocol（温室効果ガスプロトコル）

**概要:**

- 1986年、World Resources Institute + World Business Council for Sustainable Developmentが共同開発
- Corporate Standard（2001年公開）
- Fortune 500の90%が使用（自己申告）

**Scope構造:**

| Scope | 対象 | ソフトウェア例 |
|-------|------|-------------|
| **Scope 1** | 直接排出（自社所有のエネルギー使用） | 自社データセンターの燃料使用 |
| **Scope 2** | 購入エネルギー（電力、蒸気、冷暖房） | データセンターの購入電力 |
| **Scope 3** | その他の間接排出（購入資材、廃棄、出張、製品使用等） | クラウドベンダーのエネルギー使用、エンドユーザーデバイスのエネルギー使用 |

**Scope 2の2つの方法:**

- **Location-based Method**: 実際に使用した電力のグリッド平均排出係数を使用
- **Market-based Method**: 購入した電力（RECs, GOs, PPA等）の排出係数を使用

**ソフトウェアでの課題:**

| インフラタイプ | Scope 2 | Scope 3 | 課題 |
|-------------|---------|---------|------|
| **プライベートクラウド** | エネルギー | エンボディドカーボン | サーバー資産管理が必要 |
| **パブリッククラウド** | - | エネルギー + エンボディドカーボン | 顧客のScope 3に計上 |
| **ハイブリッドクラウド** | 一部エネルギー | 一部エネルギー + エンボディドカーボン | 複雑な配分計算 |
| **フロントエンド** | - | エネルギー + エンボディドカーボン | ユーザーデバイスのデータアクセス困難 |

**ソフトウェア開発者視点の問題:**

SDK/ライブラリのような自社ホスティングしないソフトウェアの排出は、主に顧客のScope 2/3に計上される。自社で削減努力をしても、自社のScope報告には反映されにくい。

**利点:**

組織内の経営陣や関係者が既にGHGプロトコルの用語を理解しているため、コミュニケーションが容易。

### 3.2 Green Software Foundation: Software Carbon Intensity (SCI) Specification

**概要:**

- Green Software Foundation (GSF) が開発
- Beta版: 2021年COP26、Version 1: 2022年COP27
- ISO/IEC 21031として標準化進行中

**特徴:**

| 特徴 | 説明 |
|-----|------|
| **対象** | 組織全体ではなく、ソフトウェアシステム単位の炭素強度（レート） |
| **目標** | 0に到達不可能（エンボディドカーボンとエネルギー消費は避けられない） |
| **Market-based Reductions** | 禁止（Location-basedのみ） |
| **レート指標** | 機能単位あたりの炭素排出量 |

**SCI式:**

```
SCI = ((E × I) + M) / R

E: エネルギー消費（kWh）
I: カーボンインテンシティ（kg CO₂e/kWh、Location-based）
M: エンボディドカーボン（kg CO₂e）
R: 機能単位（例: ユーザーあたり、APIコールあたり）
```

**ソフトウェア境界（Software Boundary）:**

ユーザー定義。運用に大きく寄与するすべてのシステム・インフラを境界内に含めるべき。

**SCI = 0にならない3つの理由:**

1. ハードウェアなしではソフトウェアは動作しない（M > 0）
2. Market-based reductionsが禁止（I > 0）
3. エネルギー消費は必ず発生（E > 0）

**実例: Wonderful Tiny Farm（Raspberry Pi）**

- E: 0.003 kWh/日
- I: 0.713 kg CO₂e/kWh
- M: 55 kg CO₂e（3年ライフスパン: 55 / (365 × 3) = 0.05228 kg CO₂e/日）
- R: Raspberry Piデバイスあたり

```
SCI = ((0.003 × 0.713) + 0.05228) / 1デバイス
    = 0.002139 + 0.05228
    = 0.052 kg CO₂e/日・デバイス
```

**示唆:**

エンボディドカーボンがエネルギーコストより高い → Raspberry Piのライフスパン延長が最優先アクション。

### 3.3 ISO 14064 Standard

**概要:**

- 3部構成: ISO 14064-1（組織レベル）、-2（プロジェクトレベル）、-3（検証）
- 初版: 2006年、最新改訂: 2018年
- GHGプロトコルをベースにしているが、若干の相違あり

**GHGプロトコルとの違い:**

| 項目 | GHG Protocol | ISO 14064-1 |
|-----|-------------|------------|
| **排出分類** | Scope 1/2/3 | 直接排出 + 間接排出（Scope 2+3相当） |
| **Market-based Reductions** | 支援 | 支援 |
| **コスト** | 無料 | 有料 |
| **認証** | なし | ISO認証機関による第三者認証可能 |

**利点:**

ISO Committee on Conformity Assessment (CASCO) が認証プロセス標準を提供。第三者認証により監査性が向上。

### 4.2 オープンソースオプション

**Cloud Carbon Footprint (CCF):**

- スポンサー: Thoughtworks Inc.
- 対応: AWS, Azure, Google Cloud
- 計算式:
  - **Total CO₂e = operational emissions + embodied emissions**
  - **operational emissions = (usage) × (energy conversion factors) × (PUE) × (grid emissions factors)**
  - **embodied emissions = estimated CO₂e from datacenter servers manufacturing**

**Kepler (Kubernetes-based Efficient Power Level Exporter):**

- CNCF Sandbox Project（2023年〜）
- プロセス、コンテナ、Kubernetes Podレベルのエネルギー消費推定
- ベアメタル: リアルタイムシステムメトリクス読取
- VM: 事前学習MLモデルで推定
- エクスポート: Prometheusメトリクス

**比較表:**

| 項目 | Cloud Carbon Footprint | Kepler |
|-----|----------------------|--------|
| **粒度** | 0.001 metric ton CO₂e | N/A（エネルギーのみ、カーボン変換はユーザー側） |
| **Scope 1** | ❌ | ❌ |
| **Scope 2** | ✅（Location-based） | ✅（ユーザーがグリッド強度を掛ける） |
| **Scope 3** | ✅（コンピュートのみ） | ❌ |
| **配分方法** | クラウドタイプにより異なる（請求情報ベース） | ボトムアップ（リアルタイムメトリクス） |
| **時間感度** | クラウドプロバイダー請求データ遅延に依存 | リアルタイム |
| **PUE** | ✅（各クラウドプロバイダーの静的データ） | ❌ |

### 4.3 クライアントサイドツール

**モバイルアプリ:**

- **Android Studio**: 開発段階でのエネルギーモデリング
- **iOS Xcode**: 開発段階でのエネルギーモデリング
- **Greenspector**: 実デバイスでのエネルギー測定（スナップショット）

**Webアプリ:**

- **Green the Web**: 多数のツール集約（分析、画像圧縮等）

**ML:**

- **ML CO₂ Impact Tool**: 複数クラウド + オンプレミス対応のMLトレーニング炭素コスト推定

**共通の制約:**

すべてのツールに境界/限界があるが、最大の利点は**最小限の労力で詳細な洞察を得られること**。一貫した測定と科学的手法のペアリングで、炭素削減を達成可能。

## 5. モニタリング

### 5.1 可用性をNorth Starに

**可用性（Availability）の定義:**

システムが意図した目的を果たせる時間の割合。

```
Availability = Uptime / (Uptime + Downtime)
```

**"Nines"の表:**

| Availability Level | Allowed Downtime/Year | Allowed Downtime/Month | Allowed Downtime/Day |
|-------------------|----------------------|------------------------|---------------------|
| 90% ("1 nine")     | 36日 5時間 23分       | 3日 27分               | 2時間 24分           |
| 99.9% ("3 nines")  | 8時間 42分            | 43分 28秒              | 1分 26秒             |
| 99.999% ("5 nines")| 5分 13秒              | 26秒                   | 0.86秒              |

**Chubbyの教訓（Google）:**

高可用性システムは偽の安心感を生む。Chubbyは99.9999%の可用性を実現したが、ユーザーが過度に依存し、障害時の影響が甚大化。Googleは意図的に定期的にダウンさせることで、宣伝可用性と一致させた。

**可用性と炭素排出:**

高可用性を追求しすぎると、冗長性（Redundancy）による炭素排出増加。モダンなレジリエンス（Resilience）アプローチが推奨される。

### 5.2 4つのメトリクスベースモニタリング（Four Golden Signals）

Google SREの4つのゴールデンシグナル:

| シグナル | 説明 | 例 |
|---------|------|---|
| **Latency** | リクエスト処理時間 | APIレスポンスタイム |
| **Traffic** | 需要の量 | 1時間あたりのリクエスト数 |
| **Errors** | 失敗リクエストの割合 | HTTP 500エラー率 |
| **Saturation** | リソース使用率 | CPU/メモリ使用率 |

**カーボンメトリクスへの応用:**

Prometheus形式のリアルタイム炭素メトリクスが利用可能になれば、これらのゴールデンシグナルと並列して追跡可能。

### 5.3 Service Level Indicators/Objectives/Agreements

**定義:**

- **SLI (Service Level Indicator)**: システム特性の直接測定値（例: リクエストレイテンシ）
- **SLO (Service Level Objective)**: SLIの目標値（例: 過去28日のレイテンシ < 3ms）
- **SLA (Service Level Agreement)**: SLOの集合を含む正式な契約

**カーボンメトリクスが利用可能な場合のシナリオ:**

**例: Chuck（"1 nine" サービス）**

- Uptime SLO: 90%
- Renewable Energy Use SLO: 90%

太陽光発電が雨で利用不可になった場合、Chuckはエラーバジェットを確認。先週の大規模障害でアップタイムのエラーバジェットは消費済みだが、再生可能エネルギーのエラーバジェットは3日分残存 → 非再生可能エネルギーに切り替えて全SLOを維持。

**エラーバジェット:**

SLO達成後の残余許容範囲。例: 99.9%の可用性 → 月43.5分のダウンタイムバジェット。

**炭素モニタリングの統合:**

カーボンメトリクスを既存のプロダクションモニタリングに統合することで、業界標準ツール・プラクティスを再利用し、効率的・効果的にエコ関連メトリクスを監視可能。

### 5.4 オブザーバビリティ（Observability）

**モノリス vs マイクロサービス:**

- **モノリス（2000年代初頭）**: 自己完結型、監視が比較的単純
- **マイクロサービス（2020年代）**: 分散システム、"unknown unknowns"の複雑性

**Rumsfeld Matrix:**

```
┌────────────────┬────────────────┐
│ Known Knowns   │ Known Unknowns │
│ (既知の既知)   │ (既知の未知)   │
├────────────────┼────────────────┤
│ Unknown Knowns │ Unknown Unknowns│
│ (未知の既知)   │ (未知の未知)   │
└────────────────┴────────────────┘
```

**Monitoring vs Observability:**

- **Monitoring**: Known Unknownsの特定（予測可能な問題）
- **Observability**: Unknown Unknownsの探索（予測不可能な問題）

**観測可能性の定義:**

システムの内部状態を外部出力から理解できる度合い。テレメトリー（traces, metrics, logs）の収集だけでなく、それを利用してシステムを説明できることが真の観測可能性。

**グリーンソフトウェアの観測可能性:**

実現したら、リアルタイムメトリクス + トレースによる相関分析で、どのイベントが最大エネルギー消費を引き起こしたかを特定し、ボトルネックを効率的に削減可能。DevOps/SREのデバッグとグリーンソフトウェアのSLO遵守の両方を支援。

## 6. Green Software Maturity Matrix (GSMM)

### 6.1 GSMMの目的

**開発元:**

- Linux Foundation's Green Software Foundation

**目的:**

- 組織のグリーン原則・プロセス実装度の自己評価
- "Raising the Floor and the Ceiling"（遅れている企業の底上げ + 先進企業の啓発）
- 共通の道筋を示し、ツール・サービスの共有進捗を得る

**対象:**

- ソフトウェア実務者が日常業務で温室効果ガス排出を削減する方法
- 水使用量、種保護等の他の環境目標はカバー外（主にDC選択に依存）

### 6.2 成熟度マトリクスの歴史

**起源:**

- 1986年: Capability Maturity Model Integration (CMMI)
- Carnegie Mellon University Software Engineering Institute
- プロセス改善のアセスメント・ガイドフレームワーク

**成熟度マトリクスの構造（5レベル）:**

| レベル | 名称 | 説明 |
|-------|-----|------|
| **1** | Aspiring | プロセスは予測不可能、制御不良、リアクティブ |
| **2** | Aware | プロジェクト単位のプロセス、依然リアクティブ |
| **3** | Acting | 標準化・文書化・レビュー済みプロセス |
| **4** | Awesome | 測定・制御されたプロセス |
| **5** | Inspiring | 定量的フィードバックベースの継続的改善 |

**GSMM解釈:**

- **Level 1**: 個人レベルの関心、組織戦略なし
- **Level 2**: 基本データ取得、プロジェクト単位の削減
- **Level 3**: 組織横断プロセス、日次/週次/月次データ
- **Level 4**: リアルタイムデータで進捗を動的測定
- **Level 5**: リアルタイムデータで監視・改善、24/7 CFE達成

**注意:**

GSMMの対象者は、技術組織のシニア層と社内グリーンアドボケイト。高レベルの旅のガイドと進捗報告ツール。

### 6.3 測定なしで改善可能（良いニュース）

**Adrianth Cockcroftの指摘:**

"You can't improve what you can't measure" (Peter Drucker) はグリーンソフトウェアには当てはまらない。優れたデータがなくても、保証された炭素削減効果のあるアクションが存在する。

**レベル1→2への即時アクション（測定不要）:**

- 未使用マシン（ゾンビ）の停止、手動ライトサイジング
- グリーンな地域へのホスティング選択
- クラウドへの一部ワークロード移行、マルチテナンシー活用
- GitOps/SRE原則の学習

**初期プロキシ:**

ホスティング費用の削減 = 排出削減。今日から開始可能。

### 6.4 GSMMレベル詳細

**Level 1: Aspiring（憧れ）**

- 個人が関心を持つが、組織レベルの関心・プロセス・コミットメントなし
- 一部開発チームが炭素排出を測定するかもしれないが、全社技術由来炭素フットプリントは不明
- カーボンゼロ目標なし、または曖昧な目標のみ（例: Net Zero by 2050、計画なし）

**Level 2: Aware（認識）**

- 技術組織のScope 1/2排出を把握（週次/月次/年次）
- 社内の数プロジェクトが炭素削減を推進（データ使用は必須ではない）
- エンジニアがグリーンシステム・運用のトレーニング受講
- ゾンビサービス手動停止、ライトサイジング、スポットインスタンス切替等の基本アクション実施

**Level 3: Acting（行動）**

- Scope 1/2/3排出を把握（日次/週次/月次/年次）
- エンボディドカーボンを把握
- Scope 1/2排出削減を実施中、一貫した出力単位あたりのカーボンフットプリント減少
- 全チームに定義済みプロセス（IT関連運用のCO₂e排出の定期報告・レビュー）
- プロダクト管理チームがムダ回避の権限あり（不要データ保存回避、YAGNI原則）
- ゾンビ/低使用サービスをいつでも停止可能（LightSwitchOps）
- 電力由来炭素 vs エンボディドカーボンのトレードオフを認識、エンボディドカーボン最小化計画（サーバー・エンドユーザーデバイスのライフスパン延長）

**Level 4: Awesome（素晴らしい）**

- 技術戦略がビジネス目標としてグリーンと整合（ESG、コスト、レジリエンス、EU市場アクセス）
- Scope 1/2/3データで既にネットカーボンゼロ達成（オフセット10%以下）
- リアルタイムScope 1/2/3データの測定システムあり（ビジネスは認識しているが、日常業務駆動ではまだ）
- デマンドシェイピング、プリキャッシュ、時間・場所シフト実施
- 未使用サービス自動停止、ライトサイジング自動化（SRE原則適用）
- リーンアプリケーション、使用率不足機能の定期レビュー・廃止
- 成長に伴いIT炭素フットプリント減少
- 10年以上のハードウェアをEoL到達させない（セキュリティパッチ・後方互換性維持）
- 顧客の炭素フットプリント削減を積極推進（Scope 3レポート提供等）

**Level 5: Inspiring（感動的）**

- 24/7 CFE達成、オフセット1%以下
- チームレベルのカーボン測定目標設定
- リアルタイム情報（動的グリッドエネルギーデータ含む）で迅速・定量的意思決定、最小排出最適化（サービス経由、オープン標準準拠データ）
- SREアウェア組織、エラーバジェット思考、カーボン排出にもエラーバジェット適用
- 天気・グリッド混雑予測でほぼ全ワークロードを事前変更・移動、ローカルグリーンパワー可用性にマッチ
- 時間制約のあるワークロードは高度に最適化（最小電力使用）
- セキュリティパッチ・後方互換性でハードウェアを一切EoL到達させない

**24/7 CFE:**

グリッドから電力引き出し時、同時刻にカーボンフリー電力が十分供給されている状態。

### 6.5 GSMMの軸（Axes）

ビジネスは全技術組織で同一レベルではない（Level 1以外）。チーム単位でグリーンソフトウェア成熟度が異なるため、領域（軸）ごとに成熟度チェックリストを定義。

**9つの軸:**

1. **Climate Commitments（気候コミットメント）**
2. **Carbon Footprint（炭素フットプリント）**
3. **CO₂e Metrics（CO₂eメトリクス）**
4. **Operational Efficiency（運用効率）**
5. **Electricity Use（電力使用）**
6. **End-User Devices（エンドユーザーデバイス）**
7. **Servers（サーバー）**
8. **Product Management（プロダクト管理）**
9. **Education（教育）**

**軸別チェックリスト概要:**

- **Climate Commitments**: Level 1 (Net Zero by 2050) → Level 5 (24/7 CFE、1%オフセット、カーボンエラーバジェット)
- **Carbon Footprint**: Level 1 (不明) → Level 5 (ゼロに近い)
- **CO₂e Metrics**: Level 1 (メトリクスなし) → Level 5 (リアルタイムメトリクス + 予測、業界標準API)
- **Operational Efficiency**: Level 1 (ライトサイジングなし、ゾンビ無知) → Level 5 (厳格なカーボンエラーバジェット、Scope 3エンボディドカーボン含む)
- **Electricity Use**: Level 1 (常時稼働、電力意識なし) → Level 5 (24/7 CFE、厳格なカーボンエラーバジェット)
- **End-User Devices**: Level 1 (寿命目標なし) → Level 5 (自動プロセス、全デバイス後方互換、Ship of Theseus的永続性)
- **Servers**: Level 1 (使用率目標なし) → Level 5 (グリッドアウェア統合、エンドユーザーデバイス含むハードウェア最小化)
- **Product Management**: Level 1 (カーボン意識なし) → Level 5 (機能ごとにScope 1/2/3カーボンエラーバジェット追跡)
- **Education**: Level 1 (アドホックな個人トレーニング) → Level 5 (達成内容を他者に教育)

### 6.6 GSF 50% Challenge

Green Software Foundationは、組織にグリーンソフトウェア成熟度の向上を呼びかけている。大半の組織は現在Level 1だが、最大の成果は目前にある。

**即時アクション（Level 1→2）:**

- 未使用サーバーの停止: 温室効果ガス排出を半減可能
- コスト削減・セキュリティ向上の副次効果

                  │

### 実践ロードマップ

**短期（0-6ヶ月）:**

1. GSMM自己評価（現在のレベル特定）
2. ホスティングコストをプロキシとして削減開始
3. ゾンビサービス停止、ライトサイジング
4. グリーンな地域へのホスティング選択

**中期（6-18ヶ月）:**

1. Scope 1/2排出データ収集（クラウドプロバイダーツール、CCF等）
2. SCI計算、ベースライン確立
3. エンジニア向けグリーンソフトウェアトレーニング
4. LightSwitchOps導入

**長期（18ヶ月以上）:**

1. Scope 3データ収集、リアルタイムメトリクス導入
2. デマンドシェイピング、時間・場所シフト自動化
3. SLOにカーボンメトリクス統合、エラーバジェット設定
4. 24/7 CFE達成

**継続的:**

- カーボンメトリクスの監視・改善
- 新規ツール・手法の評価
- 組織全体への教育・啓発
- 業界標準への貢献（GSF、ISO等）
