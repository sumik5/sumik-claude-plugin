# ビヘイビアプラン（Behavior Plans）

ビヘイビアプランは、ユーザーがターゲットアクションを完遂するために通過するすべてのステップを表した行動設計の中核ドキュメントです。

## ビヘイビアプランとは

### 定義

ビヘイビアプランとは、プロダクトを使ってターゲットアクションを完遂する上で、ユーザーが行う必要があるすべてのステップ（プロダクト内外を問わず）を表したものです。

**含まれるもの**:
- プロダクト上でユーザーがすべきこと
- それに反応してプロダクトがすべきこと
- 実世界で起こるすべてのこと

このプランがあれば、途中のひとつひとつのステップでユーザーに何が起きていて、なぜ次のステップに移ろうとしているかがわかります。

### 表現ツール

ビヘイビアプランは様々なデザインツールで表現できます：

| ツール | 特徴 |
|-------|------|
| カスタマーエクスペリエンスマップ | 体験ステージ、ユーザータイプ、感情を表現 |
| タッチポイントマップ | 接点の一覧化 |
| エンパシーマップ | ユーザーの共感ポイント |
| ジャーニーマップ | ユーザーの旅路 |
| 粗いフローチャート | シンプルな流れ図 |
| 階層化アウトライン | 各接点の詳細を見出し形式で記載 |

## ビヘイビアプランの作成手順

### 1. 書き出して詳細化する

**ステップ1: 大まかなステップを書き出す**
- ユーザーが行動を完遂するために必要な一連のステップを列挙
- プロダクト上だけでなく、実世界でやらなければならない行動も含める
- 最初は粗く、後から詳細化

**例：ラジオ番組に電話をかける**
1. 静かな時間と場所を見つけ、ラジオと電話を用意する
2. ラジオ番組を見つける
3. 電話がかけられるときに番組を聞く
4. 電話番号を知る
5. 本当に電話する勇気を奮い立たせる
6. 電話する
7. スタッフに興味深い意見を言えそうだと思ってもらう
8. 番組本番でわかりやすく話をする
9. ボランティア事務局に放送されたことを伝える

**ステップ2: ステップにラベルを貼る**
- プロダクト上でユーザーがすべきこと（例：アプリで番組情報を検索）
- その行動に反応してプロダクトがすべきこと（例：自動で電話番号を表示）
- その他の実世界で起こるすべてのこと（例：実際に電話をかける）

**ステップ3: 新規ユーザーの観点で抜けを探す**
- 全くの新規ユーザーの視点で考える
- 初めに加えるべきステップ（例：ユーザー登録）はないか

**ステップ4: 1回限りのステップを見つけ出す**
- プロダクト利用経験があるユーザーの視点で考える
- リピーターが飛ばしてよいステップはないか

### 2. ペルソナ視点でウォークスルーする

**各ペルソナで確認**:
1. 主要なペルソナそれぞれの視点でプランを通しで確認
2. 各ステップで以下を問う：
   - 行動に慣れているか？
   - 何を考えながらやっているか？
   - 何が動機になっているか？
   - 自分がユーザーだったら使い続けたいか？
3. 答えを付箋に書き、ビヘイビアプランに貼る

**障壁の特定**:
- ユーザーが自力で各ステップを完了できそうか考える
- ユーザーに立ちはだかる個々の障壁を書き出す
- 障壁を避けるためにステップを変えられるか検討

**例：初心者vs経験者**
- 初心者：「電話する勇気を奮い立たせる」が障壁
- 解決策：例文付きのわかりやすい指示を提供
- 経験者：余計なおせっかいなしに効率的にサッと終わらせたい
- 解決策：指示があっても邪魔にならない配置

### 3. 実用最小限のアクション（MVA）に絞り込む

**切り詰めの原則**:
| 原則 | 説明 | 例 |
|------|------|-----|
| 繰り返し → 一度きり | 同じことを何度もさせない | 自動保存、設定の記憶 |
| 大きなアクション → シンプル | ステップを簡略化 | 全局への電話 → 1局のみ |
| 「あったらいい」を無くす | 必須でない機能は削除 | メモ機能は後回し |
| リスク部分を優先 | 不確実な部分を先に解決 | 「気楽に電話をかける」に注力 |

## 行動の構造化テクニック

### チート戦略（自動化とデフォルト化）

**自動化できるステップを探す**:
- プラン全体を自動化できないか検討
- 一部のステップだけでも自動化・デフォルト化できないか

**具体例：ラジオ番組キャンペーン**
| ステップ | 自動化の方法 |
|---------|------------|
| 番組を探す | 自動的にユーザーと番組をマッチング |
| ラジオを用意 | アプリで番組をストリーミング |
| 話す内容を考える | デフォルトの台本を提供（カスタマイズ可能） |

**チートが最適な場面**:
- ユーザーの積極的な関与なしに解決策を適用できる
- ユーザーが自動実行に同意してくれる

### 簡素化（チャンキングとステップ削減）

**なるべく大きくまとめる**:
- 各ステップは、ユーザーが理解し実現できる範囲で、なるべく大きなかたまりで表現
- 2つ以上のステップで1つにまとめられるものを探す
- ステップ数が10以上になってきたら意識する

**「簡単そう」にする方法**:
| 方法 | 説明 |
|------|------|
| わかりやすく伝える | 2行以内で説明、専門用語を避ける |
| 実現できそうにする | 自己効力感を高める、障壁を低く見せる |

**小さな勝利を提供**:
- 各ステップに達成感を感じられる意義を持たせる
- 成功と失敗の明確な定義
- ステップ完了がはっきりと伝わるようにする

**よくある失敗**:
| 失敗パターン | 説明 | 対策 |
|------------|------|------|
| 簡単でしょ！ | 自分にとって簡単だから大丈夫と思い込む | 未経験者の提案を採用 |
| 大変だからこそ頑張る | わざと困難にして強いコミットメントを求める | 本当に難しい作業にだけ達成感を |

### 習慣化できる行動を探す

**習慣候補の特定**:
- 同じような行動を繰り返し行うステップはないか
- 明確なキューがあるか（既存または追加可能）
- ステップの終わりに有意義なリワードがあるか

**効果**:
「ユーザーに1000回、Xという作業を実行してもらう」という問題を、「ユーザーにXという習慣をつける」という問題に変換できる

## 環境を構築する

行動を取りやすい環境を構築するための5つの方法：

### 1. 動機を高める

**4つの軸で考える**:

| 軸 | 選択肢 | 推奨 |
|----|--------|------|
| 新しい vs 今ある | 新しい動機 vs 既存の動機を強調 | まずは今ある動機を活用 |
| 報酬 vs 懲罰 | 行動した人に報酬 vs しなかった人に懲罰 | ユーザーを罰することは避ける |
| 動機づけの種類 | 金銭、評価、習熟、達成など | ユーザーごとに適した動機を見つける |
| 現在 vs 将来 | 今の動機 vs 先の動機 | 将来の動機を現在に意義あるものへ変換 |

**内発的動機 vs 外発的動機**:
- 内発的動機：活動そのものに内在する楽しみ
- 外発的動機：特定の成果（お金、勝利）を得ようとする欲求
- プロダクトはユーザーの既存動機（内発的/外発的）を両方活かせる
- 新たに動機を持ってもらうと外発的動機になる傾向

**外発的動機が有効な場合**:
- 特定ステップにもともとの強い動機がないとき
- 一度きりの行動で内発的動機を締め出してもあまり関係ないとき
- 外発的動機を内発的動機に移し替える手助けをするとき

**未来の動機を現在に引き込む方法**:
| 方法 | 説明 | 例 |
|------|------|-----|
| コミットメント契約 | 決意の瞬間に将来の動機をロックイン | 目標未達成時に寄付される仕組み |
| 未来を現在に持ち込む | 将来の自分の姿を見せる | 加工写真で将来の見た目を表示 |
| 代替報酬 | 長期的動機を現在のものに変換 | 薬物治療と映画鑑賞を結びつける |
| 別の短期的動機を探す | 遠い未来の動機を諦める | 長期健康より「すぐ効く性生活改善」 |

### 2. 行動を促すキューを出す

**キューの3つの効果**:
| 効果 | 説明 |
|------|------|
| 注意（Attention） | 人は忙しく、注意の範囲は限定的 |
| 義務（Obligation） | 相手からの頼みを断るのは居心地が悪い |
| 即時性・緊急性（Urgency） | 「今すぐ」行うよう頼むと後回しの壁を乗り越えやすい |

**キューの方法**:
- シンプルに頼む：メール、テキストメッセージ、大きな「Act Now」ボタン
- 環境の取っ掛かりをキューに：日常的に見聞きするものを行動のトリガーに
- 実行意図（implementation intentions）：「Yが発生したらXをする」という具体的な計画

### 3. フィードバックループを生み出す

**効果的なフィードバックの条件**:
| 条件 | 説明 |
|------|------|
| 即時性（Timely） | 行動中にフィードバック、即座に修正可能 |
| 明確性（Clear） | 情報の意味をユーザーが理解できる |
| 実用性（Actionable） | フィードバック後どう行動すべきかわかる |

**必須要素**:
- ユーザーがフィードバックを気にかけている
- パフォーマンス向上に必要な調整をユーザー自身がしたいと望んでいる
- ユーザーが注意を払い続けている

**例：Fitbit One**
- どれだけ運動したかを常にフィードバック
- ルーティンを変えるとすぐに反映される
- 目標に合わせた行動を1日を通じて取ることができる

### 4. 競合に打ち勝つ

**競合の3つの側面**:
| 側面 | 質問 |
|------|------|
| 動機の競合 | 何がユーザーのやる気をそいでいるか？他のことをするように動機づけているものは？ |
| 注意の競合 | 環境の中で何がユーザーの注意をひきつけ、ターゲットアクションに気づくのを阻んでいるか？ |
| やりやすさの競合 | 環境に簡単でシンプルな他の行動があふれていないか？ |

**対抗の3つの戦略**:
1. **競合要因を低下させる**：
   - アプリケーション上の他の行動に関する動機、注意、やりやすさを低下
   - ユーザーの注意を一度に1つのことに集中させる
   - アプリケーションを単純化、気の散る文章を削除

2. **競合要因を逆に活用**：
   - ユーザーが本当に関心を持っていることにターゲットアクションを接続
   - ユーザーの関心がすでにある場所が最もよい接続点
   - 例：Facebookに対応する（すでにユーザーの注目を集めている場所）

3. **ゴリ押し**（推奨しない）：
   - 注意を促すために大声で呼びかける
   - 動機を高める
   - プロダクトを息をするよりも簡単に使えるようにする
   - 時間をかけて新たな習慣や経験をつくり上げ、競合する行動を締め出す

### 5. 障害を取り除くか回避する

**障害の例と対処**:
| 障害 | 対処法 |
|------|--------|
| 身体的障害 | チームでよく考え、プロダクトに適した解決策を探す |
| リソースの制約 | 企業やプロダクトが利用者を支援する方法を検討 |
| スマートフォンの不慣れ・未所持 | デスクトップ版やウェブ版を用意 |

**インタラクションデザインの基本**:
- ユーザーがアプリケーション上で行うすべてのタスクでの抵抗を取り除く
- 各ステップで動機、キュー、フィードバック、競合排除、障害対処を確認

## ユーザー自身を準備する

行動に先駆けてユーザーを準備する3つの戦術：

### 1. 物語る（Narrate）- ユーザーの自身の捉え方を変える

**セルフナラティブの活用**:
- 過去の自分の行動をどう表現するか
- 「行動を変えること」がその人の本来持つ自然な姿の延長線上にあると考えられるように手助け

**方法**:
| 方法 | 説明 | 例 |
|------|------|-----|
| 過去の行動を思い出す | 「その行動に関連する、どんなことを過去にしましたか？」 | 運動アプリ：すでに歩いている距離を見せて驚かせる |
| 達成を祝福 | ユーザーがすでにやってきたことを心から祝福 | 小さな成功も認める |
| ネガティブな経験の再解釈 | 克服可能な壁として捉えられるように手助け | 大学の成績悪化を一時的困難として再解釈 |
| 自己効力感の構築 | 過去の成功体験を思い出させる | 関連タスクでの成功を強調 |

**ストーリー編集の効果**:
- セルフナラティブの変化が未来の行動に影響
- 行動が変わると未来の経験も変わる
- 新しい経験が内部ストーリーを変化させる
- 行動変容の新しいサイクルを引き起こす

### 2. 関連づける（Associate）- ユーザーの行動の捉え方を変える

**行動ブリッジの構築**:
- ユーザーがとりたい行動と過去の経験との間に心理的なつながりをつくる
- ある行動から別の行動にスムーズかつ簡単に移行できるように手助け

**例：Speek（電話会議アプリ）**
- 新しい側面：URLリンクで電話会議に接続（簡単）
- 馴染みのある側面：必要なら普通の電話番号+PINコードも使える
- → ユーザーが新しい方法を試しやすくなる

**ポジティブで馴染みのあることに関連づける**:
- 過去の経験の連想を活用
- プロダクトをつくり変えて心理的つながりをつくる

### 3. 教育する（Educate）- ユーザーの世界の捉え方を変える

**教育が効果的でない場合**:
- 行動が習慣的・自動的で意識的に考え抜かれたものではない
- 人があまりにも多くの情報に圧倒されている
- 情報を得るのが意思決定のタイミングのずっと前（またはずっと後）

**3つの教育アプローチ**:
| アプローチ | 内容 | 効果的な場面 |
|-----------|------|------------|
| なぜ重要か（Why） | 行動の重要性 | ユーザーが重要性を知らない場合 |
| どのように機能するか（How） | システムの内部構造、金融リテラシー等 | 意思決定の瞬間の直前でないと忘れる |
| 何をすべきか（What to do） | 具体的な手続き | 明確なガイダンス、すぐに行動に必要 |

**推奨**:
- 自発的な行動変容では「何をすべきか」が最も有効
- 知識のない人には「どのように」は情報量が多すぎる
- 「何をすべきか」は実行に際しての明確なガイダンス

**スキル構築の訓練**:
1. ユーザーが実行しているアクションシークエンスを確認
2. 最後の部分から逆算して、失敗する可能性が最も高い部分を特定
3. ユーザーがすでに必要な技能を持っているか評価
4. 不足している場合、早い段階に似たような小さなチャレンジを組み込む

**ゲームデザインとの類似**:
- 徐々に難易度が高まる一連のチャレンジを構築
- プレイヤーが経験を積むにつれてゲームが難しくなる
- ビヘイビアプランをユーザーが持っている技能の観点から見返す

## ビヘイビアプランの更新

### 環境構築後の確認項目

各ステップで以下をメモ：

| 確認項目 | 質問 |
|---------|------|
| 動機 | ユーザーの動機は何か？環境はその動機にどう貢献しているか？ |
| キュー | 今すぐ行動を起こせるキューを用意できるか？ |
| フィードバック | パフォーマンスに対する明確なフィードバックを受けるか？それで行動を改善できるか？ |
| 競合 | 他にユーザーの気を散らすものはないか？ |
| 障害 | ユーザーを阻害しているものは何か？ |

### ユーザー準備後の確認項目

| 確認項目 | 質問 |
|---------|------|
| セルフストーリー | ユーザーは自分を達成できる人間だとみなしているか？ネガティブな自己認識を変えられるか？ |
| 行動ブリッジ | 類似のポジティブな経験をプロダクトに取り込めるか？橋渡しできるか？ |
| 必須情報 | プロダクトを使い始める前に絶対に知っておかなければならないことはあるか？ |
| スキル構築 | 後続の困難なステップに備え、早い段階に練習や経験を加える必要があるか？ |

## 行動変容テクニックと思考レベルの対応

| 思考レベル | 行動の種類 | 有効なテクニック |
|-----------|-----------|----------------|
| ほとんど思考不要 | 頻繁に繰り返される慣れ親しんだ状況 | 習慣化、習慣を変える |
| 思考全く不要 | ユーザーが承認すれば後は自動 | 自動化、チート |
| 多少の直感 + 意識的選択 | やや馴染みのある状況 | 過去の経験との関連、セルフナラティブ変更 |
| 意識的思考必要 | やることがたくさんある | 小さく簡単なステップのシークエンスに分解 |
| 負担が大きい | 未知の状況、メリット・デメリットを天秤にかける | ユーザー教育（より広範な戦術の一部として） |

## まとめ

**ビヘイビアプランの成果物**:
- ユーザー行動の具体的な一連のステップ
- 各ステップでの動機、キュー、フィードバック、競合対処、障害対処の記述
- ユーザー自身の準備方法
- 意思決定のための環境の記述
- プロダクトの機能の見せ方についてのアイデア一覧（ビヘイビアプランとは別）

**ビヘイビアプランの特徴**:
- ユーザーが行動を完遂するまでの全体のストーリー
- プロダクト内外のすべてのステップを包含
- ペルソナごとにカスタマイズ
- 反復的に改善される生きたドキュメント
