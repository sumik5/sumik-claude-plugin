# インターフェース戦術（Interface Tactics）

CREATEアクションファネルの各段階で使えるUI/UX戦術の実践的ガイド。

## 概要

インターフェースデザインをレビューする際、以下2つの観点から確認します：
1. **プロダクト全体を通じて、ユーザー体験のギャップ（落差）が大きいところを探す**
2. **行動戦術が活かせるところを探す**

## CREATEアクションファネル復習

人が行動するために同時に必要な5つの前提条件：

| 段階 | 説明 | 具体例 |
|------|------|--------|
| **Cue（キュー）** | 行動について考え始めるのに必要なもの | カラーリリーを見ると海岸散歩を想像、スマホの振動でメッセージ確認 |
| **Reaction（反応）** | 直感的な心理の自動的でスピーディな反応 | おもしろそうか、楽しそうかの基本的感覚、離脱や拒否をしない |
| **Evaluation（評価）** | 意識的な費用対効果の評価 | 価値と軋轢・試練のバランス、相対的な評価 |
| **Ability（アビリティ）** | そのとき実際にすぐできること | 何をすべきかわかる、必要なものがある、失敗しなさそう |
| **Timing（タイミング）** | 今それをやる理由 | 他の急ぎのことより優先する理由がある |

## CREATEファネル別戦術一覧

### Cue（キュー）の戦術

#### 1. 行動を明確に求める（Call-to-Action）

| 戦術 | 説明 | 実装例 |
|------|------|--------|
| 明確なCTA | 遠慮せず行動を依頼 | 「今すぐ登録」「申し込む」大きなボタン |
| 簡潔な指示 | 2行以内で伝える | 「あなたのメールアドレスを入力してください」 |
| アクション志向の言葉 | 動詞を使う | 「始める」「発見する」「作成する」 |

**効果**:
- 注意（Attention）：忙しい人の注意を引きつける
- 義務（Obligation）：頼まれたことを断るのは居心地が悪い
- 即時性・緊急性（Urgency）：「今すぐ」と頼むと後回しの壁を乗り越えやすい

**ベストプラクティス**:
- テキストメッセージ、メール、大きな「Act Now」ボタン
- 頻繁すぎないように注意
- 文脈に合わせた適切なタイミング

#### 2. アフォーダンスを活用

| 要素 | 設計原則 |
|------|---------|
| ボタン | ボタンらしく、押せることが明確 |
| リンク | 下線や色でクリック可能と示す |
| フォーム | 入力フィールドが明確に区別される |
| インタラクティブ要素 | ホバー時に変化、フィードバックを提供 |

**認知過負荷を避ける**:
- 見ていることを理解するために脳が行う論理的な結合や飛躍を最小化
- 当てずっぽうに選ばせない
- 多少実行が難しくなっても、わかりやすさを優先

#### 3. 余計なものを削除

| 原則 | 実装 |
|------|------|
| 1ページ1アクション | 主要なアクションは1つに絞る |
| 気の散る文章を削除 | 必要不可欠なもの以外はすべてページから取り去る |
| 視覚的階層 | 最も重要な要素を目立たせる |
| ホワイトスペース | 余白を活用して焦点を明確に |

**競合する要素の排除**:
- 他の行動喚起要素を取り除く
- カテゴリーが異なるところに配置
- ユーザーの注意を一度に1つのことに集中させる

### Reaction（反応）の戦術

#### 1. 信頼を構築

| 要素 | 重要性 |
|------|--------|
| 初回体験 | 最も重要、第一印象が決定的 |
| ブランド一貫性 | ロゴ、色、トーンを統一 |
| プロフェッショナルなデザイン | 質の高いビジュアル、エラーなし |
| セキュリティ指標 | SSL証明書、プライバシーポリシー |
| 透明性 | 何が起きるか明確に説明 |

**バグの特別な意味**:
- 信用を傷つける
- 安心感に水を差す
- 没入感に割り込む
- 破産した人が資産管理を教えられないように、動かないアプリは生活を変える手段として信用できない

#### 2. 社会的証明（Social Proof）

| タイプ | 実装例 |
|--------|--------|
| ユーザー数 | 「100万人が使用中」 |
| レビュー・評価 | 星評価、ユーザーレビュー |
| 推薦 | 有名人、専門家の推薦 |
| メディア掲載 | 「〇〇誌で紹介されました」 |
| 行動の可視化 | 「今、50人がこれを見ています」 |

**効果**:
- 他者の行動を見ることで自分の行動を動機づける
- 不確実性を減らす
- 正しい選択であるという安心感

#### 3. ピア比較（Peer Comparison）

| 手法 | 説明 | 例 |
|------|------|-----|
| 近所との比較 | 地理的に近い人との比較 | Opower：近所の電力使用量と比較 |
| 友人との比較 | ソーシャルネットワーク内の比較 | 運動アプリ：友人の運動量と比較 |
| 匿名化 | プライバシーを保護しながら比較 | 「あなたは上位25%です」 |

**注意点**:
- ネガティブにならないように
- 改善の余地を示す
- 達成可能な目標として提示

### Evaluation（評価）の戦術

#### 1. 損失回避を活用（Loss Aversion）

| 手法 | フレーミング | 効果 |
|------|------------|------|
| 失う表現 | 「年間10万円を失っています」 | 得る表現より強力 |
| デフォルトからの変更 | 「現在のプランを失います」 | 現状維持バイアスを活用 |
| タイムリミット | 「あと3時間でオファー終了」 | 機会損失の恐怖 |
| 所有権錯覚 | 「あなたの〇〇」と表現 | まだ所有していないものも失いたくない |

**コミットメント契約**:
- ルールを守らなかった場合に大事なものを没収される約束
- 罰が自分で決められ（self-imposed）、自分で調整（self-calibrated）できる
- 例：stickK.com - 失敗したら嫌いなNGOに寄付

#### 2. メリットを強調（Benefit Highlighting）

| 要素 | 実装 |
|------|------|
| 価値提案 | 明確で具体的な利益 |
| ビジュアル化 | 利益を視覚的に表現 |
| 即時性 | 「今すぐ手に入る」利益を強調 |
| パーソナライズ | 「あなたは〇〇を節約できます」 |

**報酬の種類を試す**:
| 種類 | 例 |
|------|-----|
| 金銭的報酬 | 現金、ポイント、割引 |
| 社会的報酬 | 地位、尊敬、仲間内での評価 |
| 内発的メリット | 新しいものの発見、習熟感、達成感 |

**注意**:
- 時間の経過とともに動機づけを変える
- 人は報酬に飽きる

#### 3. コストを下げる（Reduce Friction）

| 摩擦の種類 | 削減方法 |
|-----------|---------|
| 認知的摩擦 | 情報を簡潔に、選択肢を減らす |
| 物理的摩擦 | クリック数を減らす、入力を最小化 |
| 時間的摩擦 | ページ読み込みを高速化、待ち時間を減らす |
| 感情的摩擦 | 不安を取り除く、サポートを提供 |

**フレーミング効果**:
- 同じ情報でも提示方法で印象が変わる
- ポジティブなフレーミングを選ぶ
- 文脈に合わせた表現

### Ability（アビリティ）の戦術

#### 1. デフォルト設定（Smart Defaults）

| 原則 | 実装 | 例 |
|------|------|-----|
| 賢いデフォルト値 | ほとんどのユーザーに適した初期値 | 推奨設定、一般的な選択肢 |
| オプトアウトよりオプトイン | デフォルトで有効にする | 自動保存、通知設定 |
| 最小限の調整 | デフォルトのままでも使える | カスタマイズは任意 |

**効果**:
- 意思決定の負担を減らす
- 行動のハードルを下げる
- 初心者でもすぐに使い始められる

#### 2. 自己効力感を高める（Self-Efficacy）

| 手法 | 実装 |
|------|------|
| 小さな勝利 | 達成可能なステップに分解 |
| 進捗の可視化 | プログレスバー、チェックリスト |
| ポジティブフィードバック | 「よくできました！」「あと少しです」 |
| 過去の成功を示す | 「あなたはすでに〇〇を達成しました」 |

**自己効力感とは**:
- 自分自身が課題に対する実行力を持っているという信念
- 無理だと感じた場合は試そうとすらしない
- 各ステップがユーザーから見て実現できそうに見えることが重要

#### 3. 実行意図を引き出す（Implementation Intentions）

| 要素 | 実装 |
|------|------|
| 具体的な計画 | 「いつ」「どこで」「何を」を明確に |
| トリガー設定 | 「Yが起きたらXをする」 |
| 事前コミットメント | 将来の行動を今決める |

**例**:
- 「木曜日の朝9時にジムに行く」
- 「朝の番組が終わったら走りに行く」
- 「地下鉄を出たらランニングシューズを買いに行く」

**効果**:
- 将来のことを先取りして考える
- 行動の計画を今立てる
- 環境のキューに出会ったときに自動的に実行

### Timing（タイミング）の戦術

#### 1. 緊急性を高める（Urgency）

| 手法 | 実装 | 例 |
|------|------|-----|
| 期限 | 明確なデッドライン | 「3日以内に申し込む」 |
| 限定性 | 数量や時間の制限 | 「残り5個」「あと2時間」 |
| カウントダウン | 視覚的なタイマー | カウントダウン表示 |
| 季節性 | 時期限定のオファー | 「夏季限定」「年末セール」 |

**注意点**:
- 偽りの緊急性は避ける
- 本当に限定的である場合にのみ使用
- 頻繁すぎると効果が薄れる

#### 2. コミットメント＆リマインダー

| 要素 | 実装 |
|------|------|
| 約束を取り付ける | ユーザーに行動の約束をしてもらう |
| リマインダー設定 | メール、プッシュ通知、SMS |
| 計画づくりを促す | 「いつやるか決めましょう」 |
| 進捗確認 | 定期的に約束を思い出させる |

**例**:
- ワクチン接種のリマインドメール＋計画作成プロンプト → 接種率8%増加
- twords: 定期的な執筆リマインダー
- 投資の自動積立設定

**埋没費用効果（Sunk Cost Effect）**:
- 何かに力を注ぎ込むほど手離したくなくなる
- 難しいタスクを完遂できた人はさらに続けようとする
- ただし、わざと難しくするのは逆効果

## デザインパターンの活用

### 接触回数の多いパターン

| パターン | 説明 | 適用例 |
|---------|------|--------|
| 意思決定支援 | 選択肢を分析・提示、検討プロセスを自動化 | HelloWallet（資金管理） |
| 行動変容ゲーム | 協力やコミュニケーションを要求するゲーム | Way（協力ゲーム） |
| ゲーミフィケーション | ゲームデザインの要素を借りる | Keas（健康管理） |
| タスク管理（Planners） | 複雑なタスクの計画と管理 | Mother Earth News（菜園計画） |
| リマインダー | ToDoリストと自己管理 | Omnifocus, Nozbe |
| ソーシャルシェア | 目標を友人とシェア | RunKeeper（ランニング） |
| ゴールトラッカー | 目標設定と進捗記録 | Fitbit One, Jawbone UP |
| チュートリアル | 教育ビデオとガイド | MonkeySee.com |

### 接触回数の少ないパターン

| パターン | 説明 | 適用例 |
|---------|------|--------|
| 「違う角度で考えてみよう」 | 新しい視点を提供 | Opower（隣近所との比較） |
| コールトゥーアクション（CTA） | すぐに行動できるリンク | 政治キャンペーンメール |
| ハウツーティップス | 短いハウツー情報 | Text4Baby（妊婦向け情報） |
| シンプルなリマインダー | 計画づくりを促す | ワクチン接種リマインダー |
| ステータスレポート | 定期的なイベント情報 | Meetup.com |

### パターン選択の基準

| 基準 | 考慮事項 |
|------|---------|
| 事業・プロダクトとの適合 | カルチャーに合うものを選ぶ |
| ターゲットユーザーへの馴染み | ユーザーの日常に早く浸透するもの |
| 心理の肝をおさえる | CREATEファネルの条件を満たす |
| 柔軟性 | 複数のパターンを組み合わせてもよい |

## インターフェースレビューの手順

### ステップ1: ギャップ分析

各スモールステップごと（ページクリックやフォーム入力ごと）にCREATEの条件が適っているかを確認：

| ファネル段階 | 確認項目 |
|------------|---------|
| Cue | ユーザーはこのステップについて考え始めるか？ |
| Reaction | 直感的に「やってみよう」と思えるか？離脱したくなる要素はないか？ |
| Evaluation | メリットがコストを上回っているか？他の選択肢と比べて魅力的か？ |
| Ability | ユーザーはこのステップを実行できるか？必要なものはすべて揃っているか？ |
| Timing | 今これをやる理由があるか？後回しにされないか？ |

### ステップ2: 戦術の配置

ギャップが見つかったら、適切な戦術を適用：

| ギャップ | 戦術候補 |
|---------|---------|
| Cueが弱い | 明確なCTA、アフォーダンス、余計なもの削除 |
| Reactionが悪い | 信頼構築、社会的証明、ピア比較 |
| Evaluationが低い | 損失回避、メリット強調、コスト削減 |
| Abilityが不足 | デフォルト設定、自己効力感、実行意図 |
| Timingが合わない | 緊急性、コミットメント、リマインダー |

### ステップ3: 優先順位付け

| 基準 | 説明 |
|------|------|
| インパクト | 最も大きな効果が期待できる戦術 |
| 実装コスト | 少ない工数で実装できる戦術 |
| リスク | ユーザー体験を損なわない戦術 |
| 測定可能性 | 効果を測定しやすい戦術 |

### ステップ4: プロトタイプでテスト

| テスト方法 | 目的 |
|-----------|------|
| ユーザビリティテスト | 実際にユーザーが使えるか |
| A/Bテスト | どちらの戦術が効果的か |
| アナリティクス | 実際の行動データで検証 |
| ヒートマップ | ユーザーの注目点を確認 |

## 倫理的配慮

### 操作 vs 手助け

| 操作（避けるべき） | 手助け（推奨） |
|-----------------|-------------|
| ユーザーを騙す | 正直で透明性のある情報提供 |
| 不利益を隠す | リスクを明確に説明 |
| 選択肢を不当に制限 | 適切な選択肢を提供 |
| 中毒性を利用 | 本当に有益な行動を支援 |

### チェックリスト

- [ ] ユーザーの最善の利益を優先しているか？
- [ ] 透明性があり、隠れた意図がないか？
- [ ] ユーザーが十分な情報に基づいて決定できるか？
- [ ] ユーザーがいつでも選択を変更できるか？
- [ ] 脆弱な人々を不当に利用していないか？

## 戦術の組み合わせ方

### 相乗効果のある組み合わせ

| 組み合わせ | 効果 |
|-----------|------|
| 社会的証明 + ピア比較 | 他者の行動と自分の比較で動機づけ強化 |
| デフォルト設定 + 自己効力感 | 簡単に始められ、続けられる |
| 緊急性 + 損失回避 | 今すぐ行動する強い動機 |
| リマインダー + 実行意図 | 計画を思い出し、実行につなげる |

### 避けるべき組み合わせ

| 組み合わせ | 問題 |
|-----------|------|
| 過度の緊急性 + 複雑なタスク | ストレスと失敗 |
| 多数のCTA + 複雑な選択肢 | 認知過負荷 |
| 強すぎる損失回避フレーミング | ネガティブな感情 |

## まとめ

**インターフェース戦術の成功の鍵**:
1. CREATEファネルのすべての条件を満たす
2. 各ステップで適切な戦術を適用
3. ユーザー視点でギャップを見つける
4. プロトタイプでテストし、データで検証
5. 倫理的配慮を忘れない

**行動変容デザインの基本原則**:
- 行動だけに軸足をおきすぎず、使いやすさなどのプロダクトデザインの基本を忘れない
- プロダクトが使われなければ、行動を変えることはできない
- 興味の湧かない、つまらないプロダクトに振り向いてもらうのは難しい
- 計画性と創造性を両立させる

**ビヘイビアプランとの関係**:
- ビヘイビアプランは「何を（what）」を定義
- インターフェース戦術は「どのように（how）」を実装
- 両者を分けることで、相互に強く意識しすぎることを避ける
- ビヘイビアプランに引っ張られすぎないインターフェースデザイン
