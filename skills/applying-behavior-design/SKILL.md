# 行動変容デザイン (Behavioral Design)

## 概要

行動変容デザインとは、プロダクトを通じてユーザーの行動を変え、望ましい成果を達成するための体系的な方法論です。ユーザーに「やるべきこと」を説得するのではなく、自然に行動したくなる・行動できる環境と仕組みをデザインします。

### いつ使うか

- ユーザーの習慣を変えたい（運動、食事、学習など）
- エンゲージメントを高めたい
- 有益な行動を促進したい（貯蓄、セキュリティ対策など）
- 悪い習慣をやめさせたい（座りすぎ、無駄遣いなど）
- 複雑な行動を簡単にしたい

### UI/UXデザインとの違い

このスキルは**行動心理学に基づくプロダクト設計手法**です。視覚的なUI/UXデザイン原則については `applying-design-guidelines` スキルを参照してください。

## CREATEアクションファネル

ユーザーが行動を実行するには、5つの前提条件がすべて同時に整う必要があります。

| ステージ | 説明 | プロダクトの役割 |
|---------|------|----------------|
| **Cue（キュー）** | 行動を思いつくきっかけ（外部キュー：通知、画面配置／内部キュー：空腹、退屈） | 明確で一貫性のあるキューを提供し、既存のルーティンに関連付ける |
| **Reaction（反応）** | キューへの直感的・瞬間的な反応（楽しそう？信頼できる？） | 信頼を構築し、初回体験を最適化し、ポジティブな連想を形成する |
| **Evaluation（評価）** | 費用対効果の意識的判断（メリット vs コスト） | 価値を明確化し、摩擦を減らし、他の選択肢より魅力的にする |
| **Ability（アビリティ）** | 実行可能性（プラン・リソース・スキル・自信） | 明確なアクションプラン、必要なリソース、自己効力感の向上 |
| **Timing（タイミング）** | 今やるべき理由（緊急性・具体性・一貫性） | 緊急性の創出、具体的な実行時刻、事前コミットメント |

### ファネルの特性

- **相対的**: すべてのステージで他の選択肢との競合がある
- **個人差**: ユーザーごとに通過難易度が異なる
- **トレードオフ**: あるステージの改善が他のステージに影響する
- **繰り返し**: 行動のたびにファネルを通過する
- **変化**: 通過するたびにファネル自体が変化する

### 習慣の場合

習慣化された行動では、キュー→反応→行動が自動的に実行されるため、評価とタイミングのステージは省略されます。

## 3つの行動変容戦略

戦略の優先順位は **チート > 習慣化 > 意識的行動** です。

| 戦略 | 概要 | いつ使うか | 効果 |
|------|------|-----------|------|
| **1. チート戦略** | ユーザーの作業負荷をプロダクトが肩代わりする | 行動を自動化・デフォルト化できる場合 | 最も効果的 |
| **2. 習慣化戦略** | キュー→ルーティン→リワードのループを構築する | 繰り返す行動で明確なキュー・リワードがある場合 | 持続的な行動変容 |
| **3. 意識的行動戦略** | 意識的な行動決定を支援する | 上記2つが使えない場合 | 最も労力がかかる |

### 1. チート戦略：ユーザーの負荷を減らす

**目的**: 行動そのものをなくし、同意だけで済むようにする。

#### 一回きりの行動

| 手法 | 説明 | 事例 |
|------|------|------|
| **デフォルト化** | デフォルトで「はい」、オプトアウト可能にする | 401(k)自動登録（加入率約2倍）、カメラの賢いデフォルト設定 |
| **「ついで」化** | 既存の行動に組み込む | ヨード添加塩、宝くじ付定期預金 |

#### 繰り返す行動

| 手法 | 説明 | 事例 |
|------|------|------|
| **自動化** | 繰り返す行動を一回きりの設定にする | 活動量計（運動自動記録）、パーソナルファイナンスアプリ（支出自動分類） |
| **自動化+デフォルト化** | 最も強力な組み合わせ | 401(k)の自動引き落とし |

#### チートの倫理性

- ユーザーの同意が前提（インフォームドコンセント）
- オプトアウトの手段を提供
- 最終目標（健康、貯蓄など）が明確で価値がある
- ユーザーの努力を強いることが目的ではない

### 2. 習慣化戦略：自動操縦モードをつくる

**目的**: キュー→ルーティン→リワードのループで直感的な行動を形成する。

#### 習慣の構築手順

1. **ルーティン**: 深く考えず繰り返せる行動を設計
2. **リワード**: ユーザーにとって価値あるリワードを設定
3. **キュー**: シンプルで曖昧さのないキューを配置
4. **認識可能性**: キュー・ルーティン・リワードを明確に認識できるようにする
5. **実行可能性**: ユーザーが行動したいと思い、実行可能であることを確認
6. **繰り返し**: キュー→ルーティン→リワードを繰り返す
7. **記録**: 完了率と時間を測定し、改善する

#### 習慣の特性

| 要素 | ポイント |
|------|---------|
| **キュー** | 目的が絞られ、曖昧さがない。他の行動に直接結びついている、または決まった時刻に発生する |
| **ルーティン** | 複雑でもよい（運転など階層化された習慣の集合でも可） |
| **リワード** | 常に必要ではない。ランダムなリワードは強力だが消滅しにくい |

#### 既存の習慣を変える

習慣は脳内に永続的に残るため、完全に消すことはできません。以下の5つの選択肢を検討します。

| 選択肢 | 説明 | 事例 |
|-------|------|------|
| **1. キューを避ける** | 環境からキューを取り除く | 禁煙のために灰皿を捨てる |
| **2. ルーティンを置き換える** | 同じキューに別の行動を結びつける | 退屈→ネットサーフィンの代わりに運動 |
| **3. 意識で干渉する** | 習慣に気づき意識的に止める | マインドフルネス、意志の力（最も困難） |
| **4. マインドフルネス** | キューに反応しないよう訓練する | 瞑想・呼吸法 |
| **5. 新しい行動で追い出す** | より強い習慣で上書きする | 小さなステッカー→大きな看板への同意 |

### 3. 意識的行動戦略：熟慮を支援する

**目的**: CREATEアクションファネルのすべてのステージを通過させる。

#### プロダクトの役割

- **キュー**: 日常環境に配置、通知を工夫、既存ルーティンに関連付け
- **反応**: 信頼構築、初回体験の最適化、ユーザー行動の観察
- **評価**: メリットの明確化、コストの削減、価値の実感
- **アビリティ**: 明確なプラン、リソース提供、自己効力感の向上
- **タイミング**: 緊急性の創出、具体的な時刻設定、事前コミットメント

#### ユーザー確認の原則（AskUserQuestion）

以下の場合は必ずユーザーに確認してください：

- 複数のアプローチが考えられる場合
- デフォルト化の是非が曖昧な場合
- 倫理的配慮が必要な場合
- ユーザーの価値観に依存する判断
- プライバシーやデータ利用の範囲

## 戦略選択フローチャート

```
行動変容が必要
    ↓
[Q1] 行動を自動化・デフォルト化できるか?
    YES → チート戦略を採用
    NO  → [Q2]へ
    ↓
[Q2] 繰り返す行動で、明確なキュー・リワードがあるか?
    YES → 習慣化戦略を採用
    NO  → [Q3]へ
    ↓
[Q3] 既存の悪い習慣を変える必要があるか?
    YES → 習慣化戦略（習慣の変更手法）を採用
    NO  → 意識的行動戦略を採用
```

## デザインプロセス

行動変容デザインは5段階のプロセスで進めます。各段階の詳細は対応するファイルを参照してください。

### 1. 心の働きを理解する

**目的**: 意思決定の仕組みと行動心理学の基礎を学ぶ。

- 二重プロセス理論（システム1/システム2）
- 意思決定の12の学び
- 認知バイアスとヒューリスティクス
- 習慣の心理メカニズム

→ 詳細は [PSYCHOLOGY.md](PSYCHOLOGY.md) を参照

### 2. 適切な成果・行動を見つける

**目的**: プロダクトの目標と変えるべき行動を明確にする。

- ターゲットアウトカムの定義
- ユーザー行動リストの作成
- MVA（実用最小限のアクション）の特定
- 行動ペルソナの作成
- ターゲットアクション選択

→ 詳細は [DISCOVERY.md](DISCOVERY.md) を参照

### 3. コンセプトデザイン

**目的**: ビヘイビアプランを作成し、行動を促す文脈をデザインする。

- 行動の構造化（チャンキング、簡素化、デフォルト化）
- 環境の構築（動機づけ、キュー、フィードバック、気が散るものの排除）
- ユーザーの準備（セルフナラティブ、自己効力感、教育）

→ 詳細は [BEHAVIOR-PLANS.md](BEHAVIOR-PLANS.md) を参照

### 4. インターフェースデザイン

**目的**: CREATEファネル各段階に対応するUI戦術を実装する。

- キューの戦術（CTA、アフォーダンス、余計なもの削除）
- 反応の戦術（信頼構築、社会的証明、ピア比較）
- 評価の戦術（損失回避、フレーミング、報酬）
- アビリティの戦術（自己効力感、デフォルト、実行意図）
- タイミングの戦術（緊急性、コミットメント、リマインダー）

→ 詳細は [INTERFACE-TACTICS.md](INTERFACE-TACTICS.md) を参照

### 5. 測定と改善

**目的**: プロダクトの効果を測定し、ボトルネックを発見して改善する。

- 効果測定の設計
- A/Bテスト・ランダム化比較試験
- 因果マップによる障害発見
- CREATEファネルを使った仮説検証
- 改善サイクルの実践

→ 詳細は [MEASUREMENT.md](MEASUREMENT.md) を参照

## 行動変容デザインチェックリスト

プロダクト設計時に以下を確認してください：

### 戦略レベル

- [ ] **戦略選択**: チート→習慣化→意識的行動の優先順位で検討したか？
- [ ] **目標明確化**: プロダクトが達成したい最終成果が明確か？
- [ ] **倫理性**: ユーザーの同意とオプトアウトの手段が確保されているか？

### CREATEファネル

- [ ] **Cue（キュー）**: 明確で一貫性のあるキューが設計されているか？
- [ ] **Reaction（反応）**: 初回体験と信頼構築が最適化されているか？
- [ ] **Evaluation（評価）**: メリットが明確で、コストが最小化されているか？
- [ ] **Ability（アビリティ）**: アクションプラン・リソース・スキル・自信が揃っているか？
- [ ] **Timing（タイミング）**: 緊急性や具体的な実行時刻が設定されているか？

### 習慣化（該当する場合）

- [ ] **キュー**: 目的が絞られ、曖昧さがないか？
- [ ] **ルーティン**: 深く考えず繰り返せる行動か？
- [ ] **リワード**: ユーザーにとって価値があり、即座に得られるか？

### 測定

- [ ] **データ収集**: 各ステージの通過率を測定できるか？
- [ ] **改善サイクル**: 仮説検証とイテレーションの計画があるか？

## ユーザー確認の原則

行動変容デザインでは、以下の場合にAskUserQuestionツールでユーザーに確認してください。

### 確認が必須の場面

1. **戦略選択の分岐点**
   - チート戦略（デフォルト化）を採用する場合
   - 習慣化と意識的行動のどちらが適切か判断できない場合

2. **倫理的配慮が必要な場合**
   - ユーザーのプライバシーに関わる自動化
   - 行動の自動化がユーザーの自律性を損なう可能性がある場合

3. **価値観に依存する判断**
   - 何をメリット・コストと感じるかがユーザーによって異なる場合
   - リワードの種類（金銭的・社会的・内発的）の選択

4. **複数のアプローチが考えられる場合**
   - 習慣のキューを何にするか（時刻・場所・既存行動）
   - リワードの設計（即座・遅延、固定・ランダム）

### 確認の例

```typescript
AskUserQuestion({
  questions: [{
    question: "貯蓄促進の戦略を選択してください",
    header: "行動変容戦略",
    options: [
      {
        label: "チート戦略（自動引き落とし）",
        description: "給与から自動的に貯蓄口座に振り込む。最も効果的だがユーザーの設定が必要"
      },
      {
        label: "習慣化戦略（毎週のリマインダー）",
        description: "毎週決まった時刻にリマインドし、手動で貯蓄を促す"
      },
      {
        label: "意識的行動戦略（ダッシュボード）",
        description: "貯蓄の価値を可視化し、ユーザーが自分で決める"
      }
    ],
    multiSelect: false
  }]
});
```

## 実装時の注意事項

### やるべきこと

- **ユーザー調査**: 現在の行動と障壁を観察する（アンケートだけに頼らない）
- **小さく始める**: MVA（実用最小限のアクション）から始める
- **測定**: 各ステージの通過率とボトルネックを測定する
- **イテレーション**: 仮説検証と改善を繰り返す

### やってはいけないこと

- **努力の美化**: ユーザーに負荷をかけることが目的化してはいけない
- **教育偏重**: 教育だけで行動は変わらない（環境と構造が重要）
- **動機づけ依存**: 動機だけでは不十分（アビリティとタイミングも必要）
- **倫理無視**: ユーザーの同意なしに自動化・デフォルト化してはいけない
