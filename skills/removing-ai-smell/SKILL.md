# AI臭の除去

## 概要

Claude Codeが出力する文章（コードコメント、説明文、ドキュメント、ユーザーへの応答）から「AI臭」を検出・矯正するスキル。日本語出力に特化し、英語LLMの訓練データに起因する不自然なパターンを除去して、人間が書いたような自然な文章にする。

---

## 適用タイミング

以下のケースで必ずこのスキルを参照する:

- コードコメント記述・レビュー時
- 技術ドキュメント（README、設計書、API仕様書）作成時
- ユーザーへの応答文章作成時
- ブログ記事、解説記事の執筆時
- プルリクエストのdescription作成時
- エラーメッセージやログメッセージの記述時

## AI臭6パターン（概要）

| パターン | 典型例 | 対処方針 |
|---------|--------|---------|
| ①視覚的な記号の残留 | emダッシュ（—）、（）多用、／並列 | 記号の濫用をやめる |
| ②文のリズムが単調 | 同じ語尾の連続、接続詞過多、温度一定 | 短文と長文を混ぜ、語尾を変える |
| ③導入と進行が説明書っぽい | 前置きが長い、「結論から言うと」、STEP多用 | 前置き削除、構造宣言をやめる |
| ④事なかれ主義なスタンス | 「一概には言えませんが」「状況によります」 | 明確に主張する、保険をかけない |
| ⑤抽象語と万能語への偏重 | 「本質」「最適化」「価値最大化」連発 | 動詞中心の具体表現に置き換える |
| ⑥変な比喩の多用 | 羅針盤、土台、柱、DNA、車の両輪、エンジン | テンプレ比喩を削除、具体的に書く |

各パターンの詳細な検出ルールと具体例は `./references/PATTERNS.md` 参照。

## 書き換えの基本原則

### 1. 簡潔さを最優先する

冗長な前置き、言い換え連打、まとめの繰り返しをすべて削る。短い文で端的に述べる。

### 2. 記号に逃げない

括弧や鍵括弧で補足を閉じ込めず、本文の流れに溶かす。記号での視覚的な構造化は最小限にする。

### 3. 断定する勇気を持つ

「一般的には」「多くの場合」「状況によります」といった保険表現を削除し、主張を明確にする。根拠があれば「これがいい」とはっきり言う。

### 4. 抽象語を動詞で置き換える

「最適化」「価値創造」「本質」といった万能抽象語を避け、「何をどうするか」を動詞中心で書く。

### 5. リズムを崩す

同じ語尾を3回以上続けない。短い文と長い文を混ぜる。接続詞は必要最小限にする。

## コードコメントでの適用ルール

コードコメントはプログラマに向けた簡潔なメモであり、説明書ではない。以下を厳守する:

### WHYを書く、WHATは書かない

```typescript
// ❌ AI臭い
// 以下の処理では、ユーザーIDをキャッシュから取得し、存在しない場合はDBから取得しています

// ✅ 人間っぽい
// キャッシュミス時のDB問い合わせを削減するため
```

### 過剰な丁寧語を使わない

```typescript
// ❌ AI臭い
// この関数は、入力された文字列を検証させていただき、エラーがあれば例外をスローさせていただきます

// ✅ 人間っぽい
// 入力文字列を検証してエラー時は例外をスロー
```

### 体言止めを活用して簡潔に

```typescript
// ❌ AI臭い
// NOTE: この実装は、将来的にリファクタリングする必要があります

// ✅ 人間っぽい
// NOTE: 将来リファクタリング予定
```

### 前置き宣言を入れない

```typescript
// ❌ AI臭い
// 以下のコードでは、次の3つの処理を順番に実行します:
// 1. バリデーション
// 2. データ変換
// 3. 保存

// ✅ 人間っぽい
// バリデーション → 変換 → 保存
```

## 記号・表記の厳守ルール（最重要）

### 「」は最小限（引用・固有名詞のみ）

```markdown
❌ 「実装」においては「品質」を「担保」することが重要です
✅ 実装では品質を担保することが重要だ
```

### （）で補足を逃がさず本文に溶かす

```markdown
❌ この方法は効率的です（ただし、メモリ使用量が増えます）
✅ この方法は効率的だが、メモリ使用量が増える
```

### ：（コロン）の直後に半角スペースを入れない

```markdown
❌ 理由: 性能上の問題
✅ 理由:性能上の問題
または
✅ 理由は性能上の問題
```

### ／で並列しない

```markdown
❌ フロントエンド／バックエンド
✅ フロントエンドとバックエンド
```

### →を避ける

```markdown
❌ 処理A → 処理B → 処理C
✅ 処理A、処理B、処理Cの順で実行
```

### 締めの定型句を入れない

```markdown
❌ ご不明な点がございましたらお気軽にお問い合わせください
❌ 以上、〜についての解説でした
❌ 参考になれば幸いです

✅ （何も書かずに終わる）
```

## 文章崩しの3原則

### 1. バランスを崩す

すべてを丁寧に説明しない。重要な箇所は詳しく、補足は簡素に。段落の長さも揃えない。

### 2. 客観を崩す

常に中立を装わず、必要に応じて主観や判断を示す。「筆者は〜と考える」ではなく「これは〜だ」と断定する。

### 3. 論理を崩す

順接の接続詞（したがって、そのため、さらに）を削る。論理構造を毎回宣言しない。読み手に推論させる余地を残す。
