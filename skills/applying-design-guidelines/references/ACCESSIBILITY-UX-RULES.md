# アクセシビリティ・UX原則ルール

> すべてのユーザーが等しく製品を利用できること。アクセシビリティは「追加機能」ではなく「基本要件」。

## 詳細ルール

### 085: リンクテキストは読み上げ機能に配慮して

- **核心**: リンクテキストは前後の文脈なしで単独で意味を成すものにする
- **理由**: 音声読み上げソフトは「リンク番号とリンクテキストのみ」を順番に読み上げるため、「ここをクリック」「続きを読む」では訳がわからない
- **実装指針**:
  - ❌ 「弊社のパンフレットのダウンロードは［ここをクリック］」
  - ✅ 「こちらから［弊社のパンフレットをダウンロード］できます」
  - ❌ 「ブログ記事 2 ［続きを読む］」
  - ✅ 「ブログ記事 2 ［ブログ記事 2 を読む］」
  - 記述的なリンクテキストは検索エンジンのインデックス精度も向上させる

---

### 086: 本文へ進むリンクを追加

- **核心**: ページトップに「本文へ進む」リンクを配置し、ナビゲーションをスキップ可能にする
- **理由**: 音声読み上げソフト利用者がページ読み込みのたびに全メニュー項目を聞かされる苦痛を回避
- **実装指針**:
  - ページトップに `<a href="#main-content">本文へ進む</a>` を設置
  - 視覚障害のないユーザーには非表示（W3C推奨CSS）:
    ```css
    #skiptocontent {
      height: 1px;
      width: 1px;
      position: absolute;
      overflow: hidden;
      top: -10px;
    }
    ```
  - テンプレートに組み込み、全ページで自動適用

---

### 087: 色情報は補助情報として扱う

- **核心**: 色情報を単独では使わず、必ず他のインジケータと併用する
- **理由**: 色覚障害者（米国だけで2,700万人）は赤と緑などの区別が困難。色情報のみでは伝わらない
- **実装指針**:
  - ❌ 赤丸・緑丸のステータスインジケータのみ
  - ✅ 赤丸 + 「エラー」テキスト、緑丸 + 「正常」テキスト
  - ❌ リンクを色のみで区別
  - ✅ リンクに下線 + 本文と明確に区別できる色
  - 色は補助的情報源として有効だが、唯一の情報源にしてはならない

---

### 088: 拡大・縮小は常に可能にする

- **核心**: `maximum-scale=1.0, user-scalable=no` を設定してはならない
- **理由**: 視覚障害者を含むすべてのユーザーが画面表示を自分の思い通りにできる権利を奪う
- **実装指針**:
  - ❌ `<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no" />`
  - ✅ `<meta name="viewport" content="width=device-width, initial-scale=1.0" />`
  - レスポンシブ対応を実装し、あらゆる画面サイズに対応
  - iOS・Android のアクセシビリティ機能（拡大鏡、ハイコントラスト等）をサポート
  - 1ピクセルも崩れない完璧なデザインは不可能。ユーザーの自由を尊重する

---

### 089: Tab移動順序は論理的に

- **核心**: `tabindex` 属性で Tab キーによるフォーカス移動順序を論理的に設定する
- **理由**: 視覚障害・運動障害ユーザーは Tab キーでページを操作するため、順序が不適切だとコンテンツを理解できない
- **実装指針**:
  - フォーカス移動順序がコンテンツの正しい理解を促すように設定
  - 例: フォーム入力フィールドで順不同にフォーカスが移動しないよう `tabindex` を指定
    ```html
    <input type="text" name="field1" tabindex="1" />
    <input type="text" name="field2" tabindex="2" />
    ```
  - 支援技術を使って実際にテストする

---

### 090: ラベル + プレースホルダを併用

- **核心**: 入力欄の上にラベルを配置し、**かつ**入力欄にプレースホルダを表示する
- **理由**: 音声読み上げソフトはラベルを読み上げるが、プレースホルダは読まない。プレースホルダのみだと支援技術で利用不可
- **実装指針**:
  - ❌ プレースホルダのみ（フォーカス時に消える、ブラウザ非対応の場合あり）
  - ❌ ラベルなし + プレースホルダ
  - ✅ ラベル（常に表示） + プレースホルダ（入力例を示す）
  - 例:
    ```html
    <label for="email">メールアドレス</label>
    <input type="email" id="email" placeholder="例: user@example.com" />
    ```
  - すべてのユーザー（障害の有無を問わず）の利便性が向上

---

### 091: ユーザーの期待に応える

- **核心**: ユーザーが過去に使った製品の使い方を手本にする（車輪の再発明をしない）
- **理由**: ユーザーは過去の経験という「お荷物」を背負っており、類似製品と大きく異なる操作方法はストレスを生む
- **実装指針**:
  - ユーザーが既知の製品と同じように動作することを期待している
  - 確立された慣行を土台に製品を構築する
  - 例: iOS・Android・Tizen のホーム画面が似通っているのは理にかなっている
  - 革新よりもユーザーの使い勝手を優先

---

### 092: デフォルト設定を慎重に選定

- **核心**: デフォルト設定は製品の UX を大いに左右する。大多数のユーザーはデフォルトを変更しない
- **理由**: ユーザーは設定メニューを開かず、デフォルト設定のまま使う。デフォルトが唯一の設定と見なすべき
- **実装指針**:
  - 以下3つの要因のバランスを取る:
    1. この設定をデフォルトにしてほしいと望むユーザーの割合
    2. 設定変更の難易度
    3. 別の選択肢の見つけやすさ
  - ❌ 新機能だからという理由だけでデフォルト化
  - ✅ エビデンスとプロの勘に基づいてデフォルトを選定
  - 例:
    - 車に乗り込むとスマホが自動で車のスピーカーに切り替わる
    - サイト分析ツールのデフォルト期間が「今週」「先週」で比較表示
    - 電話アプリで相手の名前をタップすると電話をかける（メッセージ・ビデオ通話はメニューに格納）

---

### 093: 賢いデフォルトで作業軽減

- **核心**: ユーザーテスト・A/Bテスト・実データ分析で利用頻度の高いジャーニーを見極め、最適なデフォルトを設定
- **理由**: 気の利いたデフォルトはユーザーがこなすタスク数を大幅に減らす
- **実装指針**:
  - 例: 通販サイトで「子供用パジャマ」検索時、以下を自動設定
    - カテゴリー: 子供服
    - 年齢: 2〜15歳
    - 在庫: あり
  - 「80:20の法則」を活用（上位20%のジャーニー最適化で80%のユーザーに好影響）
  - 限定的な見直しで製品全体のデフォルト設定を大きく改善できる

---

### 094: ベストプラクティスを借用

- **核心**: 確立され定着したパターンを土台にサイト・アプリを構築する。盗用ではなく「賢い選択」
- **理由**: ユーザーは人生の大半を「あなたの製品以外」で過ごしている。他のサイトと同じ動作を期待している（ヤコブの法則）
- **実装指針**:
  - 以下のようなパターンを踏襲:
    - 簡単に入力でき、フィールド間・ボタンへの移動が容易なフォーム
    - オン・オフを選択するトグルコントロール
    - 隠された手数料のない明確な価格表示
    - リンクはリンクらしく、ボタンは本物のボタンに似た体裁
    - 迅速で関連性の高い順に結果を表示する検索機能
  - 恐れず遠慮せず、他の製品のベストプラクティスを「借用」する

---

### 095: フラットデザインにも視覚的シグニファイアを

- **核心**: ミニマリズム・フラットデザインでも、操作の手がかり（シグニファイア）を削ぎ落としすぎない
- **理由**: 極端なミニマリズムは「ユーザーにとっての見つけやすさ」を低下させる
- **実装指針**:
  - ❌ Google カレンダー（iOS版）: タップ可能な要素と不可能な要素が見分けづらい、メニューが小さな `...` のみ
  - ✅ Stripe の設定画面: 画面が論理的に区切られ、ボタンが一目瞭然、削除アイコン + テキストラベル併用、省略記号（`...`）で「次の処理あり」を明示
  - ミニマリズムとシグニファイアのバランスが重要
  - UI要素の統一で使い方を覚えやすくする

---

### 096: ファイルシステムを理解できないユーザーは多い

- **核心**: 複雑なファイルシステムをユーザーに見せず、作業の成果がどこに保存されるかを明示する
- **理由**: 多くのユーザーはファイルシステムを理解していないし、理解する必要もない
- **実装指針**:
  - iPad の成功例: ファイルシステムを隠し、アプリ内にドキュメントを格納（うっかり重要なシステムファイルを削除する事故を防止）
  - ユーザーが作業の成果の保存先・読み出し方法についてメンタルモデルを組み立てられるよう明示する
  - 例: 「アプリ内に保存」「ダウンロード方式」「デスクトップとスマホで同期」等を明確に伝える

---

### 097: モバイル対応は必須

- **核心**: 「モバイルでも動く」はオプションではなく必須要件。レスポンシブ・モバイルファーストが当然
- **理由**: モバイル非対応は致命的なバグであり、SEOでは死刑判決
- **実装指針**:
  - 最新のフロントエンド開発フレームワークでレスポンシブデザインを実装
  - グレイスフルデグラデーション（小画面で不向きな要素を非表示）
  - モバイルファーストの視点でデザイン段階の作業を簡素化
  - ウェブアプリ vs ネイティブアプリ: ウェブアプリはインストール不要、プラットフォーム非依存、瞬時にアップデート可能

---

### 098: メッセージ機能は定着パターンを踏襲

- **核心**: 既に確立され定着したメッセージ機能のパターンを踏襲する
- **理由**: 完成の域に達したメッセージ機能があるのに、わざわざ新しく作ってユーザーを混乱させるな
- **実装指針**:
  - 以下のパターンを踏襲:
    - 未読メッセージ件数を表示
    - 起動すると受信トレイが開き、相手ごとにメッセージが新しい順に表示
    - 受信トレイに最新メッセージの冒頭部分を表示
    - 相手を選択するとスレッド表示（日時順）
    - メッセージを読むと未読が消え、未読件数が減る
    - スレッド内のテキスト領域で返信可能
    - **改行をサポート**（return キーで未完成メッセージが送信されない）

---

### 099: ブランドより UX を優先

- **核心**: ユーザーはブランドなど気にしない。着目するのは「製品で何ができるか」
- **理由**: メガブランド以外のブランドに目を留める者はいない。UX そのものがブランドとなる
- **実装指針**:
  - 以下のようなブランディング指針に振り回されるな:
    - ❌ 読みづらいコーポレートフォント → ✅ システムフォント使用
    - ❌ 派手なスプラッシュスクリーン → ✅ 通常の作業画面をスプラッシュに使用
    - ❌ 悪夢の自己流 UI コントロール → ✅ 標準コントロール
    - ❌ 判読不能のコントラスト比 → ✅ WCAG 基準準拠
    - ❌ 奇をてらったコピー → ✅ 明快な表現
  - ブランドではなく UX に照準を合わせる

---

### 100: ダークパターン禁止

- **核心**: ユーザーを中毒にさせたり、意のままに操る悪質なパターン（ダークパターン）を使わない
- **理由**: 守るべきは会社ではなくユーザー。道義的・倫理的責任がある
- **実装指針**:
  - 以下のようなダークパターンを避ける:
    - ❌ 覚えのない項目を勝手にカートに追加
    - ❌ 検索結果を関連性順ではなく売り込みたい順に表示
    - ❌ 広告らしくない広告
    - ❌ ユーザーの設定を勝手に変更
    - ❌ 無数のチェックボックスを正しく解除しないと登録取消不可
    - ❌ ホコリに見せかけたバナー広告
  - 中毒化を狙った設計（フックモデル、ギャンブル性）も避ける
  - 自分でも使いたくなるような UI・UX をデザインする

---

### 101: 本物のユーザーでテスト

- **核心**: 開発の早い段階から本物のユーザー（同僚・上司ではなく実際のユーザー）を対象にテストを重ねる
- **理由**: ユーザーテストは製品とユーザーを理解する必須手段。問題の早期発見で開発時間短縮
- **実装指針**:
  - 対象: 極力広範な社会層から抽出した多様なグループ（さまざまな年齢・性別）
  - 時期: できるだけ早期（プロトタイプ段階、ペーパープロトタイプ段階でも可）
  - 規模: わずか5人のユーザーテストで85%のユーザビリティ問題を掘り起こせる（Nielsen Norman Group 研究）
  - 小グループ対象でも大きな成果が得られる
  - テストなしで公開すると、ユーザーは不満を言わず離脱するだけ

## 実装チェックリスト

### アクセシビリティ基本要件

- [ ] リンクテキストは単独で意味を成すか
- [ ] 「本文へ進む」リンクを配置したか（CSS で非表示可）
- [ ] 色情報を補助情報として扱っているか（色 + テキスト/アイコン）
- [ ] 拡大・縮小を禁止していないか
- [ ] Tab移動順序は論理的か（`tabindex` で制御）
- [ ] ラベル + プレースホルダを併用しているか

### UX 設計原則

- [ ] ユーザーの期待（過去の経験）に応えているか
- [ ] デフォルト設定を慎重に選定したか
- [ ] 賢いデフォルトで作業を軽減しているか
- [ ] ベストプラクティスを借用しているか
- [ ] フラットデザインでも視覚的シグニファイアを保持しているか
- [ ] ファイルシステムを隠し、保存先を明示しているか

### モバイル・メッセージ・倫理

- [ ] モバイル対応（レスポンシブ）は完了しているか
- [ ] メッセージ機能は定着パターンを踏襲しているか（改行サポート含む）
- [ ] ブランドより UX を優先しているか
- [ ] ダークパターン（悪質な操作誘導）を排除しているか
- [ ] 本物のユーザーでテストしたか（5人以上、多様なグループ）

## よくある間違いと修正例

| 間違い | 正しい実装 |
|--------|----------|
| リンクテキスト「ここをクリック」 | 「弊社のパンフレットをダウンロード」 |
| プレースホルダのみ | ラベル + プレースホルダ |
| 色のみで状態表示 | 色 + テキストラベル/アイコン |
| `user-scalable=no` | `user-scalable` 未指定（拡大・縮小可能） |
| 独自の斬新な UI | 確立されたベストプラクティス踏襲 |
| 新機能を安易にデフォルト化 | 使用頻度・見つけやすさで判断 |
| メッセージ機能で return キー送信 | 改行サポート |
| ブランディング優先の読みづらいフォント | システムフォント or 読みやすいフォント |
| 同僚・上司のみでテスト | 本物のユーザー5人以上でテスト |

## 参考リソース

- **W3C Web Content Accessibility Guidelines (WCAG)**: アクセシビリティ国際標準
- **Nielsen Norman Group**: ユーザビリティ研究の権威
- **ヤコブの法則**: ユーザーは他のサイトで過ごす時間が圧倒的に長い
- **フックモデル**: 中毒化設計の仕組み（倫理的に避けるべき）
