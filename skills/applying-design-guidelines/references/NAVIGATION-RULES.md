## 詳細ルール

### 056: 初期ページは説明の好機
- **核心**: 新規ユーザーが開く空白ページを説明機会として活用する
- **理由**: 何も表示しないのは説明機会の損失。ベテランを邪魔せず初心者を誘導できる唯一のタイミング
- **実装指針**:
  - ToDoリストページ→リスト作成のコツ
  - プロフィールページ→略歴・アバター追加のコツ
  - 機能ベースで表示し、そのページのタスクに焦点を当てる
  - 親しみのあるイラスト・アイコンを添える

### 057: 初心者向けTipは簡単にスキップ
- **核心**: 使い方ガイドは1タップで全体スキップ可能にする
- **理由**: ベテランに毎回強制すると激怒。アップデートでリセットされる悲劇を回避
- **実装指針**:
  - 初回完了後は二度と表示しない
  - 全体を1回のアクションでスキップ可能に
  - 詳細説明の前にかいつまんだ内容紹介を表示
  - Tip自体を必要最低限に（慣習に則ったUIなら細かな説明は不要）

### 058: 無限スクロールはフィード型限定
- **核心**: 無限スクロールはInstagram・Twitterなどフィード型コンテンツにのみ使用
- **理由**: 有限リスト（メッセージ・メール・ToDoリスト）では見つけやすさ半減・迷子化
- **実装指針**:
  - フィード型（時系列・アルゴリズム並替え）のみ採用
  - スクロールバー機能不全に注意
  - フッターアクセス不可能化に対策
  - 位置保存機能必須

| 判断テーブル | フィード型 | 有限リスト |
|------------|---------|----------|
| **特徴** | 終わりのないコンテンツ | 終わりのある項目群 |
| **最適UI** | 無限スクロール | ページネーション |
| **例** | Instagram・Twitter | メール・ToDoリスト |

### 059: 無限スクロールでは現在位置を保存
- **核心**: 詳細ページから戻る時は「離れた位置」に復帰する
- **理由**: トップに戻されるとユーザーが迷子になり、見ていた項目が見つからなくなる
- **実装指針**:
  - お気に入り登録・コメント等で別ページ移動時に位置記憶
  - 戻る操作で元の位置に復元
  - 技術的に難しくても実装すべき（迷子化防止のため）
  - ページネーション時は「さっきいたページ」に戻す

### 060: 始まり・中間・終わりのあるコンテンツにはページネーション
- **核心**: 有限のリストはページ単位で区切り、現在位置を明示する
- **理由**: 始まり・中間・終わりを伝え、直感的な移動とフッターアクセスを可能にする
- **実装指針**:
  - 現在ページ、前後数ページ、最初/最後のページを表示
  - 「前へ」「次へ」ボタンは不要
  - 3ページ超なら検索・並替え・絞込み必須（認知負荷軽減）
  - ページ数を明示（例: 9,999ページ）

### 061: フィードリフレッシュ後は既読項目の次へ
- **核心**: 既読項目の詳細から戻ったら、その次の未読項目に移動
- **理由**: トップに戻すとユーザーが現在位置を見失う
- **実装指針**:
  - 記事詳細から戻る際、既読項目の次にスクロール位置を維持
  - 勝手にフィード更新しない
  - 手動更新オプション提供（Twitterの「○件の新着ツイート」表示例）
  - スクロール位置を常に把握

### 062: ジャーニーには明確な「始まり」「中間部」「終わり」
- **核心**: タスク完了時に必ずユーザーに通知する
- **理由**: 保存されたか不明なまま放置されるとユーザーは混乱する
- **実装指針**:
  - 広義ジャーニー: ユーザー登録→利用終了
  - 狭義ジャーニー: 設定変更など小規模作業
  - 完了時メッセージ例: 「メッセージは送信されました」「変更内容は保存されました」
  - Googleドキュメントの「変更内容を保存しました」表示参照
  - MailChimpのハイタッチ演出参照

### 063: どのジャーニーでも常に現在位置を明示
- **核心**: 製品内での現在位置をユーザーが把握できる手がかりを提示
- **理由**: メンタルモデル不足のユーザーが迷子になり、最悪のUXを招く
- **実装指針**:
  - セグメント化プログレスインジケータ
  - パンくずリスト
  - 保存済み/未保存インジケータ
  - 言葉による説明（現状と次のステップ）
  - 階層ごとに視覚的独自色を持たせる

### 064: 階層順に現在位置をたどれるパンくずリスト
- **核心**: トップ › 製品 › 衣料品 › パーカー 形式で階層を表示
- **理由**: 現在位置確認・前階層復帰・構造把握が容易になる
- **実装指針**:
  - 小さく目立たない配置（デスクトップ・タブレット・モバイル対応）
  - 「戻る」ボタン不要化（ブラウザ「戻る」との二重実行回避）
  - JavaScriptシングルページアプリでも実装推奨
  - 古臭いと切り捨てるトレンドより、ユーザビリティ改善を優先

### 065: オプションのジャーニーはスキップ可能
- **核心**: 必須でないページ・タスクにユーザーを閉じ込めない
- **理由**: 脱出手段がないと製品に閉じ込められる最悪体験
- **実装指針**:
  - 使い方ガイド・初回Tip等にスキップオプション
  - Slack例: 「既にSlackをお使いですか？」のテキスト表示
  - リニアでないジャーニーでは常にスキップ提供
  - 利用状況データを分析してスキップボタンサイズを最適化

### 066: eコマースのサイトは標準的なパターン踏襲
- **核心**: 製品→買物かご→支払手続きの標準フローを遵守
- **理由**: 20年間磨かれた標準パターンを変えると離脱率が上がる
- **実装指針**:
  - **製品ページ**: カテゴリ分類、価格・サイズ・色表示、「かごに入れる」ボタン
  - **買物かご**: 数量変更・削除・空に・支払いへ進む
  - **支払手続き**: 合計表示、配送先・カード情報入力、サインイン/ゲスト購入選択
  - この標準を変更しない（長年実証済み、何十億の商品販売実績）

### 067: 既存ファイル複製して編集フロー
- **核心**: ユーザーが「複製して編集」を選ぶとデータ事前入力されたファイルが開く
- **理由**: 同じ詳細を毎回入力させるのは言語道断。生産性向上に直結
- **実装指針**:
  1. ユーザーが「複製して編集」クリック
  2. システムが指定ファイルをコピー
  3. 新ファイル名（「のコピー」付加）で編集可能状態で開く
  4. フィールドに元データが事前入力済み
  5. ユーザーが変更・修正して「保存」

### 068: UI要素を必須・容易・可能の3種類に分類
- **核心**: インタラクションを頻度と重要性で3分類し、配置レベルを決定
- **理由**: 全機能平等配置すると画面混雑、パワーユーザー向け機能で初心者が威圧される
- **実装指針**:
  - **必須**: 毎回使う中核機能（例: カメラシャッター、カレンダー新規イベント）→よく目立つ配置
  - **容易**: 頻繁だが毎回ではない（例: 自撮り切替、既存イベント編集）→メニュー項目・第2レベル
  - **可能**: 稀にのみ使用（例: ホワイトバランス、繰返し設定）→深い階層に隠す
  - ユーザーテストでイテレーション繰返し

| 分類 | 使用頻度 | 配置レベル | 例 |
|-----|---------|----------|---|
| **必須** | 毎回 | 最表層・目立つ | カメラシャッター |
| **容易** | 頻繁 | メニュー項目 | 自撮り切替 |
| **可能** | 稀 | 深層・隠す | ホワイトバランス |

### 069: ハンバーガーメニューなんて使うな
- **核心**: 三本線アイコンは見つけやすさ半減、タスク難易度上昇、所要時間増加を招く
- **理由**: ナビゲーション非表示でユーザーが迷子になる
- **実装指針**:
  - **代替1**: ボトムメニュー（iOS流、4-5機能を常時表示）
  - **代替2**: タブ表示（Android流、トップに表示）
  - **代替3**: 縦ナビゲーション（左揃え、6個以下）
  - デスクトップでは絶対使用禁止
  - やむを得ない場合は「メニュー」ラベル必須

### 070: メニュー項目は下部で再表示
- **核心**: フッターでメインメニュー項目またはショートカットを再提示
- **理由**: ページトップボタンはスペースの無駄。フッターを袋小路にしない
- **実装指針**:
  - 最上位セクションへのリンクをフッターに整然配置（Firefox例）
  - ミニ・パンくずリストで1つ上のレベルに戻れる
  - フッター検索フィールドも検討
  - モバイルブラウザのトップバータップショートカットと併用

### 071: 言葉で説明するのではなく、見せろ
- **核心**: テキスト説明より実例・デモで使い方を提示
- **理由**: ユーザーは画面のテキストを読まない（ユーザーテスト実証済み）
- **実装指針**:
  - 初回「ようこそ画面」で実例提示（リピーターはスキップ可能）
  - 複雑・専門ソフトはデモ動画（終了まで視聴しんどい→スキップボタン必須）
  - 既存製品の慣習を活用（UIデザインではベストプラクティス採用は盗用ではない）
  - 一般消費者向けアプリではデモ動画不要

### 072: 隠れた部分もチラッと見せる
- **核心**: 画面外にまだコンテンツがあることをささやかな視覚的手がかりで暗示
- **理由**: ユーザーは窓（画面）に見えるもので全体のメンタルモデルを構築する
- **実装指針**:
  - YouTube例: 一時停止時に関連動画の端を表示
  - Opus One例: カラータブでほかの存在を示す
  - 画面スペース節約効果大
  - 唯一のナビゲーション手段ではないが、有効なヒント

### 073: パワーユーザー向け設定は通常非表示
- **核心**: パワーユーザー向けオプションは階層下位に隠し、通常ユーザーに見せない
- **理由**: 記憶項目数削減、複雑オプションで威圧されない、親しみやすさ向上
- **実装指針**:
  - 構成基準: JTBD（ユーザーのやりたい作業）
  - macOS「システム環境設定」例: キーボード・マウス・トラックパッド別ビュー
  - 設定が長い・複雑なら検索機能追加
  - デフォルト設定を厳選（大多数が変更不要）
  - 「7 ± 2」ルール遵守

### 074: 処理所要時間明確なタスクには全体で1本のプログレスバー
- **核心**: 小出し複数バーではなく、全体進捗を統合した1本の水平バーを表示
- **理由**: あいまいさゼロ、完了までの割合が明確、予定どおり処理されていることがわかる
- **実装指針**:
  - 左→右へ徐々に満たされる形
  - 明確な始まりと終わり
  - ソフトウェアがタスク全体量を把握・常時更新できる場合に使用
  - 「コピー中」「解凍中」「インストール中」を統合

### 075: 処理所要時間不明確なタスクにはスピナー
- **核心**: 完了までの量が不明な場合、アニメーションスピナーで処理中を示す
- **理由**: プログレスバーより情報量は劣るが、処理進行とスピナー消失=完了は伝えられる
- **実装指針**:
  - 所要時間が短いタスク（ページ再読込み等）に最適
  - 問題発生時は回転停止またはスピナー消去
  - Gmail例: 「loading」→「still loading」と段階表示
  - プログレスバーが過剰な場面で使用

### 076: ループするプログレスバーなんて最悪
- **核心**: 0→100%を繰り返すGIFアニメーションバーは使用禁止
- **理由**: 「もう少しで完了」と誤解させる悪質UI、ユーザー嫌がらせツール
- **実装指針**:
  - ローカル環境だけでなく実世界のインターネット環境でテスト
  - 進捗不明なら素直にスピナー使用
  - リニアスピナーという誤用を避ける
  - 本物のユーザーでテスト

### 077: プログレスバーには進捗率・残時間を添える
- **核心**: %または「残り○分」をバーに添えて表示
- **理由**: 待ち時間が長い場合、数値情報が便利な情報源となる
- **実装指針**:
  - 待ち時間2-3秒以上なら進捗率（%）表示
  - 一瞬で消えるなら視覚的クラッター増すだけ
  - 残時間表示も可（ただし算出難題、不正確なら%推奨）
  - 正確な残時間見積もり不可なら%で十分

### 078: 検索結果は分類して表示
- **核心**: 全結果羅列ではなく、カテゴリ別に分類し項目数も提示
- **理由**: 情報吸収速度向上、関連性高い項目へ一気移動可能
- **実装指針**:
  - フード（2）、衣類（1）、日用品（2）のように分類
  - 各カテゴリーの項目数を括弧書き
  - 長いカテゴリーの価値判断を可能に
  - Google級のUX提供を目指す

### 079: 検索結果は関連度の高い順に表示
- **核心**: 最も関連度の高い結果を最上位に配置
- **理由**: ユーザーはGoogle並みの検索精度を期待する
- **実装指針**:
  - 検索アルゴリズム改善（TF-IDF等の実証済み手法活用）
  - フィルターのデフォルト見直し（「最寄り」ではなく「最適」）
  - 売りたい順・広告順の表示は顧客の怒り買う→避けよ
  - サイト利用状況データ徹底分析、関連キーワード人気推移考慮
  - 並び順変更・絞込みコントロール提供

| 悪い理由 | 対策 |
|---------|------|
| **アルゴリズムがまずい** | TF-IDF等実証済み手法、市販検索ソフト活用 |
| **フィルターのデフォルトがまずい** | 実用本位で選定、ユーザーが変更可能に |
| **売りつけ優先** | ユーザーのニーズ優先、顧客怒り回避 |

### 080: 未保存はタイトルバーで警告
- **核心**: 保存済み/未保存状態をタイトルバーに視覚的に明示
- **理由**: ユーザーの作業時間・労力への敬意表明。推測・記憶を強いる製品はあり得ない
- **実装指針**:
  - 黒丸表示または「保存されていません」テキスト
  - 自動保存機能も検討（ただし創作系アプリでは破壊的可能性あり）
  - ユーザーが⌘+S・Ctrl+Sで保存必要性をひと目で認識
  - 作業現況を常にユーザーに知らせる

### 081: アプリの評価依頼のポップアップなんてやめろ
- **核心**: 画面全体を覆う評価強要ポップアップは絶対禁止
- **理由**: ユーザーの作業を邪魔し、アプリではなく企業のためにしかならない
- **実装指針**:
  - 評価リンクを控えめに配置
  - ポップアップ・プッシュ通知での評価依頼禁止
  - 「気に入りましたか？」→「はい」→「今はしない」ボタンなし画面の悪質パターン回避
  - 本当に気にかけるユーザーは自発的にレビュー書く
  - リピーター確保に大量プッシュ通知は逆効果

### 082: 起動画面で自社ミッション・ビジョンの宣伝なんかするな
- **核心**: スプラッシュスクリーンで通常作業画面をそっくり再現
- **理由**: ユーザーはビジョン・スローガンに関心なし、素早く作業開始したいだけ
- **実装指針**:
  - 通常作業画面を形だけ再現（実体未読込でも）
  - 読込完了後に本物のインタフェースに移行
  - 準備間に合わない要素はクリック時にスピナー表示
  - ログイン画面のみ企業ブランディング可
  - 読込速度向上の印象を与える

### 083: 「弊社のビジョン」に関心のあるユーザーなんていない
- **核心**: 企業理念・使命・ビジョンの押し売りを排除し、JTBD解決に集中
- **理由**: ユーザーの目標はJTBD解決。「より良き世界」スローガンは客観性の欠如の証
- **実装指針**:
  - マッチングアプリ例: プロフィール作成・出会い実現がJTBD
  - 御大層なミッション・ストック写真の使い方ガイド禁止
  - Google初期UI例: シンプル・ブランドなし・社是なし、「関連性高い検索結果」で勝利
  - 客観性重視、ユーザーの立場で考える

### 084: 通知項目は細かく指定できるように
- **核心**: 通知イベントごとにON/OFF細かく設定可能にする
- **理由**: 重視イベントはユーザー・時期で異なる。細かな設定不可→すべてオフの確率高
- **実装指針**:
  - Instagram例: 「いいね！」は通知不要、ダイレクトメッセージは通知可とユーザーが選択
  - システム・ブラウザレベルで「通知すべてオフ」可能
  - 細かな設定実装でライバル引き離せる
  - 大量プッシュ通知はUX・維持率の点で逆効果
  - 雑音（ノイズ）の低さ→「すべてオフ」率低下
