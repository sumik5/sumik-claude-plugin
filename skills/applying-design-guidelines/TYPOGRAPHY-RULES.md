# タイポグラフィ・テキストルール

> テキストはUIの最重要要素。書体・サイズ・コントラストの選択がユーザビリティを左右する。

## クイックリファレンス

| # | ルール | Do | Don't |
|---|-------|----|----|
| 002 | 書体は最多でも2種類に | 見出し用1種＋本文用1種 | 3種類以上の書体混在 |
| 003 | システムフォント優先 | OSネイティブフォント活用 | カスタムフォントで本文表示 |
| 004 | フォントサイズで情報階層化 | 2-3段階のサイズで重要度表現 | 均一サイズでフラットな表示 |
| 005 | 本文は標準サイズで | 16px / 行間1.5 / letter-spacing: normal | 固定サイズ・拡大縮小不可 |
| 006 | 省略記号で次ステップ予告 | ラベルに「...」で追加操作示唆 | 即実行と多段階操作の区別なし |
| 007 | 大文字小文字の区別なし | メール・URLは区別なしに | パスワード以外で区別を要求 |
| 008 | コントラスト比4.5:1以上 | 文字色と背景色のコントラスト確保 | コントラスト不足の配色 |
| 009 | 用語は製品内で統一 | 同一概念には一貫した名称 | 画面ごとに異なる呼称 |
| 010 | sign in/outを使用（英語） | 現実世界の経験と対応 | log in/out等の技術用語 |
| 011 | sign upを使用（英語） | 親近感ある表現 | registerでの強制的印象 |
| 012 | 人間中心の表現 | 「顧客の詳細情報を編集」 | 「顧客を編集」 |
| 013 | 能動態で記述 | 「再起動してください」 | 「再起動される必要があります」 |

## 詳細ルール

### 002: 書体は最多でも2種類に

- **核心**: アプリ・サイト全体で使う書体を最多でも2種類（見出し用・本文用）に限定する
- **理由**: 多数の書体は視覚ノイズを生み、画面把握の認知負荷を増やす。ブランディング用カスタムフォントは読みやすさより視覚インパクト重視のため本文に不向き
- **実装指針**:
  - **見出し・タイトル**: カスタムフォント可（企業・製品ブランディング効果）
  - **本文・UIラベル**: 長年実証された読みやすい書体
  - **強調**: 既存書体内で斜体・太字・傍点を使用
  - **禁止**: 3種類以上の書体併用

**判断基準**:

| 要素 | 推奨書体 | 理由 |
|-----|---------|------|
| 見出し・ロゴ | カスタムフォント可 | ブランディング・視覚インパクト |
| 本文・説明文 | システムフォント・実績ある書体 | 長時間読む要素は読みやすさ優先 |
| ボタン・ラベル | 本文と同じ書体 | 一貫性・視覚ノイズ削減 |

### 003: あえてウェブフォントを使う必要はない

- **核心**: 本文にはカスタムフォントではなくシステム搭載フォントを使用する
- **理由**:
  - カスタムフォントはダウンロード時間（3秒以上）が発生
  - FOUC（Flash of Unstyled Content）でスタイル未適用状態が一瞬表示される
  - レスポンシブデザイン・機種差で表示が微妙に異なる
  - システムフォントは読み込み高速・レンダリング鮮明
- **実装指針**:
  - **許容**: 見出し・タイトルでのカスタムフォント（ブランディング効果）
  - **推奨**: 本文・UIコントロールはCSSでシステムフォント指定
  - **理由**: デバイスごとに最適化された美しい書体が既に搭載されている

**システムフォント指定例**:

```css
body {
  font-family: -apple-system, BlinkMacSystemFont,
               "Segoe UI", Roboto, "Helvetica Neue", Arial, sans-serif;
}
```

### 004: フォントサイズで情報の階層を表現せよ

- **核心**: 情報の重要度をフォントサイズの段階（2-3種類）で視覚的に表現する
- **理由**: ジャーナリズムで実証された手法。ユーザーは大きいサイズから順に視線が移動し、重要情報を瞬時に把握できる
- **実装指針**:
  - **最重要情報**: 最大サイズ（見出し・タイトル）
  - **詳細情報**: 中サイズ（サブ見出し）
  - **補足情報**: 標準サイズ（本文）
  - **制限**: フォントサイズは最多3段階まで
  - **アンチパターン**: すべて同一サイズで表示

**カレンダーUIの例**:

| 要素 | サイズ | 目的 |
|-----|-------|------|
| イベントタイトル | 24px | 最初に目を引く |
| 開始時刻・場所 | 16px | 次に確認する情報 |
| 詳細説明 | 14px | 必要時に読み進める |

### 005: 本文は標準の文字サイズで

- **核心**: 本文テキストはデフォルトで読みやすいサイズに設定し、ユーザーによる拡大縮小を許可する
- **理由**: 固定サイズは過去のもの。ほとんどのブラウザ・デバイスでユーザー補助機能（拡大・リーダーモード・ハイコントラスト）が利用可能
- **実装指針**（英文基準）:
  - `font-size: 16px`（本文標準）
  - `line-height: 1.5`（行間）
  - `letter-spacing: normal`（文字間隔は指定不要）
  - **禁止**: `user-scalable=no` 等でデバイスのスケーリング機能を無効化

**注意**: 文字間隔は通常指定不要。ブラウザのテキストレンダリングエンジンが最適化済み。

**CSS例**:

```css
body {
  font-size: 16px;
  line-height: 1.5;
  letter-spacing: normal; /* または指定しない */
}
```

### 006: まだ先があることは省略記号で表せ

- **核心**: アクション実行前に追加ステップ（確認・入力）がある場合、ラベルに「...」を付ける
- **理由**: ユーザーは「...」から「直ちに実行されず、次の画面で承認/キャンセルできる」と推測する。インビジブルデザインの好例
- **実装指針**:
  - **追加ステップあり**: 「削除...」「送信...」「メールで送信...」
  - **即実行**: 「新規タブ」「コピー」「更新」
  - **効果**: 出しゃばらず、使ううちに無意識に意味が伝わる

**例**:

| ボタンラベル | 動作 |
|------------|------|
| 「削除...」 | 確認ダイアログ表示 → ユーザー承認 → 削除 |
| 「削除」 | 即座に削除（確認なし） |

### 007: 「大文字小文字の区別なし」にせよ

- **核心**: パスワード以外の入力（メールアドレス・ユーザー名・URL等）は大文字小文字を区別しない
- **理由**:
  - メール配信システム・DNSは区別しない（`Will@WillGrant.org` = `will@willgrant.org`）
  - 区別ありだとモバイルキーボードでの入力が煩雑
  - 区別ありにする理由がない場合、正体不明のエラーの原因になる
- **実装指針**:
  - **デフォルト**: 大文字小文字の区別なし
  - **例外**: パスワードは常に区別あり（セキュリティ理由）
  - **区別あり時**: ユーザーに明示的に通知

### 008: 文字色と背景色のコントラストの黄金比は4.5:1

- **核心**: WCAG 2.0基準に従い、文字色と背景色のコントラスト比を最低4.5:1以上にする
- **理由**: 視覚障害者だけでなく、視力良好なユーザーにとっても低コントラストは読みにくくイライラする。とくにモバイル画面で顕著
- **実装指針**:
  - **最低基準**: 4.5:1（WCAG 2.0 レベルAA）
  - **推奨**: 7.5:1前後（最大限の読みやすさ）
  - **除外**: ロゴ・大サイズテキスト
  - **ツール**: オンラインのカラーコントラストチェッカーで検証

**コントラスト比の例**:

| 文字色 | 背景色 | コントラスト比 | 評価 |
|-------|-------|--------------|------|
| #000000 | #FFFFFF | 21:1 | 優秀 |
| #333333 | #FFFFFF | 12.6:1 | 良好 |
| #777777 | #FFFFFF | 4.5:1 | 最低ライン |
| #CCCCCC | #FFFFFF | 1.6:1 | 不合格 |

**注意**: マーケティング部門が「ブランディングのため低コントラスト配色を要求」した場合でも、アクセシビリティ基準を優先すべき。

### 009: 用語は製品内で統一せよ

- **核心**: 同一概念・要素には製品全体で一貫した名称を使用する
- **理由**:
  - 用語は画面要素の「ラベル」としての役割だけでなく、製品の「言語環境」を構成する
  - 一貫した用語がユーザーのメンタルモデル形成を促進
  - 用語統一の欠如はユーザーの混乱・理解遅延を招く
- **実装指針**:
  - **ショッピングカート**: 全画面で「カート」（または「バッグ」で統一）
  - **ユーザープロフィール**: 全画面で「プロフィール」
  - **メール設定**: 全画面で「メール設定」
  - **禁止**: 同じ要素を画面ごとに「カート」「買い物かご」「注文リスト」等と別名で呼ぶ

**統一すべき用語の例**:

| 概念 | 統一名称（例） | 避けるべき混在 |
|-----|--------------|---------------|
| ショッピングカート | 「カート」 | カート / 買い物かご / バスケット |
| ユーザープロフィール | 「プロフィール」 | プロフィール / アカウント / マイページ |
| 設定画面 | 「設定」 | 設定 / オプション / 環境設定 |

### 010: 英語ページでは log in ではなく sign in を使え

- **核心**: 英語インタフェースではログイン/ログアウトを「sign in / sign out」と表記する
- **理由**:
  - 「log in」は「航海日誌に記入する」という18世紀の船乗り用語由来のIT用語
  - 「sign in / sign out」は現実世界（会議・受診時の受付）での経験と対応
  - ユーザーにとってより馴染み深い
- **実装指針**:
  - **推奨**: 「sign in」「sign out」
  - **非推奨**: 「log in」「log out」
  - **禁止**: 「login」「logon」（スペースなし形式は最悪）

**注意**: 日本語インタフェースでは「ログイン / ログアウト」または「サインイン / サインアウト」が定着しているため、この原則は英語UIにのみ適用。

### 011: 英語のページでは register より sign up のほうがしっくり来る

- **核心**: 英語インタフェースでの新規登録は「sign up」または「join」を使用する
- **理由**:
  - 「register（登録）」はよそよそしく強制的な印象
  - 「sign up（入会手続き）」「join（参加）」は親近感・人間味がある
  - 「sign up」は「sign in」とセットで一貫性がある
- **実装指針**:
  - **推奨**: 「sign up」「join」
  - **非推奨**: 「register」
  - **説明文**: アカウント作成の利点を明示（例: 「購入履歴の保存」「おすすめ機能」）
  - **一貫性**: ボタン・説明文で表現を統一

**例**:

| 要素 | 推奨表現 |
|-----|---------|
| ボタン | 「Sign up」 |
| 見出し | 「Join us to get personalized recommendations」 |
| 説明文 | 「Create an account to save your order history」 |

---

### 012: 常に「顧客は生身の人間」を念頭に置いた表現を

- **核心**: システム中心・企業中心ではなく、ユーザー中心の視点でテキストを記述する
- **理由**:
  - 開発者には「顧客 = データベースレコード」だが、ユーザーには「顧客 = 生身の人間」
  - システム視点の表現（「顧客を編集」「新規顧客を作成」）は違和感を生む
  - 言葉の選択がユーザーの引きつけ・混乱を左右する
- **実装指針**:
  - **システム視点**: 「顧客を編集」→ **ユーザー視点**: 「顧客の詳細情報を編集」
  - **システム視点**: 「新規顧客を作成」→ **ユーザー視点**: 「新規顧客の情報を追加」
  - **能力**: 客観性と共感力で「製品」視点を離れ「顧客」視点で見直す

**アンチパターン vs 推奨表現**:

| アンチパターン（システム視点） | 推奨表現（ユーザー視点） |
|----------------------------|---------------------|
| 「顧客を削除」 | 「顧客情報を削除」 |
| 「ユーザーを作成」 | 「ユーザーアカウントを作成」 |
| 「レコードを編集」 | 「情報を編集」 |

### 013: 動詞は受動態よりも能動態で

- **核心**: アプリ・サイト内の説明文・ラベルは受動態ではなく能動態で記述する
- **理由**:
  - 能動態は直接的で、ユーザーが頭で意味を読み解く処理過程が少ない
  - 受動態は堅苦しい官僚的印象・もったいぶった紛らわしい表現
  - 能動態は語数が少なく簡潔で、画面表示に適する
- **実装指針**:
  - **受動態**: 「In order to apply updates, your computer must be restarted.」（更新内容を反映するためには、あなたのコンピュータは再起動されなければなりません）
  - **能動態**: 「Please restart your computer to apply updates.」（コンピュータを再起動して更新内容を反映してください）
  - **受動態**: 「The "search" button should be clicked once you have entered search terms.」（検索キーワードを入力したら、「検索」ボタンがクリックされる必要があります）
  - **能動態**: 「Enter search terms and click "search".」（検索キーワードを入力して「検索」ボタンをクリックしてください）

**能動態vs受動態の比較**:

| 受動態（堅苦しい） | 能動態（明快） | 改善点 |
|-----------------|-------------|--------|
| This matter will be considered by us shortly. | We will consider this matter shortly. | 語数削減・歯切れ良く |
| Your password has been changed. | You changed your password. | 主語明確・直接的 |
| An error was encountered. | We encountered an error. | 責任主体明示 |

**注意**: 大規模組織・利害関係者・法務課・グロースハッカーの要求で受動態が増える傾向があるが、ユーザビリティを最優先し能動態で簡潔に記述すべき。
