# テックブログ・技術記事執筆リファレンス

6冊の書籍から抽出・統合した、テックブログ・技術記事執筆のための実践知識。
Claude が技術記事の作成・支援・レビューを行う際のリファレンスとして使用する。

**参照書籍:**
- 武器としての書く技術（イケダハヤト）
- プロ書評家が教える 伝わる文章を書く技術（印南敦史）
- エンジニアを説明上手にする本（開米瑞浩）
- はじめての技術書ライティング（向井領治）
- 本を書く技術 取材・構成・表現（石井光太）
- 言葉でアイデアをつくる（仁藤安久）

## 1. ネタ出し・企画

### 1.1 テーマ選定の3軸

| 軸 | 問いかけ | 具体例 |
|----|---------|-------|
| 何を使って | 技術・ツール・フレームワーク | TypeScript、Docker、Next.js |
| 誰が | 対象レベル・役割 | バックエンドエンジニア、TypeScript初学者 |
| 何をする | 目的・達成したいこと | 型安全なAPIを設計する、本番環境に初デプロイする |

3軸が明確なほど記事の輪郭が定まる。「DockerでNext.jsアプリを本番デプロイするバックエンドエンジニア向け入門」のように3軸を文章化してみる。

### 1.2 ネタの5つの切り込み方（石井光太）

| 切り込み方 | 概要 | テックブログへの応用例 |
|-----------|------|----------------------|
| 反対視点 | 通説と逆の立場から | 「マイクロサービスをやめてモノリスに戻した話」 |
| 常識打破 | 「当たり前」を疑う | 「コードレビューを廃止して開発速度が上がった」 |
| 新側面 | 既知の技術の意外な使い方 | 「SQLでグラフ探索する方法」 |
| ニッチ | 専門性の高い狭い領域 | 「RustのlifetimeエラーをLLMで解読する」 |
| タブー | 業界で語られにくいこと | 「技術的負債を意図的に作る判断をした」 |

**ニッチの価値:** マスメディアが扱わないニッチなテーマほど、特定読者にとっての価値が高い。「ニッチすぎる」と感じるテーマは情熱があれば書き続けられる。

### 1.3 アイデア創出の障壁（仁藤安久）

以下の思い込みがネタ出しを妨げる。気づいて除去することが重要。

- **「ゼロから生まなければならない」** → 既存技術の組み合わせや自分の経験で十分
- **「自分は創造的でない」** → 日々の失敗・疑問・気づきが材料になる
- **「大ヒットでなければダメ」** → 1人でも助かる記事は価値がある
- **「正しいことだけ書かなければ」** → 試行錯誤の過程も価値ある情報

### 1.4 ネタ収集の実践手法

```
毎日の問いかけ: 「今日の出来事で記事にできそうなことは？」

ネタ化できるもの:
- 詰まったバグと解決策
- 公式ドキュメントと実挙動のギャップ
- チームで議論した設計の選択
- 読んだ本・論文から気づいたこと
- トラブルシューティングの記録
- 「部分的引用 + 自分の感想」（1つのソースから4〜5記事生産可能）
```

**解像度を高める:** 書き手として日常を観察する習慣を持つ。ラーメンマニアが一杯から多くの情報を引き出すように、技術者も日々の作業から豊富なネタを見つけられる。

## 2. 読者設定

### 2.1 ターゲット分析フレーム

記事を書く前に以下を明文化する:

| 項目 | 設定例 |
|------|-------|
| 技術レベル | TypeScriptの基本構文は理解している（any型を使ったことがある） |
| 役割・立場 | フロントエンド担当、チームの型安全化に取り組み中 |
| 悩み・課題 | any型だらけのコードをどう改善すればよいかわからない |
| 読む時間帯 | 通勤時間や昼休み（集中度が低い状態でも読める） |
| 達成ゴール | 記事を読んだ後、既存コードの型改善に着手できる |

### 2.2 ターゲット設定が変える設計

**具体的な一人を想定する:** 「TypeScriptを使い始めて半年の佐藤さん。最近 `as unknown as HogeType` というコードを書いてしまい、型安全とは何かを根本から理解したい」という具体像から記事を設計する。

**Before: ターゲット不明確**
```
「TypeScriptの型システムについて解説します」
→ 誰に向けているか不明。内容が中途半端になりやすい
```

**After: ターゲット明確**
```
「any型を使わずに外部APIのレスポンスを型安全に扱う方法」
→ 対象者・課題・解決策が一目でわかる
```

### 2.3 読者の思考を先回りする

読者が「？」と思う箇所を予測して先手を打つ。

| 読者の疑問 | 対処法 |
|-----------|-------|
| 「なぜこの方法を選ぶのか」 | 選択の理由・トレードオフを明示 |
| 「自分の環境でも使えるか」 | 前提条件・バージョンを冒頭に記載 |
| 「もっと簡単な方法はないか」 | 代替手段との比較を添える |
| 「実際の業務でどう使う」 | 実用的なユースケースを提示 |

## 3. タイトル・見出し

### 3.1 タイトルの法則

**4つの要素（優先順）:**

1. **数字を入れる** → 「TypeScript型安全化の5つのパターン」
2. **役立つ感を示す** → 「〜する方法」「〜を解決する」
3. **対象者を明示する** → 「〜な人のための」「〜を使っている人へ」
4. **簡潔に** → 40文字以内を目標

**マジックワード（注意を引くワード）:**

| 種類 | 例 |
|------|---|
| 数量 | 「5つの」「10分で」「100倍速く」 |
| 感情 | 「悔しかった」「なぜ」「意外な」 |
| 時間限定感 | 「今すぐ」「2025年版」「最新の」 |
| 実証 | 「実際に使った」「本番で効果があった」 |
| 否定 | 「やってはいけない」「使ってはダメ」 |

### 3.2 タイトル Before / After

| Before（NG） | After（OK） | 改善点 |
|-------------|------------|-------|
| TypeScriptについて | TypeScriptの型エラーを5分で解消する方法 | 数字・目的が明確 |
| Dockerの使い方 | Dockerを使ったことがない人が本番環境を作るまで | 対象者・ゴールが明確 |
| 設計について考えた | ADRを使って設計判断を記録するようにした話 | 具体的な手法名を含む |
| GraphQLを学んだ | REST経験者がGraphQLに移行して気づいた7つの違い | 読者が共感できる立場設定 |

### 3.3 見出しの設計原則

見出しは「道しるべ」として機能させる。読者が飛ばし読みしても全体像を把握できるレベルに。

```
悪い見出し:
## はじめに
## 本題
## まとめ

良い見出し:
## なぜ型アサーションが危険なのか
## unknown型 + 型ガードで安全に扱う
## 既存コードへの段階的な適用手順
```

**見出しのチェックリスト:**
- [ ] 見出しだけ読んで記事の流れがわかるか
- [ ] 各見出しに「何を説明するか」が入っているか
- [ ] 見出しの抽象度が一致しているか（並列構造）
- [ ] H2は3〜5個程度か

## 4. 構成パターン

### 4.1 テックブログで使える構成テンプレート

**パターンA: 問題解決型（最も汎用）**
```
1. 問題の提示（この記事で解決する課題）
2. なぜ問題が起きるか（原因の説明）
3. 解決策の提示（コード例付き）
4. 実装手順のステップバイステップ
5. 動作確認・結果
6. まとめ・次のステップ
```

**パターンB: 比較検討型**
```
1. 比較する技術・アプローチの紹介
2. それぞれの特徴と使いどころ
3. 比較表
4. 推奨パターンと理由
5. まとめ
```

**パターンC: チュートリアル型**
```
1. 完成イメージと前提条件
2. 環境セットアップ
3. ステップ1: ～
4. ステップ2: ～
5. ステップ3: ～
6. 発展的な使い方・応用
```

**パターンD: 体験・考察型（ポストモーテム・実践談）**
```
1. 背景・状況の説明
2. 何をやったか
3. 結果（成功/失敗/気づき）
4. なぜそうなったかの考察
5. 学んだこと・次回への改善
```

### 4.2 ノンフィクション基本法則（石井光太）

```
事実 → 体験 → 意味の変化
```

テックブログへの応用:
- **事実**: 「XXというエラーが発生した」
- **体験**: 「原因調査に3時間かかり、Aを試したがダメでBを試したら解決した」
- **意味の変化**: 「この経験からYYの原則を理解した。今後はZZという方法で予防できる」

**「なぜこの記事を書いたか」を含める:** 単なる手順書ではなく、書き手の体験や視点が加わることで記事に個性と信頼性が生まれる。

### 4.3 構成の組み立て手順

1. 伝えたいことを箇条書きで出し切る（脳内ダンプ）
2. 読者の理解順に並べ替える（前提 → 応用の一方向性を守る）
3. 論理的な流れになっているか確認
4. 各章に「見出し」をつける
5. 肉付けして文章化する

### 5.1 サビから始める（結論ファースト）

```

**After（OK）:**
```
TypeScriptの`any`型は型チェックを無効化するため、本番バグの温床になります。
この記事では、any型を使わずに外部APIのレスポンスを型安全に扱う3つの方法を紹介します。

## 6. 技術的説明

### 6.1 技術文書の3条件（向井領治）

| 条件 | 意味 | 注意点 |
|------|------|-------|
| 正確 | 事実と異なる情報を書かない | バージョン情報、動作確認済みか |
| 論理的 | 理由・根拠のない主張をしない | 「〜すべき」には理由を添える |
| 平易 | 読者のレベルに合わせた言葉を使う | 専門用語は初出時に説明する |

### 6.2 分類・ラベリングの原則（開米瑞浩）

**4項目以上は分類する:**
```
悪い例（羅列）:
- キャッシュをクリアする
- 依存関係を再インストールする
- 設定ファイルを確認する
- プロセスを再起動する
- ログを確認する
- 権限を確認する

良い例（分類あり）:
## 環境系の確認
- 依存関係を再インストールする
- 設定ファイルを確認する
- 権限を確認する

## 実行系の確認
- キャッシュをクリアする
- プロセスを再起動する
- ログを確認する
```

**分類ラベルの2種類:**

| ラベル種類 | 例 | 特徴 |
|-----------|---|------|
| 単純ラベル | 「キャッシュ」「認証」 | 内容から直接取り出せる |
| 分類ラベル | 「環境系」「実行時エラー」「設計判断」 | 抽象化・一般化が必要 |

分類ラベルを使うことで、読者は「今どの範囲の話か」を把握しやすくなる。

### 6.3 エンジニア読者への説明テクニック

**結論を先に、詳細は後から（食いついたら詳しく）:**
```
悪い例:
「まずXXというライブラリについて説明します。このライブラリは1999年に...
（300字の前置きの後でやっと本題）」

良い例:
「XXライブラリを使うと、○○が解決できます（コード例）。
なぜこれで解決するかというと...（詳細説明）」
```

**比較で理解を促進する:**

一方だけを説明するのではなく、比較対象を並べる。「ダックスフンドとシェパードの体形の違いを比べることで犬の品種改良が理解できる」のと同様に、技術の違いも比較で際立つ。

```
【NG】
「useEffectの使い方を説明します」

【OK】
「useEffectとuseLayoutEffectの実行タイミングの違いから、
どちらを使うべきかを整理します」
```

## 7. コード要素

### 7.1 コードブロックの基本

```markdown
## コードブロック必須要素
- 言語指定（```typescript, ```bash 等）
- ファイル名（重要な場合）
- 実行可能な最小構成
- 期待する出力・結果
```

### 7.2 コード説明のパターン

**パターン1: 問題コード → 解決コード（Before/After）**
```typescript
// Before: any型に逃げている
const data = response.data as any;
console.log(data.user.name); // 実行時エラーの可能性

// After: unknown + 型ガードで安全に
function isUserResponse(data: unknown): data is { user: { name: string } } {
  return (
    typeof data === "object" &&
    data !== null &&
    "user" in data
  );
}

if (isUserResponse(response.data)) {
  console.log(response.data.user.name); // 型安全
}
```

**パターン2: ステップバイステップ（コードを段階的に組み立てる）**
```
1. まず最小限のコードを示す
2. 問題・制限を指摘する
3. 改善コードを示す
4. 最終的な完成形を示す
```

**パターン3: 実行例を添える**
```bash
# インストール
npm install zod

# 実行結果を明示
$ node index.ts
✓ バリデーション成功
```

### 7.3 コードコメントの方針

```typescript
// 悪い例: 何をしているか（コードを読めばわかること）
const filtered = arr.filter(x => x > 0); // 正の数でフィルタリング

// 良い例: なぜそうするか（コードから読めないこと）
const filtered = arr.filter(x => x > 0); // 負の数は集計外（仕様: 返金は別集計）
```

## 8. 文体・文章表現

### 8.1 ですます体 vs. である体

| 項目 | ですます体 | である体 |
|------|----------|---------|
| 印象 | 親しみやすい、丁寧 | 断定的、権威ある |
| 向いている記事 | チュートリアル、入門、体験談 | 技術仕様、設計論 |
| リスク | 冗長になりやすい | 読者を突き放す感 |

**原則:** テックブログはですます体が基本。ただし見出し・箇条書き・コードキャプションはである体・体言止めOK。

### 8.2 ワンテキスト・ワンテーマの法則

1記事で伝えることは1つに絞る。

```
NG: 「DockerとKubernetesとCI/CDパイプラインの構築方法」
→ 3記事に分割すべき

OK: 「GitHubActionsでDockerイメージを自動ビルドする方法」
→ 1記事の範囲として適切
```

### 8.3 文章の5要素（印南敦史）

| 要素 | 意味 | テックブログでの実践 |
|------|------|-------------------|
| センス | 読者の感性に訴える言葉の選択 | 「エラーを見ると頭が真っ白になった」等の共感ワード |
| 文法 | 正しい日本語の構造 | 主語・述語の対応、修飾語の位置 |
| リズム | 読みやすい文の長さと流れ | 長文・短文を交互に |
| 簡潔さ | 無駄な言葉を省く | 一文50文字以内を目安 |
| 削ぐ力 | 書いた後に不要な部分を削除する | 推敲で全体の20%を削る覚悟 |

### 8.4 弱気ワード・AI臭ワードの除去

**除去すべきワード:**

| 除去ワード | 代替表現 |
|-----------|---------|
| 〜かもしれません | 〜です（または断言できない場合は理由を添える） |
| 〜と思います | 〜です / 〜と考えます（根拠付き） |
| 〜な場合もあります | 〜です（例外を明示する） |
| とても重要です | なぜ重要かを説明する |
| さまざまな | 具体的に列挙する |
| 〜していただく必要があります | 〜してください |

### 8.5 リズムを作る文章技術

```
【短文と長文の交互使用】
型エラーが発生した。（短）
原因を調査したところ、APIのレスポンス型とコード内で定義した型の間に、
想定していなかったオプショナルフィールドの差異があることがわかった。（長）
その差異はこれだ。（短）

【箇条書きで視覚的変化】
問題を整理する:
- パターンA: レスポンスに `user` が存在しない場合
- パターンB: `user.email` が null になる場合
- パターンC: 配列が空配列になる場合
```

### 9.2 段落レベル（ミドル）

- [ ] 1段落1トピックになっているか

### 9.4 技術記事固有チェック

## 10. Web特有の書き方

### 10.1 Web読者の特性

| 特性 | 対策 |
|------|------|
| 最初の3行で離脱を判断する | 冒頭で記事の価値を宣言する |
| 全文を読まない（飛ばし読み） | 見出し・太字・箇条書きで要点を強調 |
| 時間が限られている | 「読了時間の目安」を冒頭に表示する |
| スマートフォンで読む | 1段落3〜4行以内、適度な空白 |
| 検索エンジン経由で来る | 記事単体で完結する自己完結性を持たせる |

### 10.2 見出し・太字の活用

**見出しは道しるべ:** 読者が途中から読んでも理解できる見出しにする。

```markdown
## NG例（内容がわからない）
## 手順

## OK例（内容が明確）
## TypeScriptの型定義ファイルを生成する手順
```

**太字は強調ではなく道標:** 1段落に1箇所以内。すべて太字にすると何も強調されない。

```
NG: **TypeScript**は**型安全**な言語で、**Javaと違い**動的な側面も持つ。

OK: TypeScriptの最大の特徴は**漸進的型付け**だ。JavaScriptのコードをそのまま使いつつ、
段階的に型を追加できる。
```

### 10.3 検索エンジンを意識したタイトル・見出し

**キーワードを自然に含める:**
- 読者が検索しそうな言葉をタイトルに入れる
- 同義語・略称は同じ文中に並記する（「TypeScript / TS」）
- 「how to」形式は日本語の「〜する方法」「〜のやり方」に対応

**ストック型記事の価値:**
「一瞬だけ話題になり1万人が読んだ記事」より「検索経由で毎日50人が読む記事」の方が長期的な価値が高い。トレンドより普遍的な問題を扱うことでストック型の記事になる。

### 10.4 段落・改行の設計

```
PC向けテックブログの基準:
- 1段落: 3〜5行（150〜250字）
- 段落間: 1行空白
- 見出し後: 本文直入り（余計な前置きを省く）

スマートフォン対応:
- 1文が長い場合は読点で改行を検討
- 箇条書きを積極的に使う
- コードブロックは横スクロール可能か確認
```

### 10.5 まとめの書き方

まとめは「記事の要約」ではなく「次の行動」へ向けた締めくくりにする。

```markdown
## まとめ

この記事で学んだこと:
- ○○の問題は、△△が原因で起きる
- 解決策は□□パターンを使うこと
- 実装のポイントは◎◎の3点

次のステップ:
- 関連記事: [○○について詳しく解説]
- 公式ドキュメント: [リンク]
- サンプルコード: [GitHubリポジトリ]
```

### 記事を書く前のチェック

```
1. ターゲット一人を具体的に想像できるか？
2. 記事を読んだ後の読者の「変化」は何か？
3. タイトルに数字・目的・対象者は入っているか？
4. 「なぜ自分が書くか」の独自視点はあるか？
```

### 文章黄金レシピ（武器としての書く技術）

```
1. 伝えたいことを1つ決める
2. 伝えたいことを箇条書きで出す
3. 読者の理解順に並べる
4. 文章に肉付けする
5. スパイス（フック・数字・具体例）を加える
```

### 良い技術記事の3要素

| 要素 | 内容 |
|------|------|
| 独自視点 | 同じテーマでも書き手の体験・考察が入っている |
| 構成力 | 読者が迷わず最後まで読める論理的な流れ |
| 普遍性 | 特定の状況だけでなく、多くの読者に応用できる知見 |
