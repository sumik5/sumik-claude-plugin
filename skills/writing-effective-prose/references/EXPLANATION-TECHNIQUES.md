## 1. 説明の基本モデル

### 脳内関所と脳内整理棚

藤沢モデルでは、人間の理解プロセスを以下のように説明する。

- **脳内関所（短期記憶領域）**: 外界の情報が最初に入る仕分け場。容量は文字にして10文字足らず
- **脳内整理棚（長期記憶領域）**: 情報が格納される場所。カテゴリー別に整理棚がある

「分かった！」とは、情報が脳内関所を通過し、脳内整理棚の一区画に収まった瞬間のことである。逆に「分からない」とは、情報が脳内整理棚に入れられなかった状態を指す。

### 脳内関所の5つの作業

| 作業 | 内容 | 説明者がすべきこと |
|------|------|------------------|
| サイズ審査 | 情報の一区切りが受け入れ可能か判断 | 短く区切って伝える（ビール瓶の原理） |
| 整理棚の選定 | 適切なカテゴリーを決定 | 最初に概要・テーマを示す |
| 情報整理・構造分析 | ムダを省き、構造を把握 | 事前に情報を整理・加工して渡す |
| 論理性審査 | 理にかなっているか判断 | 論理的に話す |
| 意味確定 | 最終的な格納先を決定 | 既存知識と接続できる表現を使う |

**核心原則**: 「分かりやすい説明」とは、脳内関所で行われるはずの作業を、説明者が事前に代行することである。説明はサービスである。

### 受け入れの素地（畑村モデル）

技術伝達において、被伝達者の頭の中に「受け入れの素地」が形成されているかどうかで、知識の吸収率は大きく変わる。

> その気になって取り組んでいるか嫌々やっているかで、知識の吸収力は10倍違ってくる。

受け入れの素地は以下によって形成される。

- まず体験させること（先に失敗させる、触れさせる）
- 最初に全体像を見せること
- 動機づけを与えること（なぜ必要かを示す）

## 2. 説明のルール集

### 基礎ルール7選（藤沢）

| ルール | 内容 | 実践のポイント |
|--------|------|--------------|
| 1. タイムラグを知れ | 聞き手は話し手より理解が遅い | ゆっくり話す。脳内整理棚の準備完了を待つ |
| 2. 要点を先に言え | 概要を最初に示すとタイムラグが短縮する | 結論から始め、詳細は後から補足する |
| 3. しみ入るように話せ | 情報を小さな塊にして間を置いて渡す | キーポイントの直後に小休止を取る |
| 4. 抽象と具体のバランス | 概要（森）と詳細（木）を行き来する | 具体例の後に「要するに〇〇です」と締める |
| 5. 説明もれを防げ | 情報共有の錯覚に注意する | 第三者チェック。初めて聞く人の視点で確認 |
| 6. 情報構造を浮かび上がらせる | 事前整理済みの情報を渡す | 情報を調理してから出す |
| 7. 相手レベルに合わせる | 省ける説明と省けない説明を判断する | 相手の知識レベルを事前に把握する |

### 要点を先に言う理由（技術ブログへの応用例）

悪い例:
```
血糖値が急激に上昇すると脂肪として蓄えられます。GI値の低い食品は...（詳細）
```

良い例:
```
低インシュリン・ダイエットとは、美味しいものを食べながらダイエットできる
画期的な方法です。カロリーではなく消化吸収の速さ（GI値）に注目します。
GI値の低い食品を選ぶだけで...（詳細）
```

→ 冒頭の概要が「脳内整理棚の選定」を助けるため、後続の詳細が整理棚に収まりやすくなる。

### 「しみ入るように話す」の正体

「ゆっくり話す」とは少し異なる。情報を小さな塊に区切り、漏斗のように適切な速度で注ぎ込む感覚である。

- 意味の区切りごとに小休止
- キーポイント直後に小休止
- 複雑な概念では小休止を多用

## 3. 論理的な伝え方

### 論理の定義（ロジカル・プレゼンテーション）

「論理的」とは「話がちゃんとつながっていること」である。つながり方は2種類しかない。

| 方向 | 意味 | チェック質問 |
|------|------|------------|
| 縦の論理（直列） | 因果関係が誰から見ても成立している | 「本当にそうなの？」と言われないか |
| 横の論理（並列） | 漏れなくダブりなく全体をカバーしている | 「それだけなの？」と言われないか |

**重要**: 論理的か否かは相手が決める。自分の論理を信奉しすぎてはならない。

### 縦の論理がつながらない3つの原因

1. **前提条件の違い**: 自分の頭の中の「隠れた前提」を言語化していない
   - 例: 「経営学を専攻したのでコンサルに必要なスキルがあります」は、経営学とコンサルのスキルが同じという前提が隠れている
   - 対処: 前提を持っていない人（経験の違う人）の視点でシミュレーションする

2. **異質なものの同質化**: 違う話を一括りにしている
   - 例: 「介護市場は拡大するから参入すべき」は、介護ベッド・介護ロボット・介護士派遣を同質に扱っている
   - 対処: 「この話はもっと細かく分けられるか」を問いかける

3. **偶然の必然化**: 偶然起こることを必然として話している
   - 例: 「店頭に置けば絶対売れます」は、顧客の来店から購買まで全て必然と見ている
   - 対処: 自分の因果関係を否定的に考えて、成立しなくなる要因を洗い出す

### MECE（横の論理）

Mutually Exclusive（互いに重複なし）、Collectively Exhaustive（全体を網羅）。

「それだけなの？」と言われないためには、全体が明示されていて漏れがない状態を作ること。ただし「全体」の定義は相手によって異なるため、「できるだけ広く細かい」状態を常に目指す。

### 4つのルール基本（出口汪）

論理を構成する基本は3つのルールである。

| ルール | 内容 | 例 |
|--------|------|-----|
| イコールの関係 | 具体例や引用で自分の主張を裏付ける | 主張＝具体例・エピソード・引用 |
| 対立関係 | 反対のものを対比して際立たせる | 自社製品の良さ＋他社の弱点を示す |
| 理由づけ | 具体例がなぜ証拠になるかを説明する | 証拠を挙げるだけでなく、なぜそれが証拠か説明する |

## 4. 「話の地図」を示す技法

### 前フリ → 結論 → 根拠の構造（出口汪）

1. **前フリ**: 「これから大切な話をします」というサインを送る
   - 「ところで、○○の件ですが」のようなフレーズでテーマを提示する
2. **結論を先に言う**: 話の最初に結論を伝える。テンションと集中力が最も高い最初の10分に結論をぶつける
3. **根拠・具体例**: 結論を裏付ける具体例は3つまで（多くても5つ）

### ピラミッド・ストラクチャー（ロジカル・プレゼンテーション）

```
[メッセージ（結論）]
    ├─ [根拠1] ─── 具体例・データ
    ├─ [根拠2] ─── 具体例・データ
    └─ [根拠3] ─── 具体例・データ
```

縦の論理（結論と根拠の因果関係）と横の論理（根拠間のMECE）の両方が成立している状態がピラミッド・ストラクチャーの理想形。

### So What? / Why So?

- **So What?（だから何？）**: 事実から何が言えるか、示唆を抽出する
- **Why So?（なぜそう言えるか？）**: 結論の根拠を問い返す

この2つの往復で論理の強度を高める。

## 5. 相手意識と「他者意識」

### 「自分の常識は誰かの非常識」（出口汪）

話が伝わらない根本原因は、他者意識の欠如にある。

> 「言いたいことを伝える初めの一歩は、誰に対しても、どんなときも『相手は自分のことを全然知らないんだ』と思うことです。」

感情語と論理語の違い:
- **感情語**: 主観に基づいた言葉。「ウザい」「かわいい」等。相手に伝わりにくい
- **論理語**: 相手に伝わるよう客観化した言葉。他者意識が前提

### 3つの伝わらない話し方のパターン

| パターン | 問題 | 原因 |
|--------|------|------|
| 話が回りくどい | 内容は伝わらず聞き手がうんざり | 聞き手の立場を考えず、丁寧さを自己満足している |
| テンポよく話すが頭に残らない | 気持ちよく聞こえるが理解されない | 「間」を取らず相手の理解を確認しない |
| 話があちこち飛ぶ | 論理が飛躍し聞き手がついていけない | 言いたいことを整理せず思いついたまま話す |

### 情報共有の錯覚（藤沢）

説明者が「相手も知っているはず」と誤解して説明を省いてしまう現象。

対処法:
1. **第三者チェック**: 自分の盲点は自分では見えないため、初めて聞く人に事前確認を求める
2. **視点を動かす**: 今日初めてあなたの説明を聞く人の気持ちになって確認する

## 6. 技術伝達の特殊性

### 「伝える」ではなく「伝わる」（畑村）

> 技術というのは本来、「伝えるもの」ではなく「伝わるもの」なのです。

伝達手段（教材・制度）をいくら充実させても、「伝わる状態」が形成されていなければ意味がない。

| 問題のある考え方 | 正しい考え方 |
|---------------|------------|
| 伝える動作を正しく行えば伝わる | 相手の頭の中に「伝わる状態」を作ることが先 |
| 教材・制度を充実させれば解決 | 受け入れの素地がなければ教材は機能しない |
| 結果でなく動作に責任がある | 伝わったかどうかは結果でしかわからない |

### 技術を伝えるための5つのポイント

1. **まず体験させる**: コツを教える前に実際にやらせる。失敗が受け入れの素地を作る
2. **はじめに全体像を見せる**: 部分を学ぶ前に、全体の中での位置づけを示す
3. **やらせたことの結果を必ず確認する**: 伝えっぱなしにしない。うまくいったら褒める
4. **一度に全部を伝えない**: 相手のレベルに合わせた量と深さで伝える
5. **個はそれぞれ違うことを認める**: 相手の理解の方法・速度・前提知識は個々に異なる

### 暗黙知の表出

技術には「何が危ないかを漠然と感じるが言語化されていない知識」（暗黙知）が含まれる。

暗黙知を形式知にする方法:
- **写真・映像**: 現場の「気（雰囲気）」も含めて伝達できる
- **陰陽二つの世界を見せる**: 良い例と悪い例の対比で暗黙の基準を可視化する
- **裏図面**: 表には出ない判断の経緯・失敗の教訓を明示的に記録する
- **語り部の育成**: 経験者が体験談として語ることで文脈ごと伝える

### 理解の壁

知識が正しく伝わるかどうかは、伝えられる側が持っている「理解のためのテンプレート（型紙）」に依存する。

- 完全に一致するテンプレートがあれば即理解
- 似たテンプレートがあれば新しいテンプレートを構築できる
- 何も関連がない場合は「理解の壁」となる

対処法: 相手が持っているテンプレートに合わせた形で情報を提示する。

## 7. 図解と視覚化

### 図解の基本原則

チャートを構成する要素は3つだけ（ロジカル・プレゼンテーション）:
- テキストボックス（概念）
- 矢印（関係・方向）
- 表（比較）

レイアウトの基本は4つ:
- 左右対比
- 上下階層
- 時系列フロー
- マトリックス

### 「一目で理解でき、誰にも誤解されない」

資料作成の最高原則。そのために捨てるべき3つ:
1. 不必要な情報（ノイズ）
2. 曖昧な表現
3. 読み手に負担をかける構成

### 位置関係・時系列の表現

- **位置関係**: 概念の上下関係・包含関係は空間的な配置で表す（上位概念を上に置く等）
- **時系列**: 左から右への流れでプロセスを示す
- **人間の感覚に逆らわない**: 重要なものを左上、同格のものを同列に配置する

## 8. 提案・プレゼンの技術

### 提案の4つの基礎能力（ロジカル・プレゼンテーション）

| 能力 | 内容 | 主要ツール |
|------|------|-----------|
| 論理思考力 | 話をつなぐ（縦と横の論理） | ピラミッド構造、MECE |
| 仮説検証力 | 相手の疑問に答える | 論点把握、仮説構築、示唆抽出 |
| 会議設計力 | 議論をまとめる | 着地点の設定、スタイル調整 |
| 資料作成力 | 紙に落とす | メッセージ作成、チャート化、スライド配置 |

### 提案が通らない4つの原因

1. 論理の縦（因果関係）がつながっていない → 「本当にそうなの？」
2. 論理の横（全体カバー）がつながっていない → 「それだけなの？」
3. 相手の論点を把握できていない → 議論がかみ合わない
4. 示唆（So What）を示せていない → 結論が「当たり前のこと」で終わる

### 仮説検証の5ステップ

1. 目的を理解する（議論のスタンスと相手の要望の把握）
2. 論点を把握する（相手が本当に知りたいことを特定する）
3. 仮説を構築する（調査前に答えを仮定する）
4. 検証を実施する（仮説を証明・反証する）
5. 示唆を抽出する（ファクトから何が言えるかを導く）

### メッセージを3行で言い切る

良いメッセージの3要素:
- **説明**: 何についての話か（文脈・背景）
- **ファクト**: 何が分かったか（客観的事実）
- **示唆**: だから何をすべきか（So What）

## 9. アウトプット型学習と伝達の強化

### 伝えることで理解が深まる（畑村）

自分が伝える立場になることを意識させることで、受け入れる側の理解が深まる。

> 一番自分の頭の中に入る知識は、自分で行動して自分でつくり上げたものです。

- **アウトプット型学習**: 覚えたことを誰かに説明することで定着する
- **フィードバックループ**: やらせて → 確認して → 褒める（または修正する）のサイクルを回す
- **守・破・離**: 最初は型を守り（守）、型を破って応用し（破）、型から離れて自分の技を作る（離）

### 技術をむしり取れる環境

理想の伝達環境とは、「欲しい人が自分でむしり取れる場」である。

- 尊敬できる先輩と共に作業する
- 工程の全体が見えている
- 実践でインプットをすぐアウトプットできる
- チェックとフィードバックが即座に得られる

## 10. チェックリスト

### 書く・話す前のチェック

- [ ] 誰に向けて伝えるか（対象読者・相手の前提知識レベル）が明確か
- [ ] 相手は「自分のことを全然知らない」という前提で書いているか
- [ ] 最初に全体像（結論・要点）を示しているか
- [ ] 隠れた前提条件を言語化しているか

### 論理のチェック

- [ ] 「本当にそうなの？」（縦の論理・因果関係）に答えられるか
- [ ] 「それだけなの？」（横の論理・MECE）に答えられるか
- [ ] 異質なものを同質として扱っていないか
- [ ] 偶然を必然として話していないか

### 表現のチェック

- [ ] 概要と詳細のバランスが取れているか（ヘリコプター視点と地上視点の往来）
- [ ] 曖昧な表現を排除しているか（「なるべく」「すごく」等）
- [ ] 具体例が適切に含まれているか（範囲が確定しているか）
- [ ] 情報共有の錯覚（説明もれ）がないか
- [ ] ノイズ（不要な情報）がカットされているか

### 図解・資料のチェック

- [ ] 一目で理解でき、誰にも誤解されない構成か
- [ ] チャートのオブジェクトとレイアウトが適切か
- [ ] メッセージが3行で言い切られているか
- [ ] 示唆（So What）が明示されているか

### 技術ブログ特有のチェック

- [ ] 読者の「受け入れの素地」を作る導入があるか（なぜこの技術が必要かを示す）
- [ ] 全体像（記事の構成・扱う範囲）を最初に示しているか
- [ ] 暗黙知（「なんとなく良くない」という感覚）を言語化しているか
- [ ] 良い例と悪い例の対比（陰陽）を使っているか
- [ ] 読者がアウトプットできる（実践できる）内容になっているか
