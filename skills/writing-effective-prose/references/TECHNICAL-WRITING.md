# 技術文書特有のノウハウ

技術書・IT系文書・電子書籍を書く際の固有ルールと実践知識。
一般的な文章術（論理構成・文レベルの技術）は他のreferenceファイルを参照すること。

---

## 1. 技術文書の3条件

技術文書はすべて以下の3条件を満たす必要がある。

| 条件 | 意味 |
|------|------|
| **正確であること** | 事実を確認し、うろ覚えや思い込みで書かない |
| **論理的であること** | 前後の文に論理的整合性がある |
| **平易であること** | わかりやすい表現を選ぶ |

この3条件が欠ければ、どれだけ文章が流麗でも技術文書としては失格になる。

## 2. IT系表記ルール

### 英数字・記号

- 英数字は**半角**（全角禁止）
- 括弧・句読点などの記号類は**全角**
- 1文書内で表記を**統一**する（混在させない）

### キーボード操作の表記

大括弧でくくり「キー」と併記する：

```
正しい例：［Ctrl］＋［C］キー
```

- 複数キーの同時押しは全角の「＋」でつなぐ
- 初心者向け文書では冒頭に表記ルールの説明を入れる
- Windows/Mac両対応の場合、**Windows式を基本**にMacを括弧で併記

### メニュー・コマンドの表記

- 画面の表示に**正確に書き写す**（「ペースト」と「貼り付け」を混同しない）
- プルダウンメニュー操作の記述：

  ```
  「ファイル」→「書き出す」→「ファイル...」
  ```

- コマンド名末尾の「...」は**半角ピリオド3個**で正確に記述
- ボタン名が不明な場合は具体的に記述：
  ```
  ✅ 「画面右上の歯車アイコンのボタン」
  ❌ 「設定ボタン」（根拠なく命名しない）
  ```
- 頻出する場合は最初に定義する：
  > （以下、本書では「設定ボタン」と呼びます）

### カタカナ表記

長音（ー）の扱いを**1文書で統一**する。

| 統一例 | 方針 |
|--------|------|
| コンピュータ | JIS規格準拠（省略形） |
| コンピューター | 長音あり（一般向け） |

どちらを選んでも構わないが、同一文書内で混在させない。

## 3. コード掲載の鉄則

### 実行結果はコピペ

> 手作業で書き写さない。実際に動作確認したコードをそのままコピー&ペーストする。

転記ミスは読者に実害を与える最大のリスクである。

### インデントにタブ禁止

電子書籍ではタブ文字が無視されてインデントが崩れる。

- **半角スペース**を使用する
- 1段につき**4個**（長いコードの場合は2個でも可）

### コード解説の2パターン

**パターンA（短い解説）**：コードの右にコメントをインラインで付ける

```bash
$ hostname  ←ホスト名を表示
```

**パターンB（長い解説）**：コードに番号を振り、下に解説を列記

```python
result = process(data)    # ①
output = format(result)   # ②
```

①`process()` は入力データを変換する
②`format()` は変換結果を表示用に整形する

## 4. スクリーンショットの撮り方

### 撮影環境の準備

- **専用環境を別途用意**（仮想PC・別ユーザーアカウント）
- 個人的なブックマーク・履歴・通知が**映り込まないようにする**

### 画面の整理

- デスクトップ背景は**無地の白・グレー**に設定する
- macOSのドロップシャドウを無効化：

  ```bash
  defaults write com.apple.screencapture disable-shadow -boolean true
  ```

### 出版物に画像を渡す場合

- オリジナルファイル（PNGなど）をそのままフォルダに入れてZIP圧縮して渡す
- 縮小・JPEG変換は出版社側が行うため、編集者に確認なく加工しない

## 5. 電子書籍特有の注意

電子書籍はリフロー（レイアウトが端末に応じて変動）するため、固定レイアウトを前提とした表現は機能しない。

### 位置参照の禁止

| ❌ NG（固定レイアウト前提） | ✅ OK（順序参照） |
|---------------------------|----------------|
| 「上の図」「下の図」 | 「前の図」「次の図」 |
| 「30ページの図」 | 「図2-1参照」（番号参照） |
| 「右の表」「左の表」 | 「次の表」 |

### キャプションの配置

キャプションは図の**前**に置く（後置するとリフロー時に図と分離するリスクがある）。

### グレースケール対応

- カラーだけで情報を区別しない（グレースケール端末でも読める設計にする）
- 実線・破線・二重線など**線種や形状でも区別できるようにする**

## 6. 見出しルール

### 階層の制限

- **4階層まで**（1.1.1.1が上限）
- 5階層以上は禁止——構成を見直す

### 文体の統一

見出しの文体を**1文書で一貫**させる：

| 統一例 | 方針 |
|--------|------|
| 「インストール方法」「設定手順」 | 体言止め |
| 「インストールする」「設定を変更する」 | 動詞止め |

体言止めと動詞止めを混在させない。

### 番号体系

番号体系も1文書で一貫させる（「第1章」「1.1節」を混在させない）。

## 7. 箇条書き・表の使い方

### 並列情報は箇条書きに

並列情報を文中に挟むのは読みにくい。独立した箇条書きにする。

**悪い例**：

> たとえば、・ワード ・エクセル ・パワーポイント などが有名です。

**良い例**：

> たとえば、以下の製品が有名です。
> - ワード
> - エクセル
> - パワーポイント

### 表のルール

- 縦横の関係（行・列の意味）を明確にする
- セル内は**簡潔**に——長文は箇条書きか別段落へ

---

## 8. 図表の参照方法

### 文中での参照

- 図表は**番号で参照**する（「図2-1」「表3-2」など）
- 「以下の図に示す」だけでは不十分——番号を必ず添える

### キャプション

図表のキャプションは**内容を端的に示す**。タイトルとして機能する文言にする。

```
✅ 図2-1 OAuth 2.0認可フロー
❌ 図2-1 図
```

## 9. 接続詞の積極的使用

技術書では接続詞を積極的に使うことで論理の流れが明確になる。

| 接続詞 | 使う場面 |
|--------|---------|
| よって・このため・したがって | 前文を根拠として次を述べる |
| なぜならば | 前文の理由を述べる |
| しかし・ところが | 前文と反対のことを述べる |
| 一方・他方 | 別の見方を述べる |
| ただし | 前文を部分的に打ち消す |
| なお・ちなみに | 付加的情報を補足する |
| たとえば | 具体例を述べる |

「接続詞が多いと稚拙に見える」という誤解があるが、技術文書では**むしろ省略すると論理の流れが不明確になる**。

## 10. 企画・構成の注意

### テーマ設定の3軸

テーマは以下の3軸で具体化する：

> **「何を使って」×「誰が」×「何をする」**

軸が曖昧なままだと、書き始めてから迷走する。

### ショートカットは後まわしの原則

- **GUI操作を基本**として説明する
- キーボードショートカットは補助的に記述する

初心者はショートカットを知らない。まずGUIで操作を理解させ、効率化手段として後から紹介する。

## 11. 推敲・校正の実務

### 粗原稿を出さない

> 推敲していない原稿は下書きにすぎない。著者の推敲が「0」の原稿は、編集者へ渡せる状態ではない。

### 長文書の注意点

3万字を超えると誰でも文章が粗雑になる。**後半のセクションは特に念入りに確認**する。

### 著者校正のツール

- **Acrobat推奨**（互換性の理由）
- 他社製PDFアプリの注釈は読み手側で表示されない場合がある

### 修正指示の書き方

- 1修正箇所には注釈1つ（複数修正を1注釈にまとめない）
- 修正指示の形式：

  ```
  旧：XX → 新：YY
  ```
