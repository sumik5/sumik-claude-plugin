# 書き換えルール詳細

## 記号・表記ルール（最重要）

### 「」は最小限（引用・固有名詞のみ）

日本語では「」を多用すると幼稚に見える。特に抽象概念を囲むのは避ける。

#### 不要な鍵括弧

```markdown
❌ 「実装」においては「品質」を「担保」することが重要です
✅ 実装では品質を担保することが重要だ

❌ 「型安全性」と「テスト」は「車の両輪」です
✅ 型安全性とテストの両方が必要だ
```

#### 必要な鍵括弧

```markdown
✅ この記事では「車輪の再発明」の問題を扱う（慣用句の引用）
✅ TypeScriptの「unknown」型を使う（キーワードの引用）
✅ ユーザーから「エラーが出る」と報告があった（発言の引用）
```

---

### （）で補足を逃がさず本文に溶かす

括弧は責任回避や補足の先送りに使われやすい。本文の流れに組み込む。

#### 括弧での補足

```markdown
❌ この方法は効率的です（ただし、メモリ使用量が増えます）
✅ この方法は効率的だが、メモリ使用量が増える

❌ TypeScriptを推奨します（Pythonも選択肢です）
✅ TypeScriptを推奨する。Pythonも選択肢として検討してほしい
```

#### 例外的に許容される括弧

```markdown
✅ この実装（バージョン2.0以降）では新しいAPIを使う（バージョン情報）
✅ SOLID原則（Single Responsibility等）を遵守する（略語の展開）
```

---

### ：（コロン）の直後に半角スペースを入れない

英語の訓練データの影響で`: `（コロン+スペース）が頻出する。日本語では不要。

#### コロンの使い方

```markdown
❌ 理由: 性能上の問題
✅ 理由:性能上の問題

または

✅ 理由は性能上の問題
```

---

### ／で並列しない

視覚的な省略記号を避け、自然な日本語で書く。

#### 並列表現

```markdown
❌ フロントエンド／バックエンド
✅ フロントエンドとバックエンド

❌ TypeScript／Python
✅ TypeScriptまたはPython

❌ 開発／運用
✅ 開発と運用
```

---

### →を避ける

図解やフローチャートの記号を本文に持ち込まない。

#### 流れの表現

```markdown
❌ 処理A → 処理B → 処理C
✅ 処理A、処理B、処理Cの順で実行

❌ リクエスト → バリデーション → レスポンス
✅ リクエストを受信してバリデーションを実行し、レスポンスを返す
```

---

### 締めの定型句を入れない

「ご質問があれば〜」「参考になれば幸いです」といった定型句は不要。

#### 削除すべき締め

```markdown
❌ ご不明な点がございましたらお気軽にお問い合わせください
❌ 以上、TypeScriptの型安全性についての解説でした
❌ 参考になれば幸いです
❌ この記事がお役に立てば嬉しいです

✅ （何も書かずに終わる）
```

---

## 内容ルール

### 安全クッションを削除する

責任回避の常套句を削る。主張を明確にする。

#### 削除すべき安全クッション

```markdown
❌ 一般的には
❌ 多くの場合
❌ 状況によって異なりますが
❌ 一概には言えませんが
❌ プロジェクトによりますが
❌ 場合によっては
```

#### 書き換え例

```markdown
❌ 一般的には、TypeScriptを使うことが推奨されます
✅ TypeScriptを使ってほしい

❌ 状況によって異なりますが、この実装は有効です
✅ この実装は有効だ
```

---

### 抽象語だけで押し切らず動詞中心の具体表現に

名詞と形容詞を減らし、動詞を増やす。

#### 抽象語の連打

```markdown
❌ 本質的な価値を最大化し、全体最適化を図る
✅ ユーザーが求める機能を優先的に開発し、システム全体の応答速度を改善する

❌ 持続可能な成長を実現する
✅ 継続的に機能を追加し、ユーザー数を増やす
```

---

### 同義語の言い換え連打をやめる

同じことを何度も言い換えない。1回で済ます。

#### 言い換え連打

```markdown
❌ 型安全性は重要であり、肝心であり、鍵となる要素であり、核となる概念です
✅ 型安全性は必須だ

❌ この実装は効率的で、高速で、パフォーマンスが良い
✅ この実装は高速だ
```

---

### 前置き宣言を入れない

「以下で解説します」「次のセクションで説明します」といった構造宣言を削る。

#### 前置き宣言

```markdown
❌ 以下の3つのポイントについて解説します
✅ （すぐに本題に入る）

❌ 次のセクションでは、実装方法を詳しく説明します
✅ （セクションタイトルで十分）
```

---

### まとめの繰り返しを削る

段落末や記事末で同じ内容を繰り返さない。

#### まとめの繰り返し

```markdown
❌ 以上、型安全性の重要性について解説しました。型安全性は実装において必須であり、any型の使用は避けるべきです。
✅ （まとめセクションを書かずに終わる）
```

---

## リズムルール

### 短い文と長い文を混ぜる

単調なリズムを崩すため、文の長さを変える。

#### リズムの例

```markdown
❌ 単調
この実装は効率的です。パフォーマンスが向上します。メンテナンス性も高いです。

✅ リズムあり
この実装は効率的だ。パフォーマンスが向上し、メンテナンス性も高い。
```

---

### 同じ語尾を3回以上続けない

語尾のバリエーションを増やす。

#### 語尾の例

```markdown
❌ 語尾が同じ
型安全性は重要です。テストも書きます。ドキュメントも作成します。

✅ 語尾を変える
型安全性は重要だ。テストも書く。ドキュメントも作成する。
```

---

### 順接の接続詞は積極的に削る

「そのため」「したがって」「さらに」を削り、文を短くする。

#### 接続詞削減

```markdown
❌ 接続詞多い
型安全性は重要です。したがって、any型を使用してはいけません。さらに、unknown型を使うべきです。

✅ 接続詞削減
型安全性は重要だ。any型は使わず、unknown型を使う。
```

---

### 段落の閉じ方を変える

段落末が毎回きれいに閉じると機械的に見える。変化をつける。

#### 段落の閉じ方

```markdown
❌ 毎回きれいに閉じる
型安全性は重要です。any型は避けてください。unknown型を使いましょう。

✅ 閉じ方を変える
型安全性は重要だ。any型は避ける。unknown型を使う。ただし、型ガードを忘れないこと。
```

---

## スタンスルール

### 推奨するならはっきり「これがいい」と言う

曖昧な推奨を避け、明確に主張する。

#### 曖昧な推奨

```markdown
❌ この方法が推奨されることが多いです
✅ この方法を推奨する

❌ 一般的にはTypeScriptを使うことが好ましいとされています
✅ TypeScriptを使ってほしい
```

---

### ダメなものは「やめろ」「使うな」と言い切る

弱い否定を避け、強く否定する。

#### 弱い否定

```markdown
❌ any型の使用はあまり推奨されません
✅ any型は使うな

❌ この実装は避けた方が良いでしょう
✅ この実装はやめてほしい
```

---

### 全方位配慮の毒にも薬にもならない文を避ける

どちらにも転べる文を書かず、立場を明確にする。

#### 全方位配慮

```markdown
❌ アプローチAとアプローチBにはそれぞれメリット・デメリットがあり、どちらが優れているとは一概に言えません
✅ アプローチAの方が優れている。アプローチBはメンテナンスコストが高い

❌ 状況に応じて適切に判断することが重要です
✅ 小規模プロジェクトならアプローチAを選ぶ。大規模ならアプローチBが必要になる
```

---

## コードコメント専用ルール

### WHYを書く、WHATは書かない

コードを読めばわかることは書かない。なぜそうしたかを書く。

#### WHATとWHY

```typescript
// ❌ WHAT（読めばわかる）
// ユーザーIDをキャッシュから取得する
const userId = cache.get('userId');

// ✅ WHY（なぜそうするか）
// キャッシュミス時のDB問い合わせを削減するため
const userId = cache.get('userId');
```

---

### 「以下の処理では〜を行っています」→ 不要

説明書的なコメントを削る。

#### 説明書的コメント

```typescript
// ❌ 説明書っぽい
// 以下の処理では、入力文字列をバリデーションし、エラーがあれば例外をスローしています
function validate(input: string) {
  if (!input) throw new Error('Empty input');
}

// ✅ 簡潔
// 空文字列は不正な入力として扱う
function validate(input: string) {
  if (!input) throw new Error('Empty input');
}
```

---

### 過剰な丁寧語を使わない

コードコメントはメモであり、手紙ではない。

#### 過剰な丁寧語

```typescript
// ❌ 過剰な丁寧語
// この関数は、入力された文字列を検証させていただき、エラーがあれば例外をスローさせていただきます

// ✅ 簡潔
// 入力文字列を検証してエラー時は例外をスロー
```

---

### 体言止めを活用して簡潔に

冗長な文末を削り、体言止めで終わらせる。

#### 体言止め

```typescript
// ❌ 冗長
// NOTE: この実装は、将来的にリファクタリングする必要があります

// ✅ 体言止め
// NOTE: 将来リファクタリング予定

// ❌ 冗長
// TODO: エラーハンドリングを追加することが必要です

// ✅ 体言止め
// TODO: エラーハンドリング追加
```

---

### 前置き宣言を入れない

「以下のコードでは〜」といった宣言を削る。

#### 前置き宣言

```typescript
// ❌ 前置き宣言
// 以下のコードでは、次の3つの処理を順番に実行します:
// 1. バリデーション
// 2. データ変換
// 3. 保存

// ✅ 簡潔
// バリデーション → 変換 → 保存
```

---

## 補足: バランスの取り方

すべてのルールを機械的に適用すると、逆に不自然になる。以下のバランスを意識する:

### 過剰に崩さない

```markdown
❌ 崩しすぎ
型安全性マジ大事。any使うなよ。絶対ダメ。

✅ 適度に崩す
型安全性は必須だ。any型は使わず、unknown型を使う。
```

### 文脈に応じて調整

- 技術ドキュメント: やや堅め（ただしAI臭は除去）
- コードコメント: 簡潔かつ自然
- ユーザー応答: 丁寧さとAI臭除去のバランス
- ブログ記事: 自然さを最優先

### 読み手を意識

- プログラマ向け: 専門用語を使い、簡潔に
- 一般ユーザー向け: 丁寧語を残し、専門用語を減らす
- チーム内文書: カジュアルさを許容
