# テクニカルライティング実践

## 概要

**ソース**: 髙橋慈子『技術者のためのテクニカルライティング入門講座 第2版』（翔泳社、2024年）、同第1版（2018年）

- 生成AI（ChatGPT/Copilot）を使った文書作成・校正の実践手法

## 1. テクニカルライティングの基本

### 1.1 「情報をわかりやすく伝える＝技術（センスではない）」

技術者の書く文章がわかりにくい理由は3つある。

| 原因 | 内容 |
|------|------|
| 一文が長く情報量が多過ぎる | 読み手の負担になる |
| 読み手を意識せず自分視点で書いている | 「これは知っているはず」という前提が誤解を生む |
| 基本的なライティング技術を活用していない | 学校教育では実用文の書き方を教えない |

**核心**: ライティングスキルは「センス」ではなく「技術」。技術は努力で向上できる。

### 1.2 ロジカルライティングとテクニカルライティングの関係

```
ロジカルライティング（論理構成）
　→ 読み手視点の筋道を作る。情報を整理してロジックツリーに落とす

テクニカルライティング（表現技術）
　→ 技術情報を正確・簡潔・わかりやすく表現する

両者を組み合わせることで、「伝わる」技術文書が作れる
```

### 1.3 ライティングは「ポータブルスキル」

| 応用領域 | 効果 |
|----------|------|
| 議事録・報告書 | 関係者から評価が高まる |
| 問い合わせ回答 | ユーザーの信頼向上 |
| 障害報告書・提案書 | 対外的な信頼獲得 |
| プレゼンテーション | 口頭でも応用できる |
| 人事評価 | 上司は報告書で仕事ぶりを判断する |

## 2. 構成の技術

### 2.1 書き出す前にすべきこと：読み手と目的の分析

**いきなり書き始めてはいけない。**

| NG | 理由 |
|----|------|
| 思いついたまま書き始める | 説明順序が崩れ、情報のダブり・漏れが発生する |
| 書き手の論理を押しつける | 読み手の視点に変換していないため伝わらない |

**書き始める前にすること**:
1. 読み手の分析（誰が読むか、技術知識レベルは何か）
2. 目的の設定（読んだ後に何をしてほしいか）
3. ロジックツリーの組み立て

### 2.2 読み手は「目の前にいる人」だけではない

**ポイント**: 文書は提出先だけでなく、さらに上の上司や他部門、取引先の役員が読む可能性がある。

- 「これは書かなくても知っているだろう」という前提は危険
- 直属の上司に向けて書いた報告書も、その上長に回覧される可能性を想定する
- 障害報告書は情報システム担当者だけでなく、経営者も読む場合がある

### 2.3 ロジックツリーで構成を設計する

**組み立て方（トップダウン推奨）**:

```
【トップ】主題（例：「○○システムの障害のお詫びと報告」）
    ↓
【第2階層】ポイント3〜4つ（読み手の疑問に答える順）
    ・何が起こったのか？
    ・原因は何か？
    ・どう対処したのか？
    ・今後はどう取り組むのか？
    ↓
【第3階層】具体的な事実・数値・日時で裏付ける
```

**メリット**:
- 全体像が見える → チームで共有すれば手戻りが減る
- 情報のダブり・漏れをチェックできる
- 書き上がるまでの時間が短縮される

### 2.4 情報の取捨選択の基準

取捨選択に迷ったら：**「読み手にとって必要な情報かどうか」** で判断する。

- 「あれもできる」「これも説明を加えて」と情報を盛り込み過ぎない
- 詳細に書くほどよいわけではない
- 必要でない情報は「参考情報」として別枠に整理する

## 3. 文章の基本テクニック

### 3.1 一文一義で書く

**原則**: 1つの文には1つの主題のみ。

| Before | After |
|--------|-------|
| このシステムはクラウド上で動作し、どこからでもアクセスでき、複数人での同時作業も可能で、コスト削減も実現できます。 | このシステムはクラウド上で動作します。どこからでもアクセスできます。複数人での同時作業も可能です。コスト削減も実現できます。 |

**なぜ**: 人が文を読む際、句点「。」を区切りとして内容を整理する。複数の内容が盛り込まれていると整理の負荷が増し、理解が困難になる。

### 3.2 一文は50字以内を目安に

- 50字以内：一気に読め、内容を認知しやすい分量
- Wordの初期設定（A4横書き40字/行）で1行＋10字が目安
- 2行を超えたら（80字以上）：区切れないか、不要な部分がないか見直す
- 新聞・技術系Webサイトの文章は50〜60字程度が多い

### 3.3 長い文を短くするパターン

**よく見られる書き癖**:

| パターン | 対処 |
|---------|------|
| 「〜（だ）が」で文をつなぐ | 文を区切り、重要な内容を前に移動する |
| 「〜ので」で文をつなぐ | 同上 |
| 丁寧すぎる過剰な敬語表現 | 簡潔な書き言葉に書き換える |
| 話し言葉の混入（「なので」「どうしても」） | 書き言葉に変換する |

**Before**:
```
当サービスはセキュリティが広く普及しているが、小規模企業での利用が少ないので、
今回、利用促進キャンペーンを開始することになりました。
```

**After**:
```
当サービスはセキュリティが広く普及しています。
しかし、小規模企業での利用は少ない状況です。
そこで、利用促進キャンペーンを開始します。
```

### 3.4 5W2H を盛り込む

「簡潔に書く」は「情報量を減らす」ことではない。

| 要素 | 問い | 重要性 |
|------|------|--------|
| Who（誰が） | 誰に、誰から | あて先・発信者の明確化 |
| When（いつ） | 日時・期限 | 期日の明記がトラブル防止に直結 |
| Where（どこで） | 場所・環境 | 作業場所や対象システム |
| What（何を） | 対象・内容 | 何をすべきかが伝わる |
| Why（なぜ） | 目的・理由 | 読み手が行動する根拠 |
| How（どのように） | 方法・手順 | 具体的な操作手順 |
| How much（いくら） | 金額・数量 | ビジネスでは必要に応じて追加 |

**Before**:
```
来週いっぱいにレビューファイルを提出してください。
```

**After**:
```
11月15日（金）17:00までに、以下の方法でレビューファイルを提出してください。
・提出先：共有フォルダ「○○プロジェクト/レビュー」
・ファイル名：「担当者名_yyyymmdd.xlsx」
```

### 3.5 タイトル・件名から目的を伝える

| NG | OK |
|----|-----|
| 打ち合わせについて | 次回打ち合わせ日程変更のお願い |
| ご相談 | 【相談】○○システム開発スケジュール調整のお願い |
| 関連資料 | 「△△イベント」関連資料 |
| 障害対応について | ○○システム障害対応報告 |

**コツ**:
- 「〜について」「〜の件」は目的が曖昧になりやすい → 具体的な動詞・目的語を加える
- 社内メールは `【要返信】` `【依頼】` などのカッコを冒頭に使うと明快
- 検索性も意識：キーワードをタイトルに入れると後から探しやすい

### 3.6 話し言葉と書き言葉の使い分け

| 話し言葉（NG） | 書き言葉（OK） |
|--------------|--------------|
| なので | したがって・そのため |
| どうしても〜してしまう | 〜することが多い |
| お出かけ（上司に対して） | 外出中 |
| ということですので | したがって |
| けど・だけど | しかし・ですが |

**注意**: 親しい先輩へのメールでも書き言葉を使う。上長が読む可能性があり、仕事への姿勢を判断されるため。

### 3.7 接続詞の使い方

**原則**: 接続詞は最小限に。文と文のつながりを考えて順番を整えれば、接続詞がなくてもスムーズに読み進められる。

| 関係 | 接続詞の例 |
|------|-----------|
| 説明・補足 | つまり、すなわち |
| 逆接 | しかし、ただし |
| 添加・並立 | また、さらに |
| 結果・結論 | したがって、そのため |

**注意**: 連続した番号（手順リスト）の後に接続詞を加えると重複になる。削除する。

## 4. より伝わる文章のコツ

### 4.1 重要な内容を先に書く（起承転結は実用文NG）

**実用文の構成**:
```
結論・目的（最初） → 理由・背景 → 詳細 → まとめ
```

「起承転結」は物語の構成。実用文で使うと読み手が最後まで読まないと要点がわからない。

**Before**（起承転結型）:
```
過去のログは「log」フォルダに収められています。
その下には日付別フォルダがあり、「past-record」フォルダが存在します。
そこに過去のログが保存されています。
```

**After**（結論先行型）:
```
過去のログは「past-record」フォルダに保存されています。
フォルダ構成は log/日付/past-record です。
```

### 4.2 動詞で行動を促す

| Before（形容詞・名詞止め） | After（動詞） |
|--------------------------|-------------|
| セキュリティソフトウェアの重要性 | セキュリティソフトウェアを最新版に更新してください |
| 会員登録が必要です | 以下の手順で会員登録をしてください |

**注意**: 「〜が必要です」「〜が求められます」は婉曲表現。操作説明では「〜してください」が明確で親切。

### 4.3 否定表現に気を付ける

**二重否定はNG**:

| NG | OK |
|----|----|
| フィルタリングソフトを使わないと、有害サイトを閲覧しないとは限りません | フィルタリングソフトを使えば、有害サイトの閲覧を防げます |
| 水洗いを行うと、故障しないとは限りません | 全体を水に浸さないでください。内部に水が入り、故障につながります |

**改善方法**:
1. 両方とも肯定形に変換する
2. 前半の否定形を残し、後半を具体的な結果（危険性）に書き換える

### 4.4 受動態と能動態の使い分け

| 場面 | 使い方 |
|------|--------|
| ユーザーが行う操作 | 能動態で書く（ユーザーを主語に） |
| 操作の結果（システムの動作） | 受動態で書く |
| 自己評価・報告書（主体性を示す） | 能動態で書く |

**Before**:
```
アイコンをタップすることでアプリが起動させられます。
```

**After**:
```
アイコンをタップします。（操作：能動態）
アプリが起動し、ホーム画面が表示されます。（結果：受動態）
```

**注意**: 「〜となります」は受け身で主体性が伝わらない。報告書では能動態に書き換える。

### 4.5 「など」を多用しない

| NG | OK |
|----|-----|
| ドライバーやレンチなどを用意してください | ドライバーとレンチを用意してください |
| ドライバーやレンチなどを用意してください（他の工具も可） | ドライバー、レンチなどの工具を用意してください |
| 取り組み内容：品質改善、コスト削減など | 取り組み内容：品質改善、コスト削減 |

**理由**:
- 「など」は範囲が曖昧になる
- 箇条書きの末尾に「など」をつけると全体が曖昧になる
- 英文に翻訳する際、「など」に相当する語句がなく誤訳の原因になる

### 4.6 最後の文でしっかり締めくくる

報告書の最後は「どこか他人ごと」で終わらせない。

**Before**:
```
以上が今回の障害に関する報告です。
```

**After**:
```
この度はご迷惑をおかけし、誠に申し訳ございません。
再発防止策を確実に実施し、今後も安心してご利用いただけるよう努めてまいります。
引き続きご愛顧のほどよろしくお願いいたします。
```

## 5. 読みやすさの向上

### 5.1 文書の必須4要素

業務文書の冒頭には以下の4要素を必ず書く。

| 要素 | 内容 | 注意点 |
|------|------|--------|
| 発信日 | 年月日（スラッシュ不可） | 「09/07」だと海外では月日が逆に解釈される恐れがある |
| あて先 | 相手の組織名・役職・氏名 | 「（株）」と省略しない。組織名には「様」ではなく「御中」 |
| 発信者名 | 発信組織・氏名 | 省略せず正確に書く |
| タイトル | 文書の目的が伝わる内容 | 具体的なキーワードを含める |

### 5.2 ブロック化して情報を区切る

| 方法 | 効果 |
|------|------|
| 改行・空行を入れる | トピックの区切りが一目でわかる |
| 右揃え・中央揃えで配置を変える | 要素が他と区別される |
| インデントで行頭位置を変える | 階層関係が視覚的に伝わる |
| 見出しを付けてブロックに分ける | 必要な情報を拾い読みできる |

**ブロック化しないと**: 最後まで読まないと必要な情報が見つからない → 読み手の負担が増大する。

### 5.3 フォントの使い分け

| フォント | 用途 |
|----------|------|
| 明朝体 | 本文（可読性が高い。長文に適している） |
| ゴシック体 | 見出し（太く視覚的に目立つ） |

**注意**: すべてゴシック体にすると文字が黒っぽく圧迫感が出る。本文と見出しにメリハリをつける。

### 5.4 階層構造と箇条書き

| ルール | 内容 |
|--------|------|
| 階層は3階層まで | それ以上深くなると関係が把握しにくい |
| 箇条書きは7項目以内 | マジックナンバー7：人が一度に把握できる上限 |
| 箇条書きには見出しを必ずつける | 何を列挙しているかが伝わる |
| 文末表現を統一する | 「。」あり/なしを混在させない |
| 文章の途中に箇条書きを入れない | 文章を完結させてから箇条書きを配置する |

### 5.5 カッコの使い分け

| カッコ | 用途 |
|--------|------|
| 「　」鍵カッコ | 固有名詞・強調・語句の引用 |
| （　）丸カッコ | 言い換え・補足（品番・製品名など） |
| NG：" " | ダブルクォーテーションは欧文記号。日本語文章には不適 |
| NG：補足を丸カッコ内に多用 | 一文が長くなり読みにくくなる |

### 5.6 カタカナ語・表記の統一

- 一般的な用語はカタカナ表記（英字では読み方がわからないリスクがある）
- 長音表記は「サーバー」「プリンター」など長音付きが現在の主流
- 組織で表記ルールをまとめておく

### 5.7 推敲のプロセス

```
下書き作成（多めに書く）
    ↓
推敲1回目：情報の漏れ・接続詞・件名との整合性
    ↓（1日置くと客観的に見られる）
推敲2回目：文表現・冗長箇所・話し言葉の除去
    ↓
レビュー（第三者チェック）
    ↓
修正・仕上げ
```

**コツ**: 最初から少ない文字数で書こうとしない。3割多めに書いて削っていく方が締まった文章になる。

## 6. 文書種別ガイド

### 6.1 ユーザーマニュアル・取扱説明書

#### 作成の基本手順

1. **企画・設計**: 読み手と目的を分析する
   - 誰が読むか（初心者か、熟練者か）
   - 何のために使うか（インストールか、日常操作か）
2. **ユースケース設計**: 機能仕様書の整理ではなく、ユーザーが「したいこと」を基準に再構成
3. **タスク分解**: 「カレーライスを作る」タスクを「切る→煮る→味付け」の手順に分けるように

#### 情報の分類

| 要素 | 内容 | 書き方のポイント |
|------|------|----------------|
| 概要文 | 作業の全体像・機能の意味 | 200字以内。操作に不要な技術背景は書かない |
| 手順文 | 操作のステップ | ステップ番号付き。10ステップ以内に収める |
| 参考情報 | 必須ではないが役立つ情報 | MEMO/TIPS/HINTなどの見出しで区別 |
| 注意文 | 守らないと危険・問題が生じること | 禁止事項を最初に書き、理由を後に書く |

#### 手順文の書き方

**Before**（ステップを区切っていない）:
```
「文書管理」アイコンをクリックし、「新しいフォルダー」を選択して
フォルダ名を入力します。権限があることを確認してから実行してください。
```

**After**（ステップに分けて操作と結果を分離）:
```
1. 書き込み権限があるフォルダーを確認します。
2. ホーム画面を開きます。
   画面上部のメニューバーが表示されます。
3. 「新しいフォルダー」をクリックします。
   フォルダー名の入力欄が表示されます。
4. フォルダー名を入力し、「作成」をクリックします。
```

#### 注意文の書き方

**Before**:
```
水分が多いと故障につながる危険があるので、水洗いを行わないでください。
```

**After**:
```
全体を水に浸して水洗いしないでください。
内部に水が入り、故障につながります。
汚れは水で濡らして絞った布で拭き取ってください。
```

### 6.2 提案書

#### 全体の設計

1. 読み手と目的を分析（ITに詳しくない経営者か、技術担当者か）
2. ロジックツリーで全体を設計
3. スライド数はポイントごとに決める（目安：5枚前後の会社紹介）

#### スライド作成のポイント

| 要素 | ルール |
|------|--------|
| スライドタイトル | そのスライドのポイントを1行で示す |
| 箇条書き | 7項目以内 |
| 表・グラフ | 数値データで客観性を高める |
| まとめスライド | 「ご質問・ご意見をお聞かせください」など双方向を促す表現 |

**数値データの使い方**: 信頼性の高い情報源（省庁の白書など）を活用する。スポンサー付きデータは客観性に欠ける場合がある。

### 6.3 障害報告書

#### ロジックツリーの構成

```
主題：「○○障害のお詫びと報告」
    ├─ 障害内容（いつ・何が・どのような状態に）
    ├─ 障害原因（なぜ・何が原因で）
    └─ 再発防止策（どう対処したか・今後の取り組み）
```

#### 書き方のポイント

| セクション | ポイント |
|-----------|---------|
| 必須要素 | 発信日・あて先（「御中」）・発信者名・タイトル |
| 主文 | 頭語（拝啓）+ 挨拶文 + 概要（お詫び） |
| 記書き | 障害内容→障害原因→再発防止策の順で整理 |
| 末文 | 再発防止への意志と継続利用のお願い |

**読み手への意識**:
- 情報システム担当者だけでなく、役員・経営者も読む可能性がある
- 専門用語には説明を加える
- 言い訳に終始しない。「○○の確認が不十分でした」と事実を認め、今後の取り組みを示す

**再発防止策の例**:
```
【対応済み】サーバー作業時の確認手順を見直し、システムで確認メッセージを表示するよう対応しました。
【今後の取り組み】作業手順書を更新し、担当者全員に周知徹底します。
```

**チェックリスト**:
- [ ] ビジネス文書のフォーマット（必須4要素）が整っているか
- [ ] 障害内容・原因・再発防止策の3つが見出しで整理されているか
- [ ] 一文一義で書けているか
- [ ] 専門用語に説明を加えているか
- [ ] 言い訳と取れる表現がないか

### 6.4 ビジネスメール（社外）

#### 件名の書き方

| NG | OK |
|----|----|
| スケジュールについて | 【相談】デモシステム納期調整のお願い |
| ご確認ください | ○○提案書のご確認をお願いいたします（期限：11/15） |

- 重要なキーワードを件名の先頭に書く（スマートフォンでは件名が途中までしか表示されない）
- 複数のトピックがある場合は件名にも順番を反映する

#### 本文の構成

```
1. あて先（会社名・役職・氏名）
2. 挨拶（平素よりお世話になっております）
3. 要点（「言いたいことは要するにこういうこと」を最初に）
4. 詳細・理由（トピックごとに空行で区切る）
5. 依頼事項・期限（5W2Hを具体的に）
6. エモーショナルな表現（相手への共感・感謝）
7. 結び
8. 署名（氏名・会社名・部署・電話番号・メールアドレス・住所）
```

#### 伝わるメールの原則

| 観点 | ポイント |
|------|---------|
| 件名 | 目的が一目でわかるキーワードを入れる |
| 冒頭 | 相手の状況への配慮を一言添える |
| 構成 | 起承転結ではなく、結論を最初に |
| 整理 | 空行でトピックを区切る |
| 期限 | 日時を明記（「来週いっぱい」は解釈がぶれる） |
| 締め | エモーショナルな表現で共感を生む |

**エモーショナルな表現の例**:
```
【NG】10日間の短縮は困難ですが、調整した結果、10日前倒しは可能です。

【OK】追加の仕様変更にも対応しながら、チーム全員でスケジュールを見直しました。
      2週間の前倒しは難しい状況ですが、10日間なら対応できます。
      役員会に間に合うよう精一杯取り組みます。
```

## 7. 生成AI活用のライティング

### 7.1 生成AIが得意な文書作成分野

| 用途 | 内容 |
|------|------|
| 文書ドラフト作成 | 手順書・報告書・メールなどのフォーマットに従った文書作成 |
| 要約 | 長文から要点を抜き出してまとめる |
| 言い回しの調整 | 読み手に合わせた表現に変換 |
| 校正 | 誤字・冗長表現・文体の統一チェック |

### 7.2 プロンプトの書き方（テクニカルライティングに応用）

**原則**: 生成AIへの指示（プロンプト）は「漏れなく、詳しく、正確に」書く。

**プロンプトに含める情報**:

```
1. 文書の種類（障害報告書、操作マニュアル、メール文など）
2. 読み手（IT担当者か、役員か、一般ユーザーか）
3. 目的（お詫びと報告、操作方法の説明、スケジュール調整のお願いなど）
4. 素材情報（障害内容・日時・原因・対処内容など）
5. フォーマット（箇条書き・見出し付き、字数制限など）
6. 文体（です・ます調、である調）
```

**プロンプト例（障害報告書の場合）**:
```
次の条件でシステム障害報告書を作成してください。

【読み手】クラウドサービスを利用している企業の情報システム担当者（技術知識あり）
【目的】障害のお詫びと再発防止の報告
【障害内容】20XX年10月25日18:30〜19:15 にログイン後のサービス選択が不可になった
【原因】ホストコンピューター増設作業中に誤ってネットワーク機器のLinkDownが発生。
       二重化システムへの切り替えが正常に動作しなかった
【再発防止策】確認メッセージのシステム化、作業手順書の更新・徹底
【形式】ビジネス文書形式（発信日・あて先・発信者名・タイトル・記書き形式）
【文体】です・ます調
```

### 7.3 AIの出力を見直すポイント

生成AIの出力をそのまま使わず、必ずチェックする。

| チェック観点 | 内容 |
|------------|------|
| 情報の正確性 | 事実と異なる内容が含まれていないか |
| ビジネス文書としての適切さ | 頭語・結語・挨拶文が適切か |
| 表現の自然さ | AI特有の不自然な日本語になっていないか |
| 機密情報の扱い | プロンプトに入力した情報が適切か |
| 知的所有権 | 既存の文章と酷似した内容になっていないか |

**AIで校正する際のプロンプト例**:
```
以下の議事録を校正してください。
チェック観点：
- 誤字・脱字・誤表記
- 表現の統一（送り仮名、カタカナ語）
- 文章の読みやすさ（一文一義、一文50字以内）
- 要素の構造のわかりやすさ（見出し・段落の整理）
```

### 7.4 留意点（AIは主体ではなくツール）

**どんなに技術が進んでも「何を伝えるか」「どのように伝えるか」を考えるのは人間。**

| NG | OK |
|----|----|
| AIに丸投げして出力をそのまま使う | 素材情報を自分で整理してAIに渡し、出力を見直して仕上げる |
| 機密情報をプロンプトに入力する | 匿名化・抽象化した情報を使う |
| AI出力を確認なしで送付する | ビジネス文書としての適切さを必ずチェックする |

**機密情報として送信を避けるべき情報**:
- 未公開の開発技術情報
- 社外秘の組織構造・プロセス情報
- 個人情報（社員・顧客）
- NDA（秘密保持契約）対象の情報

## 8. 文章力チェックリスト

### 8.1 書く前のチェック

- [ ] 読み手（誰が読むか、技術知識レベルは何か）を明確にしたか
- [ ] 目的（読んだ後に何をしてほしいか）を明確にしたか
- [ ] ロジックツリーで情報を整理したか
- [ ] 伝えるキーワードを書き出したか

### 8.2 文章レベルのチェック

- [ ] 一文一義になっているか（複数の内容が混在していないか）
- [ ] 一文は50字以内に収まっているか
- [ ] 「〜が」「〜ので」で文をつないでいないか
- [ ] 話し言葉が混入していないか（「なので」「どうしても」など）
- [ ] 二重否定を使っていないか
- [ ] 「など」で曖昧にしていないか
- [ ] 5W2Hの情報が漏れていないか

### 8.3 文書全体のチェック

- [ ] タイトル・件名から目的が伝わるか
- [ ] 重要な内容が冒頭に書かれているか
- [ ] 見出しで情報がブロック化されているか
- [ ] 必須4要素（発信日・あて先・発信者名・タイトル）がそろっているか
- [ ] 箇条書きには見出しが付いているか
- [ ] 最後の文で適切に締めくくられているか

### 8.4 仕上げのチェック

- [ ] 下書き後に1日置いて見直したか（客観的に読める）
- [ ] 第三者（レビュアー）のチェックを受けたか
- [ ] レビューでポジティブフィードバックと修正指摘の両方をもらったか
- [ ] 生成AIで校正した場合、出力内容を人間が確認したか

### 8.5 自己評価の活用（強み・弱みの確認）

| 弱みのカテゴリ | 該当章 |
|--------------|--------|
| 書き始める前の準備が不足 | 第1章：読み手と目的の分析 |
| トピックのまとめ方が弱い | 第2章：わかりやすい文章の基本技術 |
| 文章表現と仕上げの工夫が足りない | 第3章：伝わる表現技術 |
| 文書フォーマット・表記の基本が弱い | 第4章：読みやすさを高める工夫 |
| 「200字トレーニング」を継続的に実践する | 週1テーマで4〜5文（200字前後）を書き上げる |
