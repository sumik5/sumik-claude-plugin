# 論理構成リファレンス

文書全体の骨格設計から段落レベルの情報配置まで、論理的な文章を構築するための技術を体系化する。

---

## 1. 文書全体の3部構成

### 基本構造

```
┌─────────────────────────────────────────────┐
│  総論（序・概要）                             │
│  ・目的パラグラフ                             │
│  ・要約パラグラフ（7文前後 ≈ 30秒で読める量） │
├─────────────────────────────────────────────┤
│  各論                                        │
│  ・総論で述べた重要情報をパラグラフで展開      │
│  ・総論で述べた順序を守る                     │
├─────────────────────────────────────────────┤
│  結論（まとめ）                               │
│  ・総論の繰り返し                             │
│  ・短い文章では省略可                         │
└─────────────────────────────────────────────┘
```

### 設計原則

**メンタルモデルとの整合性**
- 読み手は総論を読んだ時点で「この文書はA→B→Cの順に展開される」という予測モデルを形成する
- 各論の展開順が総論と一致することで、読み手は情報を処理しやすくなる
- 順序を変えると読み手の予測が裏切られ、理解コストが増大する

**総論の要約パラグラフ（7文前後）の役割**
- 30秒で全体の要点を把握できる量が目安
- 忙しい読み手が「続きを読む価値があるか」判断できる
- 後続の各論で何が展開されるかを先に示す「予告編」として機能する

## 2. パラグラフ・ライティング

### パラグラフの定義

> **1つのトピックについて述べる文の集まり（4〜8文が目安）**

### 基本ルール

| ルール | 説明 |
|--------|------|
| 1パラグラフ = 1トピック | 複数のトピックを1つにまとめない。1つのトピックを複数に分割しない |
| 冒頭に要約文を書く | そのパラグラフで伝えたいことを簡素にまとめた1文を最初に置く |

### 要約文の5つの注意点

1. **前パラグラフとつながる** — 唐突な転換は読み手を混乱させる
2. **1トピック1文** — 要約文に複数の話題を詰め込まない
3. **具体的に書く** — 「さまざまな」「多くの」などの曖昧表現を避ける
4. **簡潔に書く** — 補足情報は後続文に回す
5. **文頭にキーワードを置く** — 読み手が最初の数文字でトピックを把握できる

### 正しく書けているかの確認方法

> 各パラグラフの**先頭文だけを読んで意味が通れば**、正しく構造化されている

## 2-A. パラグラフ・ライティング 10のルール

体系的なパラグラフ構築のための詳細ルール。上記「基本ルール」を補完する。

| # | ルール | 内容 |
|---|-------|------|
| 1 | 1パラグラフ = 1トピック | パラグラフの中のすべての文が、1つのトピックに属すること |
| 2 | キーセンテンスを冒頭に置く | そのパラグラフを要約する文をパラグラフ先頭に配置する |
| 3 | サブセンテンスはキーセンテンスを補足する | キーセンテンス以降の文は、補足・具体例・根拠・データのみ。脱線を禁じる |
| 4 | パラグラフ間を論理的につなぐ | 前パラグラフの結論が次パラグラフの前提になるよう展開する |
| 5 | 階層型接続で展開する | 上位パラグラフから下位（詳細）へ向かって展開する |
| 6 | キーパラグラフを文章冒頭に置く | 文章全体を要約するパラグラフ（キーパラグラフ）を先頭に配置する |
| 7 | キーパラグラフと各パラグラフを対応させる | キーパラグラフの各文が、各パラグラフのキーセンテンスに対応していること |
| 8 | 新情報は文末に置く | 文の前半に既知情報、文末に新情報を配置する（旧情報→新情報の流れ） |
| 9 | テーマの一貫性を保つ | パラグラフ内で主語が不規則に変わらないようにする |
| 10 | パラグラフの長さを均一に近づける | 極端に短いパラグラフや長いパラグラフを避ける |

**要約文テスト**: 各パラグラフのキーセンテンスだけを連結して読み、論旨が一貫しているか確認する。キーセンテンス同士をつなぐことで、文章全体の骨格が浮かび上がるようになっていれば合格。

## 2-B. 接続詞の4分類

接続詞は「次の文がどんな論理関係で来るか」を読み手に予告するシグナル。適切に使うことで読み手の認知負荷が下がる。

| グループ | 役割 | 主な接続詞 |
|---------|------|-----------|
| **順接** | 因果関係・結論を示す | だから、したがって、そのため、それゆえ、よって |
| **付加** | 情報の追加・補強を示す | また、さらに、加えて、そして、その上 |
| **逆接** | 反論・転換・対比を示す | しかし、ところが、だが、けれども、一方、それでも |
| **補足・換言・例示** | 補足説明・言い換え・例示を示す | ただし、なお、すなわち、つまり、要するに、たとえば |

**接続詞の使用原則**:
- 論理関係が不明瞭になりそうな箇所では接続詞で明示する
- 「まず」「次に」「最後に」を機械的に並べない（単調になり、AI臭の原因にもなる）
- 文脈から論理関係が明らかな場合は省略してもよい
- 同じ接続詞を直近の文で連続使用しない

## 2-C. 例の4類型

説明に例を使うとき、目的に応じて例の種類を選ぶ。1つの概念に対して複数の類型を組み合わせると理解が深まる。

| 例の種類 | 目的 | 使うタイミング |
|---------|------|--------------|
| **典型的な例** | 一般的なイメージを持たせる | 概念の初出時・定義を補足するとき |
| **極端な例** | 概念の適用限界・境界を明示する | 定義の外縁を示したいとき |
| **あてはまらない例（反例）** | 誤解・過剰適用を防ぐ | 「これは当てはまらない」を示すとき |
| **一般的な例** | 読者の既知経験と結びつける | 抽象的な概念を身近に感じさせるとき |

**活用パターン**: 典型例で概念を導入 → 反例で輪郭を明確化 → 極端な例で境界を示す、という順で使うと概念の理解が段階的に深まる。

## 3. パラグラフの縦横接続

### 縦つながり（論理的接続）

パラグラフ間に論理的な因果・時系列関係がある場合。**順序は固定**する。

```
問題 → 原因 → 対策 → 効果
```

- 「現状の課題」→「その原因」→「解決策」→「期待される成果」といった展開
- 順序を変えると論理が破綻する

### 横並び（同種情報の並置）

同じ上位カテゴリに属するが、パラグラフ間に論理的接続がない場合。**重要な順**で並べる。

```
原因A / 原因B / 原因C
```

- 「売上低下の原因①」「売上低下の原因②」「売上低下の原因③」のような展開
- 読み手にとって最も重要・関心の高いものから先に置く
- 「思いついた順」や「調べた順」では書かない

## 4. 2つの論理パターン

### 選択の判断基準

```
問いを立てる
    ↓
「どうなっているか？」→ 並列型
「どうすべきか？」「どれがよいか？」→ 解説型
```

### 並列型

| 項目 | 内容 |
|------|------|
| 使用場面 | 状況説明・現状報告 |
| 構造 | 結論 + 複数の根拠（横にMECE） |
| 特徴 | 書き手の判断を前面に出さない |

```
結論（エッセンス）
  └─ 根拠A
  └─ 根拠B
  └─ 根拠C
```

### 解説型

| 項目 | 内容 |
|------|------|
| 使用場面 | 判断・推奨を伝えたい時 |
| 構造 | 事実 + 判断基準 + 判断内容 |
| 特徴 | 書き手の意見・推奨を明示する |

```
事実（観察した情報）
  ↓
判断基準（どんな基準で評価するか）
  ↓
判断内容（書き手の結論・推奨）
```

## 5. MECE（Mutually Exclusive and Collectively Exhaustive）

### 概念

> **重複なく・漏れなく・ずれなく分ける**考え方

- **ME（Mutually Exclusive）**: 各要素が重複しない
- **CE（Collectively Exhaustive）**: 全体を網羅している

### 3つの分解タイプ

| タイプ | 説明 | 例 |
|--------|------|-----|
| **要素分解** | 全体を構成要素に分解 | 3C（Customer / Competitor / Company） |
| **ステップ分け** | 時系列・プロセスで分解 | 購買前 / 購買中 / 購買後 |
| **対照概念** | 二項対立で分解 | Will / Skill、OJT / OFF-JT |

### 重要な注意点

> 既存のMECEフレームワーク（3C, 4P, SWOTなど）を知っていても、**手元の情報に本当に当てはまるか必ず検証する**

フレームワークに情報を無理やり当てはめると、実態にそぐわない分析になる。

## 6. So What? / Why So?

### 2方向の思考操作

```
複数の情報
    ↓  So What?（束ねて引き上げる）
エッセンス（要するにどういうことか）
    ↓  Why So?（支持できるか検証する）
複数の情報（根拠として成立するか）
```

### So What? の使い方

- 複数のデータや事実を眺めて「**要するにこれは何を意味するか**」を引き出す
- 「○○だった、□□だった、△△だった」だけで終わらず、1段上の意味を取り出す

### Why So? の使い方

- So What? で引き出したエッセンスを見て、「**なぜそう言えるか？**」と問い返す
- 下位の情報が本当にそのエッセンスを支持しているか検証する
- 支持できなければエッセンスを修正するか、根拠を見直す

### So What? 放棄パターン（避けるべき表現）

| パターン | 例 |
|---------|-----|
| 「以下のとおり」で逃げる | 「結果は以下のとおりです。」→ 箇条書きのみ |
| 「さまざまな」で逃げる | 「さまざまな要因がある」→ 具体化しない |
| 「次のとおり」で逃げる | 「詳細は次のとおりです。」→ 情報を並べるだけ |

### 「2段目不在」パターン

```
❌ 不完全な構造
結論（1段目）
  └─ 根拠A（3段目・箇条書き）
  └─ 根拠B
  └─ 根拠C

✅ 正しい構造
結論（1段目）
  └─ 中位の根拠X（2段目）
      └─ 根拠A（3段目）
      └─ 根拠B
  └─ 中位の根拠Y（2段目）
      └─ 根拠C
```

結論と箇条書きしかなく、**中位の根拠（2段目）が欠落している**構造は論理が弱い。

## 7. 既知→未知の情報の流れ

### 原則

> 文の**前半で前述の既知情報を引き継ぎ**、**後半で初出の未知情報を付加する**

読み手は「知っている情報」を手がかりに「知らない情報」を理解する。
この流れを逆にすると、読み手は脈絡なく新情報を受け取ることになり、理解コストが増大する。

### 3つのパターン

#### 統一型（キーワード強調）

文頭の主語を同じキーワードで統一する。特定の主体に注目させたい時に有効。

```
サービスXはAという特徴を持つ。
サービスXはBという利点がある。
サービスXはCにも対応する。
```

#### 引継型（連鎖接続）

前文の後半情報を次文の文頭に使う。A→B, B→C, C→Dと連鎖させる。

```
AはBを促進する。
Bが高まるとCが向上する。
Cの向上によりDが実現する。
```

#### 展開型（先行列挙→後続展開）

最初の文で情報を列挙し、後続の文が先の情報を文頭で参照する。

```
この方法には3つの特徴がある：A、B、Cだ。
Aは〜という点で優れる。
Bは〜を可能にする。
Cは〜に対応できる。
```

### 実践上の注意点

- **要約文だけでも既知→未知の流れを守る**
- **くどくなっても既知情報を文頭に書く** — 省略すると読み手が前後関係を推測しなければならなくなる

## 8. パラレリズム（並列情報の構成・表現統一）

### 基本ルール

> 並列する情報は**構成も表現もそろえる**

コピー&ペーストで「情報だけを差し替えるイメージ」で書くと自然にそろう。

### 4つの統一要素

| 要素 | 内容 |
|------|------|
| **表現形式** | 名詞句/動詞句/文をそろえる |
| **抽象度** | 上位概念と下位概念を混在させない |
| **文体** | ですます調/である調を混在させない |
| **粒度** | 情報の詳しさをそろえる |

### 並べる順番のルール

| 優先順 | 並べ方 |
|--------|--------|
| 1位 | **重要な順**（最も伝えたいものを先頭に） |
| 2位 | **時間順**（出来事を時系列で並べる場合） |
| 禁止 | ~~思いついた順~~ |

### よくあるミス

```
❌ 抽象度がバラバラ
・Pythonを使った開発
・テスト自動化  ← 上と同じ抽象度?
・CI/CDパイプラインの構築
・boto3（AWSライブラリ）  ← 急に具体的

✅ 抽象度をそろえる
・フロントエンド実装（React）
・バックエンドAPI構築（FastAPI）
・データベース設計（PostgreSQL）
・インフラ構築（AWS）
```

## 9. 導入部の設計

### 避けるべき3タイプ

| タイプ | 説明 |
|--------|------|
| **埋没型** | 当たり前のことを書いて始まる。本題に入るまで時間がかかる |
| **表題繰り返し型** | タイトルをそのまま文章にしただけ。情報が増えない |
| **ご挨拶に終始型** | 「お世話になっております」「〜について述べます」など。読み手への価値がない |

### 導入部に必要な5つの視点

読み手が「なぜこれを読む必要があるか」を理解できるよう、以下を意識する。

| # | 読み手の問い | 導入部で答えること |
|---|------------|-----------------|
| 1 | なぜこのテーマか？ | **背景** — テーマを取り上げる理由・文脈 |
| 2 | なぜこの反応をとるべきか？ | **メリット** — 読むことで得られる価値・利益 |
| 3 | なぜこの書き手か？ | **自己紹介** — 書き手の立場・資格・経験 |
| 4 | なぜこの読み手か？ | **読む理由** — この読み手が対象である根拠 |
| 5 | あらかじめ把握すべきことは？ | **情報源・位置づけ** — 前提知識・文書の位置づけ |

すべてを含める必要はない。文書の長さと目的に応じて取捨選択する。

## 10. 結論先出し vs 根拠先出し

### デフォルト：結論先出し

ほとんどの場面では、最初に結論・主張を述べてから根拠を示す。

```
【結論】AよりBを推奨します。
【根拠1】〜
【根拠2】〜
【根拠3】〜
```

読み手は「何が言いたいか」を最初に把握できるため、根拠を適切な文脈で受け取れる。

### 根拠先出しにすべき3つの場面

| 場面 | 理由 |
|------|------|
| **書き手が自らテーマ設定した場合** | 「なぜこのテーマを取り上げるか」を先に示す必要がある |
| **読み手の反発が予想される場合** | いきなり結論を出すと拒絶される。根拠で共感を積み上げてから結論を出す |
| **読み手にSo What?を体験させたい場合** | 意図的に「結論は何か」を考えさせることで、理解を深める |
