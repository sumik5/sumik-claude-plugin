# AI臭6パターン 詳細検出ルール

## ①視覚的な記号の残留

Markdown記法やテキスト装飾が本文に混入している状態。LLMの訓練データに含まれる構造化文書の影響。

### 検出ポイント

- `—`（emダッシュ）で言い換えや補足を接続
- 「」の多用、『』のネスト
- `：`の後に半角スペースが入る
- `（）`で補足を逃がしすぎ
- `/` で並列表現
- `→` で流れを示す

### 具体例

#### emダッシュの多用

```markdown
❌ AI臭い
この手法は効率的です—ただし、メモリ使用量が増える可能性があります—そのため、本番環境では注意が必要です。

✅ 人間っぽい
この手法は効率的だが、メモリ使用量が増える。本番環境では注意してほしい。
```

#### 「」『』の過剰使用

```markdown
❌ AI臭い
「実装」においては「品質」を担保し、『SOLID原則』に基づいた『クリーンなコード』を書くことが重要です。

✅ 人間っぽい
実装では品質を担保し、SOLID原則に基づいたクリーンなコードを書くことが重要だ。
```

#### （）での責任回避

```markdown
❌ AI臭い
この方法を推奨します（ただし、プロジェクトの状況によります）。

✅ 人間っぽい
この方法を推奨する。プロジェクトの状況に応じて調整してほしい。
```

#### ／での並列

```markdown
❌ AI臭い
フロントエンド／バックエンドの両方で型安全性を確保する必要があります。

✅ 人間っぽい
フロントエンドとバックエンドの両方で型安全性を確保する必要がある。
```

#### →での流れ表現

```markdown
❌ AI臭い
リクエスト受信 → バリデーション → ビジネスロジック → レスポンス返却

✅ 人間っぽい
リクエストを受信してバリデーションを実行し、ビジネスロジックを処理してからレスポンスを返す。
```

## ②文のリズムが単調

同じ構造の文が連続し、機械的なリズムが生まれている状態。

### 検出ポイント

- 同じ語尾が3回以上連続（〜です。〜です。〜です。）
- 接続詞が過多（さらに、また、したがって、そのため）
- 文の温度が一定（重要な箇所も補足も同じテンション）
- 段落末が毎回きれいに閉じる
- not A but B構文の多用

### 具体例

#### 同じ語尾の連続

```markdown
❌ AI臭い
この実装は効率的です。パフォーマンスが向上します。メンテナンス性も高いです。テストも書きやすいです。

✅ 人間っぽい
この実装は効率的だ。パフォーマンスが向上し、メンテナンス性も高い。テストも書きやすい。
```

#### 接続詞過多

```markdown
❌ AI臭い
まず、バリデーションを実行します。次に、データを変換します。その後、データベースに保存します。さらに、キャッシュを更新します。最後に、レスポンスを返します。

✅ 人間っぽい
バリデーションを実行し、データを変換してからデータベースに保存する。キャッシュを更新してレスポンスを返す。
```

#### 温度が一定

```markdown
❌ AI臭い
型安全性は重要です。テストも重要です。ドキュメントも重要です。コードレビューも重要です。

✅ 人間っぽい
型安全性は絶対に必要だ。テストとドキュメントも書く。コードレビューは丁寧にやる。
```

#### not A but B構文

```markdown
❌ AI臭い
重要なのはコードの量ではなく、品質です。目指すべきは完璧な設計ではなく、実用的な設計です。

✅ 人間っぽい
コードの量より品質が重要だ。完璧な設計より実用的な設計を目指す。
```

## ③導入と進行が説明書っぽい

前置きが長く、構造を何度も宣言する状態。説明書やチュートリアルの文体。

### 検出ポイント

- 前置きが長い（ご質問ありがとうございます...）
- 「結論から言うと」で始めて結論が薄い
- 構造を本文で何度も宣言する（以下の3つを解説します...）
- ステップ/STEP/Stepの多用

### 具体例

#### 前置きが長い

```markdown
❌ AI臭い
ご質問ありがとうございます。それでは、TypeScriptにおける型安全性について詳しく解説させていただきます。まず、基本的な概念から説明し、その後、実践的なテクニックをご紹介いたします。

✅ 人間っぽい
TypeScriptの型安全性について解説する。
```

#### 「結論から言うと」

```markdown
❌ AI臭い
結論から言うと、この実装は推奨されません。理由は以下の通りです...

✅ 人間っぽい
この実装は推奨しない。理由は...
```

#### 構造の過剰宣言

```markdown
❌ AI臭い
以下の3つのポイントについて解説します:
1. 型安全性
2. テスト
3. ドキュメント

それでは、1つ目の型安全性から見ていきましょう。

✅ 人間っぽい
型安全性、テスト、ドキュメントの順で見ていく。
```

#### STEP多用

```markdown
❌ AI臭い
STEP1: プロジェクトを初期化する
STEP2: 依存関係をインストールする
STEP3: 設定ファイルを作成する

✅ 人間っぽい
プロジェクトを初期化し、依存関係をインストールしてから設定ファイルを作成する。
```

## ④事なかれ主義なスタンス

常に逃げ道を用意し、中立を装いすぎている状態。責任を負わない文体。

### 検出ポイント

- 保険・逃げ道の常設（一概には言えませんが...）
- 中立を装いすぎる（どちらにもメリットがあります）
- 否定が弱い（あまり推奨されません...）
- 「場合によっては」「状況次第で」の多用

### 具体例

#### 保険表現

```markdown
❌ AI臭い
一概には言えませんが、一般的にはこの方法が推奨されることが多いです。ただし、プロジェクトの状況によっては別のアプローチも検討する価値があるかもしれません。

✅ 人間っぽい
この方法を推奨する。プロジェクトの規模が小さい場合は別のアプローチも検討してほしい。
```

#### 中立の装い

```markdown
❌ AI臭い
アプローチAとアプローチBにはそれぞれメリット・デメリットがあり、どちらが優れているとは一概に言えません。状況に応じて適切に選択することが重要です。

✅ 人間っぽい
アプローチAの方が優れている。アプローチBはメンテナンスコストが高い。
```

#### 弱い否定

```markdown
❌ AI臭い
この実装はあまり推奨されない傾向にあります。

✅ 人間っぽい
この実装はやめてほしい。
```

## ⑤抽象語と万能語への偏重

具体性のない万能抽象語で押し切る状態。動詞が少なく、名詞と形容詞が多い。

### 検出ポイント

- 抽象語だけで押し切る（本質、最適化、価値最大化）
- 評価語が強いのに根拠がない（非常に有効です）
- 同義語の言い換え連打（重要、肝心、鍵となる、核となる）

### 具体例

#### 抽象語の連打

```markdown
❌ AI臭い
本質的な価値を最大化し、全体最適化を図ることで、持続可能な成長を実現します。

✅ 人間っぽい
ユーザーが求める機能を優先的に開発し、システム全体の応答速度を改善する。これによって継続的な成長が可能になる。
```

#### 根拠のない評価語

```markdown
❌ AI臭い
この手法は非常に効果的であり、極めて有用です。

✅ 人間っぽい
この手法でレスポンスタイムが30%改善した。
```

#### 同義語の言い換え連打

```markdown
❌ AI臭い
型安全性は重要であり、肝心であり、鍵となる要素であり、核となる概念です。

✅ 人間っぽい
型安全性は必須だ。
```

## ⑥変な比喩の多用

テンプレート化された比喩表現の濫用。実体験や観察から生まれた比喩ではない。

### 検出ポイント

- 羅針盤
- 土台
- 柱
- DNA
- 車の両輪
- 潤滑油
- エンジン
- スパイス
- レシピ

### 具体例

#### 羅針盤

```markdown
❌ AI臭い
この設計原則は、プロジェクトの羅針盤となります。

✅ 人間っぽい
この設計原則に従ってアーキテクチャを決める。
```

#### 土台・柱

```markdown
❌ AI臭い
型安全性はシステムの土台であり、テストは柱です。

✅ 人間っぽい
型安全性がなければシステムは壊れる。テストがなければ保守できない。
```

#### 車の両輪

```markdown
❌ AI臭い
パフォーマンスとメンテナンス性は車の両輪です。

✅ 人間っぽい
パフォーマンスとメンテナンス性の両方が必要だ。
```

#### エンジン・スパイス

```markdown
❌ AI臭い
この機能はシステムのエンジンであり、ユーザー体験にスパイスを加えます。

✅ 人間っぽい
この機能がシステムの中核を担い、ユーザー体験を改善する。
```

## 補足: 複合パターン

実際のAI生成文章では、複数のパターンが同時に現れることが多い。

### 複合例

```markdown
❌ AI臭い（①②③④⑤を含む）
**重要**: 型安全性は本質的な価値を持ちます。一概には言えませんが、一般的には推奨されます。以下の3つのポイントを解説します:
1. 基本概念
2. 実践テクニック
3. ベストプラクティス

✅ 人間っぽい
型安全性は必須だ。基本概念、実践テクニック、ベストプラクティスの順で解説する。
```
