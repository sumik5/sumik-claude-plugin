# FAQ・ヘルプドキュメント向けライティング リファレンス

## 1. 良いFAQの3条件

FAQでユーザーが問題を解決できている割合は3人に1人以下というのが実態。残り3分の2が解決できない原因の多くは、FAQの文の質にある。

| # | 条件 | 説明 |
|---|------|------|
| 1 | ユーザーが読んで理解できる | 専門用語を避け、平易な表現。最後まで読んでもらえる文章 |
| 2 | ユーザーに信頼される | 正確で最新の情報。安心感を与える文章 |
| 3 | ユーザーが迷わず取捨選択できる | 質問タイトル（Q）だけで内容の判断が可能 |

**質の低いFAQの典型例:**

```
Q: ログインできない。
A: パスワードを忘れた人はこちらでお手続きください。
```

- Q が問題だけを述べており、解決の方向性がない
- A が問いに正面から答えていない
- ユーザーが「自分の状況に該当するか」を判断できない

## 2. Q（質問）の書き方

### 基本文体: 「問題 + 解決案」型

QはFAQ一覧でユーザーが取捨選択する唯一の手がかりである。解決への道筋を示すことが最重要。

| パターン | 例 | 評価 |
|---------|-----|------|
| 問題のみ | 「ログインできない」 | ❌ 解決の方向性がない |
| 問題 + 解決案 | 「パスワードを忘れてログインできません。再設定するには？」 | ✅ 解決案が示されている |

### Qの必須要素

1. **問題を明記する**: ユーザーが抱えている状況・困りごとを言語化する
2. **解決案（方向性）を示す**: 「〜するには？」「〜を確認するには？」など
3. **ユーザーの言葉で書く**: システム用語・社内用語ではなく、ユーザーが実際に使う言葉で

### Qを書く際の禁止事項

**Yes/No質問にしない:**
- ❌「○○は可能ですか？」
- ✅「○○を行う手順は？」または「○○ができない場合の対処法は？」

ネガティブな回答（「できません」）の場合は、代替手段・理由・問い合わせ先をAに明示する。

**誤解の余地を作らない:**
- 主語・対象・条件を明確にする
- 「かかり受け」に気をつける（修飾語が何にかかるか）
- 一意の文（複数の解釈ができない文）にする

**読む人の立場に配慮する:**
- 失敗・困惑しているユーザーを責めるような表現を避ける
- 「なぜ〜しないのですか？」より「〜できない場合は？」

### FAQ一覧での視認性

Q一覧はユーザーが自分の問題と照合しながらスキャンする場面で使われる。

- **言葉の統一**: 同じ操作・概念には同じ言葉を使い続ける
- **パターン化**: 構文を統一する（例: 「〜する方法は？」で統一）
- **粒度をそろえる**: 大きなテーマと細かいテーマを同列に並べない
- **インデックス化**: カテゴリで分けて、ユーザーが見つけやすくする

## 3. A（回答）の書き方

### 基本構成

```
1. 回答の要約（冒頭1〜2文）
2. 手順・詳細説明
3. 注意事項・補足
4. 関連FAQへのリンク
```

**結論を最初に示す（逆ピラミッド構造）:**
- ❌「〇〇の場合は△△で、□□の場合は〜〜です。なお、××の場合は……」（条件から始まる）
- ✅「□□の手順で対応できます。なお、○○の場合は△△が必要です。」（結論から始まる）

### Aの必須要素

- **基本的な作文の正確さ**: 誤字脱字・事実誤認がない
- **視認性と読みやすさ**: 箇条書き・番号付きリストを積極的に活用する
- **信頼感の醸成**: 曖昧な表現を避け、ユーザーが安心して行動できる情報を提供する

### 箇条書きに変換するテクニック

文章で書かれた手順・条件は箇条書きに変換すると読みやすくなる。

**変換前（読みにくい）:**
```
まずアカウント設定を開き、次にプロフィールタブを選択してから、
名前の欄をクリックして編集し、最後に保存ボタンを押してください。
```

**変換後（読みやすい）:**
```
1. アカウント設定を開く
2. プロフィールタブを選択する
3. 名前の欄をクリックして編集する
4. 保存ボタンを押す
```

### 装飾・記号の使い方

- 強調は重要な単語・フレーズのみに絞る
- 装飾が多すぎるとかえって読みにくくなる
- 記号（！、※、→など）はルールを決めて統一して使う
- リンクは「こちら」ではなく、リンク先の内容がわかる文言にする（例: 「パスワード再設定ページ」）

## 4. 一問一答の原則

**1つの質問に1つの回答のみ。** これがFAQの最重要ルール。

### やってはいけないパターン

**複数の問題を1つのFAQに詰め込む:**
```
Q: ログインできない、またはパスワードを忘れた場合はどうすればいいですか？
```
→ 2つの異なる問題を1つのQにまとめている。それぞれ別のFAQにする。

**A内で条件分岐をする:**
```
A: 対応方法は登録方法によって異なります。
   Aパターンの方は〜、Bパターンの方は〜、Cパターンの方は〜
```
→ ユーザーは自分がどのパターンか判断する必要がある。条件が異なる場合は別のFAQに分ける。

**情報を盛り込みすぎる:**
- FAQの目的は「特定の問題を解決すること」
- 関連情報は「関連FAQ」リンクで補う
- AとQが一致していること（Qで問ったことだけをAで答える）

## 5. 6W1H原則

QとA両方で、必要な要素が欠けていないかを確認する。

| 要素 | Q側 | A側 |
|------|-----|-----|
| When | いつ発生する問題か | いつ対応すればよいか・いつ有効になるか |
| Who | 誰（どのユーザー）が対象か | 誰が対応するか・誰に連絡するか |
| Where | どこで発生するか（画面・機能名） | どこで操作するか・どこを確認するか |
| Why | なぜ起きるか | なぜこの手順が必要か |
| What | 何が問題か | 何をすればよいか |
| Whom | 誰に影響するか | 誰に連絡するか（問い合わせ先） |
| How | どのように発生するか | どのように解決するか（具体的手順） |

**活用方法:**

回答を書き終えたら、6W1Hの各要素でユーザーが「疑問に思いそうなこと」が抜けていないか確認する。すべての要素を盛り込む必要はなく、対象の問題に必要な要素だけを選ぶ。

## 6. カテゴリ設計

### ユーザーの行動ベースで分類する

内部の管理単位（部門・機能・システム）ではなく、**ユーザーが達成しようとしていること**で分類する。

**内部管理ベース（やってはいけない）:**
```
- 会員登録
- 購入機能
- 決済システム
- カスタマーサポート
```

**ユーザーの行動ベース（推奨）:**
```
- はじめて使う方へ
- 注文・購入
- お支払い
- 変更・キャンセル
- 解約
```

### カテゴリ数の目安

5〜7カテゴリが適切。多すぎると選べなくなる。

- 10カテゴリ以上: ユーザーが迷う
- 3カテゴリ以下: 1カテゴリに詰め込みすぎてFAQが見つからない
- 5〜7カテゴリ: ユーザーが目的のカテゴリを素早く見つけられる

### FAQの選抜基準

すべての質問をFAQに掲載する必要はない。以下の基準で選抜する:

1. **問い合わせ数が多いもの**: コールセンター・メール問い合わせの頻出テーマ
2. **ユーザーターゲットに合致するもの**: 主要ユーザー層が抱える問題
3. **パレートの法則**: 問い合わせの80%は20%のFAQで解決できる

**掲載不要な例:**
- 1件しか来ない極めてニッチな質問
- 回答がユーザーごとに異なり、FAQで解決できないもの
- 公開すべきでない機密情報に関するもの

## 7. 用語・言葉遣いの統一

一貫した用語を使うことで、ユーザーの混乱を防ぎ、FAQシステムの検索精度も上がる。

### ガイドラインに定めるべき事項

- 採用する略語・略称の一覧
- 新たに略語を採用する場合の基準
- 漢字・ひらがな・カタカナの使い分けルール
- 全角・半角の使い分けルール
- 敬体（です・ます）か常体（だ・である）かの選択

### 用語統一の実践例

| 統一前（NG） | 統一後（OK） |
|------------|------------|
| マイページ / アカウントページ / マイアカウント | マイページ（1語に統一） |
| ログイン / サインイン | ログイン（1語に統一） |
| 解約 / 退会 / 利用停止 | 解約（公式サービス用語に合わせる） |

## 8. FAQのメンテナンス

FAQは公開して終わりではなく、継続的に育てるコンテンツ。

### 定期的なメンテナンス

- アクセスログを分析し、閲覧数が少ないFAQを見直す
- 閲覧後の行動を追跡する（問い合わせが減ったかどうか）
- 問い合わせ履歴から新しい質問を発見し、FAQに追加する
- 製品・サービスの変更に合わせて内容を更新・廃止する

### KPIの設定

FAQの品質を数値で評価する。

| KPI | 意味 | 目標の考え方 |
|-----|------|------------|
| 回答到達率 | FAQを検索したユーザーのうち、回答ページに到達した割合 | 高いほど検索性が良い |
| 問題解決率 | FAQを読んだユーザーのうち、問題が解決した割合 | 高いほどFAQの質が高い |
| FAQからの問い合わせ率 | FAQを読んだ後に問い合わせに移行した割合 | 低いほど自己解決できている |

### FAQを「育てる」意識

- FAQは商品・サービスと同様に品質責任を持って管理するコンテンツ
- FAQ制作の専任者と品質責任者を任命する
- ガイドラインを制定し、関係者全員で品質基準を共有する
- 「ユーザーへの思いやり」を持って書き続ける

## 9. FAQライティング チェックリスト

### Q（質問）のチェック

- [ ] 「問題 + 解決案」の文体になっているか
- [ ] Yes/No質問になっていないか
- [ ] ユーザーが自分の状況に該当するかを判断できるか
- [ ] 一意の文（複数解釈できない）になっているか
- [ ] ユーザーが使う言葉で書かれているか
- [ ] 一覧の他のQと粒度がそろっているか

### A（回答）のチェック

- [ ] 結論が冒頭に来ているか
- [ ] 一問一答（Q一つにA一つ）になっているか
- [ ] A内で条件分岐（「〜の場合は…もし〜なら…」）をしていないか
- [ ] 情報を盛り込みすぎていないか（QとAが一致しているか）
- [ ] 手順は番号付きリスト・箇条書きになっているか
- [ ] 6W1Hの必要な要素が揃っているか
- [ ] リンクが「こちら」ではなく内容を示す文言になっているか
