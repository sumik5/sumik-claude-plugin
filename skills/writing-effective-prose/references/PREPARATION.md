# 書く前の思考設計

文章の品質は、書き始める前の「設計」で9割決まる。
このドキュメントは執筆前に必ず確認すべき思考フレームワークをまとめたリファレンスである。

---

## 1. 文章の7つの要件

書く前に明確にすべき7つの要件。これが揃っていない文章は、どれだけ推敲しても「何が言いたいのかわからない文章」になる。

### 要件一覧

| 要件 | 問い | 明確化の方法 |
|------|------|------------|
| **① 意見** | 一番言いたいことは何か？ | 「問い→答え」の構造を作る |
| **② 望む結果** | この文章で何を達成したいか？ | 「読んだ人にどう言ってほしいか？」でイメージする |
| **③ 論点** | 文章を貫く「問い」は何か？ | 疑問文で表現する（テーマ≠論点） |
| **④ 読み手** | 誰が読むのか？ | 具体的な人物像を描く |
| **⑤ 自分の立場** | 相手から見た自分の位置は？ | 相手視点で距離・関係性を確認する |
| **⑥ 論拠** | 意見を支える根拠は何か？ | 4ステップで論拠を準備する（→ 後述） |
| **⑦ 根本思想** | 自分の生き方・価値観は何か？ | ひと言要約で発見する |

### 各要件の詳細

#### ① 意見
- 意見は「問いに対する答え」として生まれる
- 問いを立てていないから意見が出ない、という場合がほとんど
- 「書きたいこと」とは異なる。「最終的に主張すること」が意見

#### ② 望む結果
- 曖昧な状態で書き始めるのが最大の失敗原因
- 「理解してほしい」「行動してほしい」「承認してほしい」の3パターンで確認する
- 具体的な状態変化をイメージする。「なんとなくわかってほしい」は不可

#### ③ 論点
- **テーマ ≠ 論点**：テーマは「扱う対象」、論点は「文章を貫く問い」
  - テーマ例：「リモートワーク」
  - 論点例：「リモートワーク導入で生産性は本当に上がるのか？」
- 論点は必ず疑問文にする
- 論点と意見（①）は呼応する関係：論点が問いで、意見がその答え

#### ④ 読み手
詳細は「読み手分析」セクション（→ 後述）

#### ⑤ 自分の立場
- 「こっちのつもり」ではなく、相手視点で確認する
- 上司として書くのか？専門家として書くのか？同僚として書くのか？
- 立場が違うと使える言葉・論拠・前提知識が変わる

#### ⑥ 論拠
詳細は「論拠の準備」セクション（→ 後述）

#### ⑦ 根本思想
- 短い文章にもごまかしようなく表れる
- ひと言で「自分はどういう人間か」を要約する
- 根本思想が定まっていないと、文章全体にブレが出る

## 2. テーマを「問い」に変換する

### アクション説明文書の3種類の問い

文書の目的によって立てるべき問いが異なる：

| 問いの種類 | 具体例 | 使う場面 |
|-----------|--------|---------|
| **現状はどうか？** | 「現在の課題の全体像は？」 | 現状分析・報告 |
| **課題は何か？** | 「なぜこの問題が起きているのか？」 | 原因究明・分析 |
| **アクションは何か？** | 「どうすれば改善できるか？」 | 提案・計画 |

### 問いを立てるエリアの拡張

行き詰まったとき、問いを立てるエリアを以下のマトリクスで広げる：

```
時間軸
  ↑
未来  ──┼──────────────────────────
現在  ──┼──────────────────────────
過去  ──┼──────────────────────────
      自分  身の回り  社会  世界  → 空間軸
```

**「遠回りの法則」**：行き詰まったら視野を逆方向に広げる。
- 現在→過去から学ぶ
- 自分→社会・世界の動向を見る
- 具体→抽象の概念を確認する

## 3. 読み手分析

### 読み手について意識すべき3点

1. **どんな人物か**：年齢、職業、専門知識レベル、関心・価値観
2. **どんな状況で読むか**：読む時間、読む場所、読む前後の行動
3. **何を求めているか**：課題意識、期待する情報、使いたい情報

### 読み手の時間的制約

> **大前提：読者には時間がない**

- 管理職は30秒で行動を決める → 総論はその範囲で読める量に収める
- 「全部読んでほしい」は幻想。どこを飛ばされても伝わる設計が必要

### 「隠れた読み手」の存在確認

直接の宛先以外に読む可能性がある人を必ず確認する：
- 上司に送ったメールが部署全体に転送される
- 企画書が役員会議で使われる
- 技術ドキュメントが外部公開される

隠れた読み手の存在が確認されたら、その読み手にとって不適切な表現・情報がないか確認する。

## 4. コミュニケーション設定の確認

### 5つの基本要素チェックリスト

書き始める前に以下の5要素を確認する：

```
[ ] 書き手：自分の立場・役割は明確か？
[ ] 読み手：具体的な人物像を思い描けるか？
[ ] テーマ：扱う対象は絞れているか？
[ ] 答え：最終的に何を伝えたいか決まっているか？
[ ] 期待する反応：読んだあとに何をしてほしいか明確か？
```

### 「テーマと期待する反応が曖昧なまま書き始める」が最大の罠

特に「期待する反応」が曖昧なまま書き始めるケースが多い。

| 期待する反応 | 文章の設計が変わる点 |
|------------|------------------|
| 理解してほしい | 説明の深さ・順序・例示の量 |
| 行動してほしい | 行動へのハードル除去・メリット強調 |
| 承認してほしい | 根拠の強さ・リスクへの言及 |

## 5. 論点を決める3ステップ

### ステップ1：論点候補を集める

3エリアから洗い出す：

```
読み手のエリア：読み手が知りたいこと・悩んでいること
テーマのエリア：テーマに関わる問い・未解決の疑問
自分のエリア：自分が伝えたいこと・自分の経験から感じた疑問
```

### ステップ2：論点を絞る

- 似た問いをグルーピングして、大きな問いに統合する
- 問いが多すぎると論点が散漫になる
- 1文章1論点が原則

### ステップ3：論点を決める（4基準）

以下の4基準で最終判断する：

| 基準 | チェック |
|------|---------|
| **自分に切実な動機がある** | この論点について本気で考えているか？ |
| **読み手の要求に合致する** | 読み手が本当に知りたいことか？ |
| **自分の力量で扱える** | 根拠を揃えられるか？ |
| **社会的価値がある** | 誰かの役に立つ問いか？ |

## 6. 論拠の準備

### 特定の読み手向け：論拠の4ステップ

意見を特定の相手に伝えるときの論拠構築：

```
ステップ1：自分の理由を整理する（なぜそう思うのか）
    ↓
ステップ2：相手の都合から考える（相手にとって何がメリットか）
    ↓
ステップ3：相手の反対理由を予測する（相手はなぜ反対するか）
    ↓
ステップ4：反対理由への論拠を用意する（反論を先読みして答える）
```

### 不特定多数向け：論拠の7つの方法

広い読者に向けた文章で論拠を揃える方法：

1. **自分の体験・見聞を洗い出す** — 一次情報は強い。ただし一般化できるか確認
2. **必要な基礎知識を調べる** — 前提となるファクトを正確に把握する
3. **具体的事例を見る** — 抽象論は具体例で裏付けを取る
4. **別の立場から見る** — 反対側の視点も取り入れることで論が強くなる
5. **海外と比較して見る** — グローバル視点で相対化する
6. **歴史を押さえる** — 過去との比較で文脈を与える
7. **スペシャリストの視点を知る** — 専門家の知見を参照する

## 7. アイデアメモの作り方

### 目的

「何が書きたいのか」を断片として書き出し、それをつないで構成を見える化する。

### 5ステップの構成発見プロセス

```
1. 柱（テーマ）を決める
         ↓
2. 動機を確認する（なぜこれを書くのか）
         ↓
3. 全体像を描く（どんな内容を含むか）
         ↓
4. 対比を見つける（何と何を比較するか）
         ↓
5. 結論を見つける（言いたいことは何か）
```

### メモのコツ

- 断片で構わない。「つながった文章」にしなくていい
- 浮かんだことをとにかく書き出す
- 断片が増えてきたらグルーピングして、構成の骨格が見えてくる
- このメモが論点・意見の発見につながることも多い

## 8. 読者分析の深化

既存の「読み手分析」（セクション3）を補強する、より実践的なフレームワーク。

### 読者の3層分析

表層的な属性（年齢・職業）だけでなく、3層で分析する：

| 層 | 分析軸 | 具体例 |
|---|-------|--------|
| **層1: 知識レベル** | 専門性・背景知識はどの程度か | 「Pythonは書けるがMLは初心者」 |
| **層2: 意欲と目的** | なぜこの文章を読むのか、何を得たいか | 「課題解決のヒントを探している」 |
| **層3: 立場になる想像力** | 読み手の視点に立ち切れているか | 「自分の説明が読み手に届いているか」 |

層3が最も重要。知識・意欲がわかっていても、読み手の立場に立てていなければ文章は機能しない。

### 「驚き最小原則」

読み手が予測する文章展開と実際のギャップを最小化する。

**読み手のメンタルモデル**とは、「文章はこう展開するはず」という無意識の予測である。このモデルとのギャップが大きいほど、読み手は混乱し離脱する。

**ギャップ縮小の2戦略**：

| 戦略 | 内容 | 推奨場面 |
|-----|------|---------|
| **A. 文章を読み手のメンタルモデルに近付ける**（推奨） | 読み手の予測通りに展開する。驚きより理解を優先 | ビジネス文書・技術文書 |
| **B. 読み手のメンタルモデルを修正させる** | 冒頭で「この文章の読み方」を明示し、期待値を調整する | チュートリアル・概念解説 |

### 「読者は20%しか読まない」前提

デジタル時代の読者行動に関する重要な前提：

- 研究によれば、1,200語を超える文章では約20%程度しか読まれない
- スマホの普及により集中力は低下傾向にある
- 「全部読んでほしい」は幻想。どこを飛ばされても伝わる設計が必要

**実践的対応**：
- 重要情報は冒頭に配置（結論ファースト）
- 飛ばし読みされても本質が伝わる見出し・箇条書きを活用
- 「読んでもらえない」前提で書くことで、読んでもらえる文章になる

### 読者設定の具体化テクニック

抽象的な「読者」ではなく、特定の人物を思い浮かべて書く：

**「10年前の自分」に語りかける**
- 今の自分が持つ知識を持っていなかった頃の自分が読者
- その人が何を知らず、何につまずいていたかを思い出す
- 「その人に説明する」という具体的なイメージで書く

**「あの人」を想定する**
- 実際に存在する特定の人物（知人・同僚・友人）を読者に設定する
- 「この文章を読んだあの人はどう思うか」を随時確認する
- 抽象的な「一般読者」より、具体的な一人の方が書きやすく、伝わりやすい

## 9. 「問い」を立てる技術

文章の骨格は「問い」から生まれる。書く前に問いを設計することが、構成の質を決定する。

### 3ステップ法

```
ステップ1：「大きな問い」を立てる
  ↓  テーマではなく「問い」で考える
  ↓  例：「リモートワーク」ではなく「リモートワークで生産性は上がるのか？」
ステップ2：「小さな問い」に分解する
  ↓  大きな問いを複数の具体的な問いに分ける
  ↓  例：「なぜ上がると言われているのか？」「実際のデータは？」「どんな条件下で？」
ステップ3：「小さな問い」の答えを出す
  → 各答えが文章のパーツになり、合わさると骨格が完成する
```

### 問いの7つの型

文章の目的・性質に応じて適切な型を選ぶ：

| 型 | 問いの形 | 例 | 生まれる文章の性格 |
|---|---------|-----|---------------|
| **同格型** | 「Aとは何か？」 | 「SDGsとは何か？」 | 定義・解説 |
| **対比型** | 「AとBの違いは？」 | 「JavaとPythonの違いは？」 | 比較・選択肢提示 |
| **変化型** | 「Aはなぜ変わったのか？」 | 「リモートワークはなぜ普及したのか？」 | 原因分析・歴史 |
| **ギャップ型** | 「理想と現実の差は？」 | 「なぜ計画通りにいかないのか？」 | 問題提起・改善提案 |
| **葛藤型** | 「AかBか？」 | 「成長か安定か？」 | 意思決定・論争 |
| **説話型** | 「どうすればよいか？」 | 「生産性を上げるには？」 | 手順・ガイド |
| **原因型** | 「なぜ起きたのか？」 | 「なぜプロジェクトが遅延したのか？」 | 原因究明・再発防止 |

### 「なぜ？」の3種類を使い分ける

同じ「なぜ？」でも、問いの方向が異なる：

| 種類 | 問う方向 | 例 |
|-----|---------|-----|
| **原因を問う「なぜ？」** | 過去の因果関係 | 「なぜこのバグが発生したのか？」 |
| **結果を問う「なぜ？」** | 未来への影響・予測 | 「なぜこれが重要になるのか？」 |
| **仮説を問う「なぜ？」** | 検証可能な推測 | 「なぜAの方がBより効果的なのか？」 |

複数の「なぜ？」を組み合わせることで、より立体的な問いの構造が生まれる。

### 問いの具体化テクニック

「問いが漠然としている」と感じたとき、3方向で掘り下げる：

**言葉で分解する**
- 問いに含まれるキーワードの定義を掘り下げる
- 例：「生産性とは何を指すのか？」「時間効率か？成果の質か？」

**分野で分解する**
- 同じ問いを異なる視点（経済・技術・社会・心理）から問い直す
- 例：「リモートワーク普及」→ 経済的要因・技術的要因・文化的要因

**見方で分解する**
- 賛成/反対、短期/長期、個人/組織で切り替える
- 例：「リモートワークは良いか？」→ 従業員視点と企業視点で分離
