# よくある誤りとチェックリスト

テクニカルライティングでよくある誤りと、それを避けるための実践的なガイドを提供します。

## ⚠️ よくある誤りパターン

### 1. 冗長表現

#### 意味の重複

| ❌ 冗長表現 | ✅ 簡潔な表現 | 説明 |
|----------|-----------|------|
| まず最初に | まず / 最初に | 「まず」と「最初」は同じ意味 |
| あらかじめ予測 | 予測 | 「予測」自体が事前の概念 |
| 後で後述する | 後述する | 「後で」が重複 |
| 最も最適な | 最適な | 「最適」は既に最上級 |
| 再び再起動 | 再起動 | 「再び」が重複 |
| 各それぞれの | 各 / それぞれの | 意味が重複 |

#### 不必要な助動詞

| ❌ 冗長表現 | ✅ 簡潔な表現 |
|----------|-----------|
| することができます | できます / します |
| する必要があります | してください / します |
| することが可能です | できます |
| することが重要です | してください |

#### 不要な前置き・限定詞

| ❌ 不要な表現 | ✅ 改善例 |
|------------|---------|
| 言うまでもなく | （削除） |
| 基本的に | （削除または具体的に） |
| 一般的に | （削除または具体的に） |
| ある意味では | （削除） |
| いわば | （削除） |

### 実践例

```markdown
❌ 冗長
まず最初に、基本的な設定を行う必要があります。
一般的に、この処理には数分かかることがあります。

✅ 簡潔
最初に設定を行ってください。
この処理には通常2-3分かかります。
```

### 2. 曖昧な表現

#### 定量化できる表現

| ❌ 曖昧 | ✅ 具体的 | 理由 |
|--------|---------|------|
| パフォーマンスが向上 | 応答時間が50%短縮 | 測定可能 |
| たくさんのユーザー | 月間10万ユーザー | 具体的な数値 |
| 高速な処理 | 50ms未満で完了 | 明確な基準 |
| 軽量なライブラリ | バンドルサイズ50KB | 測定可能 |
| しばらく待つ | 5秒待機 | 具体的な時間 |

#### 程度を示す表現

| ❌ 曖昧 | ✅ 具体的 |
|--------|---------|
| 大幅に改善 | 従来比200%向上 |
| ほとんどのケース | 95%のケース |
| 少ないメモリ | 最大512MB |
| 多くの機能 | 15の主要機能 |

### 実践例

```markdown
❌ 曖昧
このAPIは高速で、たくさんのリクエストを処理できます。
パフォーマンスも大幅に向上しています。

✅ 具体的
このAPIは平均45msで応答し、秒間1000リクエストを処理できます。
前バージョンと比較して、応答時間が60%短縮されました。
```

### 3. 技術用語の不統一

#### 同一対象の用語混在

| ❌ 不統一 | ✅ 統一すべき表現 |
|---------|--------------|
| ユーザー/クライアント/顧客 | ユーザー（プロジェクトで統一） |
| 設定画面/設定ページ/環境設定 | 設定画面（UI名称に統一） |
| DB/データベース/database | データベース（初出時に略語定義） |
| API/エーピーアイ | API（略語で統一） |

#### 略語の扱い

```markdown
✅ 正しい略語の使用
データベース（DB）の設定を変更します。
DBに接続できない場合は、設定を確認してください。

❌ 略語の不統一
データベースの設定を変更します。
DBに接続できない場合は、databaseを確認してください。
```

### 実践例

```markdown
❌ 用語の不統一
ユーザーはプロフィール画面でアバターを変更できます。
クライアント一覧ページでは全顧客の画像が表示されます。

✅ 用語の統一
ユーザーはプロフィール画面でアバターを変更できます。
ユーザー一覧画面では全ユーザーのアバターが表示されます。
```

### 4. 受動態の濫用

#### 主語が不明確な表現

| ❌ 受動態（主語不明） | ✅ 能動態（主語明確） |
|-----------------|----------------|
| 処理が実行される | システムが処理を実行する |
| データが保存される | データベースにデータを保存する |
| エラーが発生する | APIサーバーがエラーを返す |
| 通知が送信される | システムが通知を送信する |

### 実践例

```markdown
❌ 受動態が多い
設定ファイルが読み込まれ、データベースへの接続が確立されます。
その後、初期化処理が実行され、サーバーが起動されます。

✅ 能動態
システムが設定ファイルを読み込み、データベースに接続します。
その後、初期化処理を実行し、サーバーを起動します。
```

### 5. 文の長さと複雑さ

#### 長文の問題

```markdown
❌ 長すぎる（100文字超）
このメソッドはユーザーの認証情報を検証してトークンを生成し、データベースに保存した後に
クライアントにレスポンスを返す一連の処理を実行するため、エラーハンドリングが重要です。

✅ 適切な長さ（50文字以内）
このメソッドは以下の処理を順次実行します。
1. 認証情報の検証
2. トークンの生成
3. データベースへの保存
4. クライアントへのレスポンス返却

各ステップで適切なエラーハンドリングを実装しています。
```

#### 一文一義の違反

```markdown
❌ 複数の意味を含む
このAPIはユーザーを作成し、確認メールを送信し、ログに記録します。

✅ 一文一義
このAPIは以下の処理を実行します。
- ユーザーの作成
- 確認メールの送信
- ログへの記録
```

### 6. 不適切なトーン

#### 不適切な表現

| ❌ 不適切 | ✅ 適切 |
|---------|--------|
| APIキーがないと動かない | APIキーの設定が必要です |
| 設定してないなら先に設定して | 先に設定を完了してください |
| これやっとけ | この手順を実施してください |
| バグってる | 不具合が発生しています |

### 実践例

```markdown
❌ 不適切なトーン
環境変数を設定してないとエラーになる。
先にやっとけ。

✅ 適切なトーン
この機能を使用する前に、環境変数の設定が必要です。
設定方法については「環境構築」セクションを参照してください。
```

### 7. 不完全な手順

#### 前提条件の欠落

```markdown
❌ 不完全
npm installを実行してください。

✅ 完全
## 環境構築

### 前提条件
- Node.js 18.0.0以上
- npm 9.0.0以上

### 手順
1. リポジトリをクローン
   ```bash
   git clone https://github.com/user/repo.git
   cd repo
   ```

2. 依存関係のインストール
   ```bash
   npm install
   ```

3. 正常にインストールされたことを確認
   ```bash
   npm list
   ```
```

## 📋 文書作成チェックリスト

### 基本品質（7つのC）

#### Clear（明確）
- [ ] 一つの文に一つの主要なアイデア
- [ ] 専門用語は定義されている
- [ ] 曖昧な表現がない

#### Concise（簡潔）
- [ ] 冗長表現を排除している
- [ ] 不要な前置きがない
- [ ] 助動詞の重複がない

#### Correct（正確）
- [ ] 技術的事実が正確
- [ ] 数値データに誤りがない
- [ ] 文法的に正しい

#### Coherent（一貫）
- [ ] 技術用語が統一されている
- [ ] 文体が統一されている
- [ ] 論理的な流れがある

#### Concrete（具体的）
- [ ] 抽象的表現を避けている
- [ ] 具体的な数値を使用している
- [ ] 測定可能な基準を示している

#### Complete（完全）
- [ ] 前提条件が明記されている
- [ ] 手順が完全に記載されている
- [ ] 確認方法が示されている

#### Courteous（丁寧）
- [ ] 適切なトーンで記述されている
- [ ] 読者への配慮がある
- [ ] 警告が明確に示されている

---

### 構造チェック

#### 文章構造
- [ ] 一文の長さは50文字以内
- [ ] 一文一義になっている
- [ ] 能動態を使用している
- [ ] 段落の最初にトピックセンテンスがある

#### ドキュメント構造
- [ ] 見出し階層が適切（H1は1つ）
- [ ] 箇条書きと表を適切に使い分けている
- [ ] コードブロックに言語指定がある
- [ ] クロスリファレンスが機能する

---

### 内容チェック

#### 技術文書
- [ ] 環境要件が明記されている
- [ ] セットアップ手順が完全
- [ ] コード例が動作する
- [ ] エラーハンドリングが説明されている

#### 報告書
- [ ] 実施内容が明確
- [ ] 成果物がリストアップされている
- [ ] 品質チェック項目が記載されている
- [ ] 次のステップが示されている

#### API仕様書
- [ ] エンドポイントが明記されている
- [ ] リクエスト形式が説明されている
- [ ] レスポンス例が提供されている
- [ ] エラーコードが説明されている

---

### 最終確認

#### 公開前チェック
- [ ] 誤字脱字がない
- [ ] リンクが正しく機能する
- [ ] 画像が表示される
- [ ] コードが正しくフォーマットされている
- [ ] 目次が正確（必要な場合）

#### レビュー観点
- [ ] 他者が読んで理解できるか
- [ ] 手順通りに実行できるか
- [ ] 必要な情報がすべて含まれているか
- [ ] 冗長な部分がないか

## 🔄 改善プロセス

### ステップ1: 初稿作成
内容を優先し、まず書き上げる

### ステップ2: 自己レビュー
- 7つのCチェックリストで確認
- 冗長表現を削除
- 曖昧な表現を具体化

### ステップ3: 構造確認
- 見出し階層の確認
- 論理的な流れの確認
- クロスリファレンスの確認

### ステップ4: 他者レビュー
- 第三者に読んでもらう
- 理解できるか確認
- フィードバックを反映

### ステップ5: 最終確認
- 誤字脱字チェック
- リンクの動作確認
- コードの動作確認
