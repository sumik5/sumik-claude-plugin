# 7つのCの原則 - 詳細ガイド

効果的なテクニカルライティングの核となる7つの原則を詳しく解説します。

## 1. Clear（明確）

**曖昧さがなく、容易に理解できる**

### 基本原則
- 一つの文に一つの主要なアイデア
- 専門用語は必要に応じて定義
- 読者が想像で補完する必要がない

### 実践例

```markdown
✅ 良い例
このAPIは最大1000件のレコードを返します。
1000件を超える場合はページネーションを使用してください。

❌ 悪い例
このAPIはたくさんのレコードを返します。
たくさん取得する場合は別の方法を使ってください。
```

### 改善のポイント
- **数値を明示**: 「たくさん」→「1000件」
- **具体的な指示**: 「別の方法」→「ページネーション」
- **前提条件を明記**: 「1000件を超える場合」

---

## 2. Concise（簡潔）

**必要な情報を最小限の言葉で表現**

### 冗長表現の排除

| 冗長表現 | 簡潔な表現 | 理由 |
|---------|----------|------|
| まず最初に | まず / 最初に | 意味の重複 |
| あらかじめ予測 | 予測 | 「予測」自体が事前の概念を含む |
| することができます | できます / します | 不必要な助動詞の重複 |
| する必要があります | してください / します | より直接的な表現 |
| 言うまでもなく | （削除） | 不要な前置き |
| 基本的に | （削除） | 曖昧な限定詞 |
| 一般的に | （削除または具体的に） | 具体性を欠く |

### 実践例

```markdown
✅ 簡潔
設定ファイルを編集してください。

❌ 冗長
まず最初に、基本的な設定ファイルの編集を行う必要があります。
```

### 改善テクニック
1. **修飾語を削減**: 「基本的に」「一般的に」などを削除
2. **助動詞を整理**: 「することができる」→「できる」
3. **不要な前置きを削除**: 「言うまでもなく」などを削除

---

## 3. Correct（正確）

**文法、事実、技術的内容に誤りがない**

### 技術的正確性

```markdown
✅ 正確
このメソッドは O(n log n) の時間計算量で動作します。
最悪ケースでも O(n²) を超えることはありません。

❌ 不正確
このメソッドは速く動作します。
パフォーマンスは良好です。
```

### 数値の正確性

```markdown
✅ 正確
応答時間: 平均 45ms、95パーセンタイル 120ms
メモリ使用量: 最大 512MB

❌ 不正確
応答時間: 速い
メモリ使用量: 少ない
```

### 検証可能な情報
- **測定可能**: 「速い」→「45ms」
- **再現可能**: 条件とバージョンを明記
- **出典明示**: 公式ドキュメントやベンチマーク結果

---

## 4. Coherent（一貫）

**論理的に結びつき、スムーズに流れる**

### 用語の統一

| 不一致の例 | 統一すべき表現 | 理由 |
|-----------|--------------|------|
| ユーザー/クライアント/顧客 | プロジェクトで統一 | 同一対象は同一用語 |
| 設定画面/設定ページ/環境設定 | UI名称で統一 | 機能名は正確に |
| です・ます調/だ・である調 | 文書全体で統一 | 文体の混在を避ける |
| DB/データベース/database | 初出時に定義して統一 | 略語の一貫性 |

### 実践例

```markdown
✅ 一貫した用語
ユーザーはプロフィール画面でアバターを変更できます。
ユーザー一覧画面では全ユーザーのアバターが表示されます。

❌ 用語の不統一
ユーザーはプロフィール画面でアバターを変更できます。
クライアント一覧ページでは全顧客の画像が表示されます。
```

### 論理的な流れ

```markdown
✅ 論理的な順序
1. 環境構築の準備
2. 依存関係のインストール
3. 設定ファイルの編集
4. アプリケーションの起動
5. 動作確認

❌ 順序が不適切
1. アプリケーションの起動
2. 依存関係のインストール
3. 動作確認
```

---

## 5. Concrete（具体的）

**抽象的でなく、測定可能で明確**

### 抽象的表現の具体化

| 抽象的表現 | 具体的表現 | 改善効果 |
|----------|----------|---------|
| 高速なパフォーマンス | 50ms未満の応答時間 | 測定可能な基準 |
| 大幅に向上 | 従来比200%向上 | 定量的な情報 |
| 効率的な | メモリ使用量を30%削減 | 具体的な効果 |
| 適切な | セキュリティ基準に準拠した | 明確な判断基準 |
| 多くのユーザー | 月間10万ユーザー | 具体的な数値 |
| 軽量な | バンドルサイズ50KB | 測定可能 |

### 実践例

```markdown
✅ 具体的
このライブラリを使用すると、バンドルサイズが50KBから15KBに削減されます。
初回ロード時間は平均2.3秒から0.8秒に短縮されます。

❌ 抽象的
このライブラリは軽量で高速です。
パフォーマンスが大幅に向上します。
```

### 具体化のテクニック
1. **数値で表現**: 「速い」→「50ms」
2. **比較基準を示す**: 「向上」→「従来比200%向上」
3. **条件を明記**: 「効率的」→「メモリ使用量を30%削減」

---

## 6. Complete（完全）

**必要な情報がすべて含まれている**

### 環境構築の例

```markdown
✅ 完全な情報
## 環境構築

### 前提条件
- Node.js 18.0.0以上
- npm 9.0.0以上
- PostgreSQL 14以上

### 手順
1. リポジトリをクローン
   ```bash
   git clone https://github.com/user/repo.git
   cd repo
   ```

2. 依存関係のインストール
   ```bash
   npm install
   ```

3. 環境変数の設定
   ```bash
   cp .env.example .env
   # .envファイルを編集してAPIキーを設定
   ```

4. データベースのマイグレーション
   ```bash
   npm run migrate
   ```

5. 開発サーバーの起動
   ```bash
   npm run dev
   ```

6. ブラウザで確認
   http://localhost:3000 を開く

❌ 不完全な情報
## 環境構築

1. Node.jsをインストール
2. npm installを実行
3. サーバーを起動
```

### 完全性のチェックポイント
- **前提条件**: 必要なソフトウェアとバージョン
- **手順**: 順序立てた具体的なステップ
- **コマンド例**: 実行するコマンドを明記
- **確認方法**: 成功したかどうかの確認手段
- **トラブルシューティング**: よくある問題と解決策

---

## 7. Courteous（丁寧）

**読者を意識した適切なトーンと構成**

### トーンの調整

```markdown
✅ 丁寧で建設的
この機能を使用する前に、APIキーの設定が必要です。
設定方法については「環境構築」セクションを参照してください。

❌ 不適切なトーン
APIキーがないと動かない。
設定してないなら先に設定して。
```

### 読者への配慮

```markdown
✅ 配慮がある
この操作は元に戻せません。実行前にバックアップを作成することをお勧めします。

⚠️ 警告
本番環境で実行する場合は、必ず事前にステージング環境でテストしてください。

❌ 配慮が不足
これを実行するとデータが消えます。

```

### 丁寧な表現のポイント
1. **可能性を示す**: 「〜してください」「〜をお勧めします」
2. **理由を説明**: 「なぜ」その操作が必要かを明記
3. **警告を明示**: リスクがある場合は明確に警告
4. **代替案を提示**: 問題がある場合の解決策を示す

---

## 実践のための優先順位

文書作成時の優先順位：

1. **Correct（正確）**: 技術的事実の正確性が最優先
2. **Clear（明確）**: 読者の理解を妨げない表現
3. **Complete（完全）**: 必要な情報がすべて含まれる
4. **Concrete（具体的）**: 測定可能で明確な表現
5. **Concise（簡潔）**: 冗長性を排除
6. **Coherent（一貫）**: 用語と構成の統一
7. **Courteous（丁寧）**: 読者への配慮
